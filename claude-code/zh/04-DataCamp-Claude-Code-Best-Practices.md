## 為何規劃紀律能改變一切

Claude 在沒有引導的情況下做出的每個決策，可能單一決策準確率很高，乍聽之下沒問題，但當一個功能有許多決策點時，全部正確的機率就會不斷下降。假設每個決策有 80% 的準確率、共有 20 個決策點，計算 0.8^20，功能完全正確實作的機率僅約 1%。

規劃將那 20 個模糊決策壓縮為一份經過審查的規格，讓每個決策都接近 100% 的確定性，因為你已事先做出判斷。

### 使用標注循環工作流程進行規劃

如 Boris Tane 所建議，在大規模應用中效果最佳的規劃工作流程是標注循環。讓 Claude 起草一份 plan.md，在你的編輯器中打開，並在 Claude 判斷錯誤或留有歧義之處加入行內備註，例如「使用 drizzle:generate，而非原始 SQL」或「這應該是 PATCH，不是 PUT」。

然後帶著護衛語句「處理所有備註，但還不要實作」將標注後的計畫送回。這個護衛語句很重要，因為少了它，Claude 會跳過計畫直接開始編碼。這個循環重複進行，直到計畫沒有歧義為止，此時 Claude 實作時會少走許多彎路，因為所有決策都已事先確定。

```markdown
# plan.md — 標注循環

## 步驟 3：資料庫遷移
為 users 表格建立新的遷移。
> 備註：使用 drizzle:generate，而非原始 SQL
> 備註：加入預設值為 NOW() 的 created_at 欄位

## 步驟 4：API 端點
新增 PUT /users/:id 端點。
> 備註：應為 PATCH，不是 PUT。僅支援部分更新。

# 標注完成後，加上以下語句送回：
# "address all notes, don't implement yet"
```

### 使用 Claude Code 規劃模式

如果標注循環感覺過於繁重，Claude Code 內建的規劃模式是一個較輕量的選項：

1. 按兩次 Shift+Tab。
2. 在對話中反覆調整計畫。
3. 按一次 Shift+Tab 切換至自動接受模式。

計畫會儲存至 `~/.claude/plans/`，因此能在壓縮和工作階段重啟後保留，使內建規劃模式成為大多數任務的可靠預設選擇。對於非常大型的功能，事先撰寫完整規格也很有效：一位開發者花了兩小時撰寫一份 12 步驟的規格，估計節省了 6-10 小時的實作時間。

### 進一步延伸規劃

無論採用哪種方式，在規劃請求中一併貼上優質的開源程式碼，都能顯著提升輸出品質，因為 Claude 使用可運作的參考範例比使用抽象描述效果更好。

規劃也可以透過 git worktrees 橫向擴展。incident.io 的工程師在不同分支上同時執行 4-5 個平行的 Claude 工作階段，每個各自執行自己的計畫。一位工程師花費 $8 美元的 Claude 額度，產出了一個將 API 生成時間提升 18% 的實作，為整個團隊每次使用的工具節省了 30 秒。

部分開發者更進一步，執行競爭性的 worktree，以不同方式實作同一個問題，再比較結果。

## CLAUDE.md 架構最佳實踐

你的 CLAUDE.md 檔案有一個你可能不知道的額度限制。HumanLayer 對 Claude Code 內部機制的分析發現，系統會在你的指令上方注入一個提醒：「此情境可能與你的任務相關，也可能無關。」

### 理解 Claude Code 的分層指令系統

Claude 會主動篩選它所遵循的內容，而非將所有內容視為持久性命令。此外，前沿推理模型大約遵循 150-200 條指令後，合規性便會下降，而 Claude Code 自身的系統提示就佔用了約 50 個槽位。這讓你的規則剩下大約 100-150 個槽位。HumanLayer 將自己的檔案保持在 60 行以內。

你可以親眼觀察這個額度的運作方式。在你的 CLAUDE.md 中加入一行「請始終稱呼我為 Mr. Tinkleberry」，看看 Claude 多快就停止這樣稱呼你——通常在幾千個 token 之內。當這個名字消失時，你的指令就被注意力機制降低了優先級，CLAUDE.md 中的其他所有內容也隨之失去影響力。

### 使用漸進式揭露進行指令參照

在這個額度限制內運作的方法是漸進式揭露。你的根目錄 CLAUDE.md 保持簡短，專注於適用所有地方的規則。對於任何特定領域的內容，參照一個獨立檔案而非內嵌其中：「當處理付款系統時，請先閱讀 docs/payment-architecture.md。」

Claude 只有在進入程式碼庫的那個部分時才會讀取參照的檔案，讓你的指令額度保留給高優先級規則。一家每月消耗數十億 token 的公司以這種方式組織他們的 monorepo CLAUDE.md，並為每個團隊分配了一個 token 額度用於其部分：

```markdown
# Monorepo CLAUDE.md — 漸進式揭露

## Python
- 函式簽名務必使用型別提示
- 使用以下方式測試：pytest -x --tb=short

## <內部 CLI 工具>
- <使用範例>
- 處理前務必驗證輸入
- 禁止使用原始 SQL，優先使用 ORM

如需 <複雜使用方式> 或遭遇 <錯誤>，請參閱 path/to/<tool>_docs.md
```

子目錄 CLAUDE.md 檔案進一步延伸這個架構。在 `src/persistence/` 放置含有資料庫專用指令的 CLAUDE.md，在 `src/api/` 放置含有端點規範的另一份。

Claude 會自動從工作目錄向上載入 CLAUDE.md 檔案至專案根目錄，因此子目錄的檔案只有在 Claude 在該區域工作時才會啟動。這讓根目錄檔案保持通用，同時在確切需要的地方為 Claude 提供針對性的指引。

一個常見的錯誤：不要用 @ 將文件檔案嵌入 CLAUDE.md。這樣做會在每次執行時嵌入整個檔案，在對話開始前就耗盡你的指令額度。

## Claude Code 的情境管理

Claude Code 提供 200K token 的情境，但可用的視窗比表面上看起來要小。

在 monorepo 中開啟一個新的工作階段，光是載入系統提示、工具定義和 CLAUDE.md，在你輸入任何內容之前就已消耗約 20K token。每個 MCP 伺服器都會加入永久佔用情境的工具綱要，因此實際限制大約是 5-8 個伺服器，超過就會開始擠壓實際工作空間。

更糟的是，品質開始下降的時間點比你預期的早很多。多位從業者獨立地得出了相同的門檻：不要讓情境超過 60% 的容量。

Claude 的輸出在視窗的 20-40% 就開始退化，遠在任何限制觸發之前，因為注意力機制在情境填滿時會給予先前指令較少的權重。

自動壓縮在大約 83.5% 時觸發，而且是有損的：一位開發者在工作階段中途進行壓縮時，損失了 3 小時的重構工作，所有遷移決策的記憶都被抹除，只保留了約 20-30% 的細節。

### 使用「記錄與清除」模式

最有效的防禦是「記錄與清除」模式。當情境變得沉重時：

1. 將當前計畫和進度傾印至一個 markdown 檔案
2. 執行 `/clear` 重置工作階段
3. 讓 Claude 讀取該檔案後重新開始

這樣，你將獲得完整的 200K 視窗，只保留你選擇保留的內容，這比 `/compact` 更好，因為你能精確控制哪些內容得以保留。

自訂的 `/catchup` 指令能使切換更順暢：清除後，它會讀取當前 git 分支中所有已變更的檔案，讓 Claude 從上次中斷處繼續，而無需舊的對話歷史。

```markdown
<!-- .claude/commands/catchup.md -->
在 /clear 之後重建情境。讀取當前分支相較於 main 分支所有已修改的檔案。
針對每個檔案，理解所做的變更。
然後總結到目前為止已實作的內容以及剩餘的工作。

此分支上已變更的檔案：
$ git diff --name-only main
```

### 使用自訂 Claude Code 技能進行情境傳遞

我使用一個 `/transfer-context` 技能將此進一步延伸。當工作階段開始退化時，該指令會傾印一個結構化的交接檔案，包含已完成的工作、待決定的事項、需避免的陷阱以及相關的檔案路徑。下一個工作階段讀取該檔案後，從上次中斷處繼續，只保留重要資訊，去除所有對話冗余。

### 節省情境的其他最佳實踐

你也可以透過延遲載入工具來主動節省情境。一個專案透過使用 UserPromptSubmit hooks，僅在使用者的提示觸發相關關鍵字時才注入技能定義（而非在啟動時預載所有內容），每個工作階段節省了約 15,000 個 token。

這一切之下最簡單的規則：每個對話只處理一個任務。重新開始只需 20K token，這與被污染的工作階段所造成的品質損失相比微不足道。

## Claude Code Hooks 作為確定性護欄

即使結構良好的 CLAUDE.md，遵循率也只有約 70%。對於編碼風格偏好來說，這還算可以。但對於「不要推送到 main」或「不要刪除生產資料」等安全規則，這就遠遠不夠。Hooks 透過在 Claude 工作流程的特定時間點執行 shell 腳本，將這個差距補足到 100%。

有兩種值得了解的類型。「提交時攔截」hooks 作為 PreToolUse 事件執行，能直接阻止動作：退出代碼 2 會封鎖動作並強制 Claude 嘗試其他方式。提示 hooks 提供非封鎖性的回饋，例如在每次編輯後執行 linter 並回傳輸出，而不中斷 Claude 的工作流程。

最完整的公開 hook 設定執行以下操作：

- 封鎖 `rm -rf` 指令（建議改用資源回收桶），
- 防止直接推送到 main，
- 記錄所有變更操作及時間戳，以及
- 執行反合理化閘門。

該閘門使用 Haiku 審查 Claude 的回應，尋找如「預先存在的問題」或「超出範疇」等逃避說詞。當它發現 Claude 宣告過早完成時，會以具體回饋拒絕該回應，並強制 Claude 繼續工作。一個較輕量的設定在每次檔案編輯後自動執行 Prettier 和 TypeScript 檢查：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "prettier --write \"$CLAUDE_FILE_PATHS\""
          }
        ]
      }
    ]
  }
}
```

需要避免的一個陷阱：絕不要在計畫執行中途封鎖 Edit 或 Write 工具。在寫入時封鎖會破壞多步驟推理，因為 Claude 會失去追蹤進度的能力。讓它完成寫入，然後透過 PostToolUse hooks 或預提交檢查來驗證。如果你想在這些基礎上進一步擴展，我們的 Claude Code hooks 教學涵蓋了更多模式。

## 測試驅動開發：最佳代理編程策略

沒有測試，Claude 驗證其工作的唯一方法就是自身的判斷，而隨著情境填滿，這種判斷會不斷退化。測試建立了一個外部的驗證機制，無論工作階段執行多久都能保持準確。

每個從紅到綠的循環都給予 Claude 明確的回饋，它可以在無需人工介入的情況下遍歷整個測試套件，這使得測試驅動開發（TDD）成為與代理編程工具協作時最強大的單一模式。

Anthropic 建議的工作流程遵循特定的順序：

```
1. 先撰寫測試
   > "使用 pytest 為 auth 模組撰寫測試。
      採用 TDD 方式，不使用模擬實作。"

2. 確認測試失敗
   > "執行測試。它們都應該失敗。"

3. 以失敗的測試作為檢查點進行提交

4. 持續實作直到全部通過
   > "撰寫實作。不得修改測試。
      持續進行直到所有測試通過。"
```

最後那條指令比看起來更重要。Claude 有時會修改測試讓其通過，而非修復實作。事先提交測試能為你提供一個安全網：如果 Claude 更改了測試，diff 會精確顯示發生了什麼變化，你可以還原。

對於前端工作，這個循環的視覺化變體也很有效：如果你給 Claude 一個設計模型加上 Puppeteer MCP 伺服器，它在實作後截圖，與模型比較，然後反覆調整。當 Claude 能夠對照視覺目標檢查自身輸出時，這個驗證循環能帶來 2-3 倍的品質提升。

## Claude Code 成本經濟學與模型選擇

Anthropic 的官方數據顯示，Claude Code 的平均成本以 API 定價計算，每位開發者每天約 $6 美元，90% 的開發者每天不超過 $12 美元。以月計算，使用 Sonnet 4.6 大約是每位開發者 $100-200 美元。

Max 訂閱方案完全改變了這個計算方式。一位開發者追蹤了 8 個月、約 100 億 token 的使用量，發現其 API 等效成本超過 $15,000 美元，而實際的 Max 訂閱費用約為 $800 美元，節省了 93%。

超過 90% 的 token 是快取讀取，這正是計量 API 定價比固定訂閱費貴得多的原因。Max 方案的損益平衡點大約在每月 $100-200 美元的 API 等效使用量，任何日常使用者都能很快超過這個門檻。

| 定價模式 | 月費 | 最適合 |
|---|---|---|
| API（Sonnet 4.6） | ~$100-200 | 輕度使用，每天不到 30 分鐘 |
| API（Opus 4.6） | ~$300-800 | 複雜的多檔案工作 |
| Max（$100/月） | $100 固定 | 日常使用者，損益平衡點約在 API 等效 $100 |
| Max（$200/月） | $200 固定 | 重度使用者，每天 5 小時以上 |

模型選擇本身就是一種優化。Claude Code 的「opusplan」模式在規劃時使用 Opus 4.6，在程式碼生成時自動切換至 Sonnet 4.6，讓你在最重要的地方獲得 Opus 品質的推理，同時在實作 token 上使用 Sonnet 便宜 5 倍的費率。

在 Anthropic 內部測試中，59% 的 Claude Code 使用者偏好 Sonnet 4.6 而非 Opus 4.5，它傾向產出更簡潔、過度設計更少的程式碼。對於大量使用子代理的工作流程，`CLAUDE_CODE_SUBAGENT_MODEL="claude-sonnet-4-5-20250929"` 可讓子代理在 Sonnet 上執行，同時保留 Opus 用於協調器。

請務必避免三大 token 浪費來源：

- 任務之間不清除情境
- 因 CLAUDE.md 結構不良造成的重複檔案讀取
- 模糊的提示讓 Claude 陷入試錯循環

僅僅修正這三點，通常就能將 token 使用量減少一半。

## Claude Code 疑難排解

在結束之前，讓我們看看開發者在使用 Claude Code 時常遇到的問題以及如何解決。

### 情境遺失

最嚴重的故障模式是情境遺失。情境管理章節中介紹的「記錄與清除」模式正是為此而生：絕不要讓一個冗長的工作階段成為你唯一的決策記錄。頻繁提交、將進度傾印至檔案，並將每個工作階段視為可拋棄的。

### 使用小眾技術時的幻覺問題

Claude 也會針對它不熟悉的技術生成看起來自信且合理的程式碼。如果你在一個自己無法親自驗證的語言或框架中工作，每個輸出都需要額外的審查。正如一位開發者所說：「在我不熟悉的技術上使用 LLM，讓我陷入了很大的麻煩。但在我熟悉的東西上，LLM 大幅提升了我的開發速度。」

### 過度設計

過度設計是一個常見傾向。Claude 會撰寫額外的抽象層、未經請求的輔助函式，以及過早的重構，除非你明確告知它不要這樣做。在你的 CLAUDE.md 中加入「使用最簡單可行的方式」有所幫助，而按問題領域而非技術層次組織程式碼庫，也能降低 Claude 和開發者的認知負擔。

### 關鍵資料遺失

有記錄以來最嚴重的故障：在開發一款語音教學應用程式時，Claude 刪除了一位開發者親自取得授權使用的所有音素音頻檔案，並用 AI 生成的聲音取而代之。它重新命名了檔案，並「深信自己之前的標記是錯誤的，儘管它本身根本無法分辨相關的音素」。

從中學到的教訓：在讓 Claude 存取不可替代的檔案之前，務必先提交或備份它們。

### 無效率的工作階段

Anthropic 自家對於工作失控的工作階段給出的建議出人意料地坦誠：在讓 Claude 工作之前儲存你的狀態，讓它執行，然後接受結果或重新開始，而非費力糾錯。

對 Claude 提交的信任程度取決於你的測試覆蓋率和風險承受度。有些開發者每天透過斜線指令提交數十次，以 PR 審查作為把關機制。對於有付費使用者的生產程式碼庫，審查每個 diff 是值得付出的代價。

## 結論

本文中的每個模式都指向同一個核心概念：在執行前積極約束 Claude，然後給予它驗證自身輸出的方式。規劃消除歧義，CLAUDE.md 和 hooks 設定邊界，測試提供驗證，情境衛生讓一切在工作階段間持續有效運作。

如果你只選擇一個改變來開始，就選「記錄與清除」。跨工作階段的情境腐爛所帶來的代價，遠超過重新開始所需的 20K token。之後：

- 新增一個在測試失敗時封鎖提交的預提交 hook
- 在觸碰程式碼之前，為每個多檔案任務先建立計畫
- 將 CLAUDE.md 保持在 100 條指令以內，並使用子目錄檔案管理特定領域的規則

## Claude Code 最佳實踐常見問題

### 最重要的 Claude Code 最佳實踐是什麼？

影響最大的實踐包括：實作前先規劃（標注循環或規劃模式）、將 CLAUDE.md 保持在 150 條指令以內並採用漸進式揭露、在 60% 容量時清除工作階段來管理情境、使用 hooks 進行確定性的安全執行，以及在實作前先撰寫測試，讓 Claude 有外部驗證機制可以對照。

### 我應該如何組織我的 CLAUDE.md 檔案？

將根目錄 CLAUDE.md 保持在 60 行以內，包含適用於所有地方的規則。透過參照特定領域的檔案而非內嵌來使用漸進式揭露。在 `src/persistence/` 或 `src/api/` 等區域放置子目錄 CLAUDE.md 檔案，提供針對性指引，只有在 Claude 在這些目錄中工作時才會載入。

### 如何管理 Claude Code 的情境視窗？

不要讓情境超過 60% 的容量。使用「記錄與清除」模式：將進度傾印至 markdown 檔案，執行 `/clear`，然後讀取該檔案後重新開始。每個對話只處理一個任務，將 MCP 伺服器限制在 5-8 個，並使用 `/catchup` 等自訂指令高效重建情境。

### Claude Code 每月費用是多少？

以 API 定價計算，平均成本為每位開發者每天 $6 美元（使用 Sonnet 4.6 約 $100-200/月）。Max 訂閱方案（每月固定 $100 或 $200）為日常使用者提供顯著節省，一位開發者報告在 8 個月的重度使用中，相較於 API 等效成本節省了 93%。

### 什麼是 Claude Code hooks，何時應該使用它們？

Hooks 是在 Claude 工作流程特定時間點執行的 shell 腳本。使用 PreToolUse hooks 封鎖危險動作（退出代碼 2 會封鎖並重新導向），使用 PostToolUse hooks 提供非封鎖性回饋（例如 linting）。它們將合規性差距從僅使用 CLAUDE.md 的約 70% 提升至關鍵安全規則的 100%。
