# Claude Code 最佳實踐

**來源：** [https://www.anthropic.com/engineering/claude-code-best-practices](https://www.anthropic.com/engineering/claude-code-best-practices)（重新導向至 [https://code.claude.com/docs/en/best-practices](https://code.claude.com/docs/en/best-practices)）

---

> 從環境設定到跨平行工作階段擴展，提升 Claude Code 使用效益的技巧與模式。

Claude Code 是一個代理式編碼環境。它不像聊天機器人只回答問題然後等待，Claude Code 能夠讀取你的檔案、執行指令、進行修改，並在你觀看、調整方向或完全離開時自主解決問題。

這改變了你的工作方式。你不再需要自己寫程式碼再請 Claude 審查，而是描述你想要什麼，讓 Claude 自己想辦法實現。Claude 會自行探索、規劃並實作。

但這種自主性仍有學習曲線。Claude 的運作有一些你需要了解的限制。

本指南涵蓋了在 Anthropic 內部團隊及使用各種程式碼庫、語言和環境的工程師中已驗證有效的模式。關於代理迴圈如何在底層運作，請參閱 [Claude Code 運作原理](https://code.claude.com/docs/en/how-claude-code-works)。

---

大多數最佳實踐都基於一個限制：Claude 的上下文視窗會快速填滿，而效能會隨著填充程度下降。

Claude 的上下文視窗保存了你的整個對話，包括每則訊息、每個 Claude 讀取的檔案，以及每條指令的輸出。然而，這可能很快就填滿。一個除錯工作階段或程式碼庫探索可能會產生並消耗數萬個 token。

這很重要，因為 LLM 效能會隨著上下文填充而下降。當上下文視窗快滿時，Claude 可能會開始「遺忘」早期指令或犯更多錯誤。上下文視窗是最重要的資源管理對象。使用自訂狀態列持續追蹤上下文使用情況，並參閱「減少 token 使用量」以了解降低 token 使用量的策略。

## 給 Claude 一種驗證其工作的方式

> 提供測試、螢幕截圖或預期輸出，讓 Claude 能夠自我檢驗。這是你能做的最具槓桿效應的一件事。

當 Claude 能夠驗證自己的工作時，例如執行測試、比較螢幕截圖、驗證輸出，其表現會大幅提升。

若沒有明確的成功標準，它可能會產生看起來正確但實際上不能運作的東西。你將成為唯一的回饋迴圈，每個錯誤都需要你的注意。

| 策略 | 之前 | 之後 |
|---|---|---|
| **提供驗證標準** | *「實作一個驗證電子郵件地址的函式」* | *「撰寫一個 validateEmail 函式。測試案例範例：user@example.com 為 true、invalid 為 false、user@.com 為 false。實作後執行測試」* |
| **視覺驗證 UI 變更** | *「讓儀表板看起來更好」* | *「[貼上螢幕截圖] 實作這個設計。拍攝結果的螢幕截圖並與原始版本比較。列出差異並修正它們」* |
| **處理根本原因，而非症狀** | *「建置失敗了」* | *「建置失敗並出現此錯誤：[貼上錯誤訊息]。修復它並驗證建置成功。處理根本原因，不要壓制錯誤」* |

UI 變更可以使用 Chrome 版 Claude 擴充套件來驗證。它會在你的瀏覽器中開啟新分頁，測試 UI，並反覆迭代直到程式碼正常運作。

你的驗證也可以是測試套件、linter，或是檢查輸出的 Bash 指令。請投入心力讓你的驗證機制非常可靠。

## 先探索，再規劃，然後編寫程式碼

> 將研究和規劃與實作分開，以避免解決錯誤的問題。

讓 Claude 直接跳到編碼可能會產生解決錯誤問題的程式碼。使用計劃模式（Plan Mode）將探索與執行分開。

建議的工作流程分為四個階段：

### 步驟一：探索

進入計劃模式。Claude 會讀取檔案並回答問題，而不進行任何變更。

```
claude (計劃模式)
閱讀 /src/auth 並了解我們如何處理工作階段與登入。
同時查看我們如何管理機密的環境變數。
```

### 步驟二：規劃

請 Claude 建立詳細的實作計畫。

```
claude (計劃模式)
我想新增 Google OAuth。哪些檔案需要修改？
工作階段流程是什麼？請建立一個計畫。
```

按下 `Ctrl+G` 在文字編輯器中開啟計畫，可在 Claude 繼續之前直接編輯。

### 步驟三：實作

切換回一般模式，讓 Claude 編寫程式碼，並依據其計畫進行驗證。

```
claude (一般模式)
根據你的計畫實作 OAuth 流程。為回呼處理程式撰寫測試，
執行測試套件並修復所有失敗。
```

### 步驟四：提交

請 Claude 使用描述性訊息提交並建立 PR。

```
claude (一般模式)
使用描述性訊息提交並開啟 PR
```

> **注意：** 計劃模式很有用，但也會增加額外開銷。對於範圍明確且修改較小的任務（例如修正錯字、新增日誌行或重新命名變數），可直接請 Claude 執行。當你不確定方法、變更涉及多個檔案，或你不熟悉被修改的程式碼時，規劃最為有用。如果你能用一句話描述 diff，就跳過計畫。

## 在提示詞中提供具體上下文

> 你的指令越精確，需要的修正就越少。

Claude 可以推斷意圖，但無法讀取你的想法。請參照具體檔案、提及限制條件，並指向範例模式。

| 策略 | 之前 | 之後 |
|---|---|---|
| **界定任務範圍。** 指定哪個檔案、什麼場景，以及測試偏好。 | *「為 foo.py 新增測試」* | *「為 foo.py 撰寫測試，涵蓋使用者已登出的邊緣情況。避免使用模擬（mock）。」* |
| **指向來源。** 引導 Claude 前往能回答問題的來源。 | *「為什麼 ExecutionFactory 的 API 這麼奇怪？」* | *「查閱 ExecutionFactory 的 git 歷史記錄，並總結其 API 是如何演變的」* |
| **參照現有模式。** 指出你程式碼庫中的模式給 Claude。 | *「新增一個日曆小工具」* | *「查看首頁現有小工具的實作方式以了解其模式。HotDogWidget.php 是一個很好的範例。遵循該模式實作一個新的日曆小工具，讓使用者可以選擇月份並向前/向後翻頁選擇年份。從頭開始建置，不使用程式碼庫中現有函式庫以外的其他函式庫。」* |
| **描述症狀。** 提供症狀、可能的位置，以及「已修復」的樣子。 | *「修復登入錯誤」* | *「使用者回報工作階段逾時後登入失敗。請檢查 src/auth/ 中的認證流程，特別是 token 更新的部分。撰寫能重現問題的失敗測試，然後修復它」* |

當你正在探索且有餘裕修正方向時，模糊的提示詞是有用的。像 `「你會如何改善這個檔案？」` 這樣的提示詞，可能會發現你從未想到要問的問題。

### 提供豐富的內容

> 使用 `@` 來參照檔案、貼上螢幕截圖/圖片，或直接輸送資料。

你可以透過多種方式向 Claude 提供豐富資料：

- **使用 `@` 參照檔案**，而非描述程式碼的位置。Claude 會在回應前讀取該檔案。
- **直接貼上圖片**。複製/貼上或拖放圖片到提示詞中。
- **提供 URL** 以供文件和 API 參考使用。使用 `/permissions` 將常用網域加入白名單。
- **輸送資料**，執行 `cat error.log | claude` 直接傳送檔案內容。
- **讓 Claude 自行取得所需資料**。告訴 Claude 使用 Bash 指令、MCP 工具或讀取檔案來自行取得上下文。

## 設定你的環境

幾個設定步驟能讓 Claude Code 在所有工作階段中大幅提升效能。關於擴充功能及各項使用時機的完整概覽，請參閱「擴充 Claude Code」。

### 撰寫有效的 CLAUDE.md

> 執行 `/init` 根據目前的專案結構產生初始 CLAUDE.md 檔案，然後隨時間精煉。

CLAUDE.md 是一個特殊檔案，Claude 在每次對話開始時都會讀取它。其中包含 Bash 指令、程式碼風格和工作流程規則。這讓 Claude 擁有僅靠程式碼無法推斷的持久性上下文。

`/init` 指令會分析你的程式碼庫，偵測建置系統、測試框架和程式碼模式，為你提供良好的起點。

CLAUDE.md 檔案沒有固定格式，但請保持簡短且易於閱讀。例如：

```markdown
# 程式碼風格
- 使用 ES 模組（import/export）語法，而非 CommonJS（require）
- 盡可能使用解構匯入（例如 import { foo } from 'bar'）

# 工作流程
- 完成一系列程式碼變更後，務必進行型別檢查
- 盡量執行單一測試，而非整個測試套件，以提升效能
```

CLAUDE.md 每次工作階段都會載入，因此只包含廣泛適用的內容。對於只有時才相關的領域知識或工作流程，請改用技能（skills）。Claude 會依需求載入，不會讓每次對話都臃腫。

保持簡潔。對於每一行，問自己：*「刪除這一行會讓 Claude 犯錯嗎？」* 如果不會，就刪掉它。臃腫的 CLAUDE.md 會讓 Claude 忽略你的實際指令！

| 應包含 | 應排除 |
|---|---|
| Claude 無法猜測的 Bash 指令 | Claude 能從程式碼推斷的任何內容 |
| 與預設值不同的程式碼風格規則 | Claude 已知的標準語言慣例 |
| 測試指令與偏好的測試執行器 | 詳細的 API 文件（改為連結至文件） |
| 儲存庫規範（分支命名、PR 慣例） | 頻繁變動的資訊 |
| 專案特定的架構決策 | 冗長的說明或教學 |
| 開發環境的特殊情況（必要的環境變數） | 程式碼庫的逐檔描述 |
| 常見陷阱或非直觀的行為 | 不言而喻的實踐，例如「撰寫乾淨的程式碼」 |

如果 Claude 持續做你不想要的事，儘管有規則明令禁止，那麼檔案可能太長，導致規則迷失其中。如果 Claude 問了在 CLAUDE.md 中已有答案的問題，措辭可能不夠明確。像對待程式碼一樣對待 CLAUDE.md：出錯時審查它、定期精簡，並透過觀察 Claude 的行為是否真正改變來測試修改成效。

你可以加入強調語氣（例如「重要」或「你必須」）來提升遵循度。將 CLAUDE.md 提交至 git，讓你的團隊可以共同維護。這個檔案的價值會隨時間累積。

CLAUDE.md 檔案可以使用 `@路徑/到/匯入檔案` 語法匯入其他檔案：

```markdown
請參閱 @README.md 了解專案概覽，以及 @package.json 了解可用的 npm 指令。

# 額外指令
- Git 工作流程：@docs/git-instructions.md
- 個人覆蓋設定：@~/.claude/my-project-instructions.md
```

你可以將 CLAUDE.md 檔案放在幾個位置：

- **主目錄（`~/.claude/CLAUDE.md`）**：套用至所有 Claude 工作階段
- **專案根目錄（`./CLAUDE.md`）**：提交至 git 以與團隊共用
- **上層目錄**：適用於 monorepo，`root/CLAUDE.md` 和 `root/foo/CLAUDE.md` 都會自動載入
- **子目錄**：當 Claude 處理這些目錄中的檔案時，會依需求載入子目錄的 CLAUDE.md

### 設定權限

> 使用 `/permissions` 將安全指令加入白名單，或使用 `/sandbox` 進行作業系統層級的隔離。這能減少中斷，同時讓你保持控制。

預設情況下，Claude Code 會針對可能修改系統的動作請求許可：檔案寫入、Bash 指令、MCP 工具等。這很安全但很繁瑣。第十次核准之後，你實際上已經不在審查了，只是在點擊確認。有兩種方式可以減少這些中斷：

- **權限白名單**：允許你知道安全的特定工具（例如 `npm run lint` 或 `git commit`）
- **沙盒隔離**：啟用作業系統層級的隔離，限制檔案系統和網路存取，讓 Claude 能在定義的邊界內更自由地工作

或者，使用 `--dangerously-skip-permissions` 跳過受控工作流程（如修復 lint 錯誤或產生樣板程式碼）的權限提示。請參閱權限模式以了解哪些內容會被跳過，哪些不會。

> **警告：** 讓 Claude 執行任意指令可能導致資料遺失、系統損壞，或透過提示詞注入造成資料外洩。只在沒有網際網路存取的沙盒環境中使用 `--dangerously-skip-permissions`。

請閱讀更多關於設定權限和啟用沙盒的資訊。

### 使用 CLI 工具

> 告訴 Claude Code 在與外部服務互動時使用 CLI 工具，例如 `gh`、`aws`、`gcloud` 和 `sentry-cli`。

CLI 工具是與外部服務互動最節省上下文的方式。如果你使用 GitHub，請安裝 `gh` CLI。Claude 知道如何使用它來建立 issue、開啟 pull request 和讀取評論。沒有 `gh` 的話，Claude 仍然可以使用 GitHub API，但未認證的請求經常會遇到速率限制。

Claude 也很擅長學習它尚不熟悉的 CLI 工具。試試像這樣的提示詞：`使用 'foo-cli-tool --help' 了解 foo 工具，然後用它來解決 A、B、C。`

### 連接 MCP 伺服器

> 執行 `claude mcp add` 連接外部工具，例如 Notion、Figma 或你的資料庫。

透過 MCP 伺服器，你可以請 Claude 從 issue 追蹤器實作功能、查詢資料庫、分析監控資料、整合 Figma 設計，以及自動化工作流程。

### 設定 hooks

> 對於必須每次都執行、零例外的動作，請使用 hooks。

Hooks 會在 Claude 工作流程的特定時間點自動執行腳本。與 CLAUDE.md 指令（屬於建議性質）不同，hooks 是確定性的，能保證動作一定會發生。

Claude 可以幫你撰寫 hooks。試試像這樣的提示詞：*「撰寫一個在每次檔案編輯後執行 eslint 的 hook」* 或 *「撰寫一個阻止寫入 migrations 資料夾的 hook。」* 直接編輯 `.claude/settings.json` 來手動設定 hooks，並執行 `/hooks` 瀏覽已設定的內容。

### 建立技能

> 在 `.claude/skills/` 中建立 `SKILL.md` 檔案，為 Claude 提供領域知識和可重複使用的工作流程。

技能以特定於你的專案、團隊或領域的資訊擴充 Claude 的知識。Claude 會在相關時自動套用它們，或者你可以使用 `/技能名稱` 直接呼叫。

透過在 `.claude/skills/` 中新增含有 `SKILL.md` 的目錄來建立技能：

```markdown
# .claude/skills/api-conventions/SKILL.md
---
name: api-conventions
description: 我們服務的 REST API 設計慣例
---
# API 慣例
- URL 路徑使用 kebab-case
- JSON 屬性使用 camelCase
- 列表端點一律包含分頁
- API 在 URL 路徑中加入版本（/v1/、/v2/）
```

技能也可以定義可直接呼叫的可重複工作流程：

```markdown
# .claude/skills/fix-issue/SKILL.md
---
name: fix-issue
description: 修復一個 GitHub issue
disable-model-invocation: true
---
分析並修復 GitHub issue：$ARGUMENTS。

1. 使用 `gh issue view` 取得 issue 詳情
2. 了解 issue 中描述的問題
3. 搜尋程式碼庫中的相關檔案
4. 實作必要的變更以修復問題
5. 撰寫並執行測試以驗證修復
6. 確保程式碼通過 linting 和型別檢查
7. 建立描述性的提交訊息
8. 推送並建立 PR
```

執行 `/fix-issue 1234` 來呼叫它。對於有副作用且想要手動觸發的工作流程，請使用 `disable-model-invocation: true`。

### 建立自訂子代理

> 在 `.claude/agents/` 中定義專門的助理，Claude 可以將其委派用於隔離任務。

子代理在自己的上下文中執行，並擁有自己允許的工具集。它們適用於需要讀取大量檔案或需要專注處理、不想污染主要對話的任務。

```markdown
# .claude/agents/security-reviewer.md
---
name: security-reviewer
description: 審查程式碼中的安全漏洞
tools: Read, Grep, Glob, Bash
model: opus
---
你是一位資深安全工程師。審查程式碼中的：
- 注入漏洞（SQL、XSS、指令注入）
- 認證和授權缺陷
- 程式碼中的機密或憑證
- 不安全的資料處理

提供具體的行號參照和建議的修復方案。
```

明確告訴 Claude 使用子代理：*「使用子代理審查這段程式碼的安全問題。」*

### 安裝插件

> 執行 `/plugin` 瀏覽市集。插件無需設定即可新增技能、工具和整合功能。

插件將技能、hooks、子代理和 MCP 伺服器打包成一個可從社群和 Anthropic 安裝的單一單元。如果你使用靜態型別語言，請安裝程式碼智慧插件，為 Claude 提供精確的符號導覽和編輯後的自動錯誤偵測。

關於如何在技能、子代理、hooks 和 MCP 之間做選擇的指引，請參閱「擴充 Claude Code」。

## 有效溝通

與 Claude Code 的溝通方式會顯著影響結果品質。

### 詢問程式碼庫問題

> 像詢問資深工程師一樣詢問 Claude。

在熟悉新程式碼庫時，使用 Claude Code 進行學習和探索。你可以問 Claude 和問其他工程師相同類型的問題：

- 日誌記錄是如何運作的？
- 如何建立新的 API 端點？
- `foo.rs` 第 134 行的 `async move { ... }` 是什麼意思？
- `CustomerOnboardingFlowImpl` 處理了哪些邊緣情況？
- 為什麼這段程式碼在第 333 行呼叫 `foo()` 而不是 `bar()`？

這樣使用 Claude Code 是一種有效的新人上手工作流程，能改善學習速度並減輕其他工程師的負擔。不需要特殊的提示詞：直接提問即可。

### 讓 Claude 訪談你

> 對於較大的功能，先讓 Claude 訪談你。從最精簡的提示詞開始，請 Claude 使用 `AskUserQuestion` 工具訪談你。

Claude 會詢問你可能尚未考慮到的事情，包括技術實作、UI/UX、邊緣情況和取捨。

```
我想建置 [簡短描述]。請使用 AskUserQuestion 工具詳細訪談我。

詢問關於技術實作、UI/UX、邊緣情況、顧慮和取捨的問題。不要問顯而易見的問題，深入探討我可能沒有考慮到的困難部分。

持續訪談直到我們涵蓋所有內容，然後將完整規格寫入 SPEC.md。
```

規格完成後，開啟一個新的工作階段來執行它。新工作階段有乾淨的上下文，完全專注於實作，而你有一份書面規格可以參考。

## 管理你的工作階段

對話是持久的且可撤銷的。善用這一點！

### 盡早且頻繁地修正方向

> 一發現 Claude 偏離軌道就立即修正。

緊密的回饋迴圈能帶來最好的結果。雖然 Claude 偶爾能在第一次嘗試就完美解決問題，但快速修正通常能更快產生更好的解決方案。

- **`Esc`**：用 `Esc` 鍵中途停止 Claude。上下文會保留，因此你可以重新引導。
- **`Esc + Esc` 或 `/rewind`**：按兩次 `Esc` 或執行 `/rewind` 開啟回溯選單，恢復先前的對話和程式碼狀態，或從選定訊息開始摘要。
- **`「撤銷那個」`**：讓 Claude 還原其變更。
- **`/clear`**：在不相關的任務之間重置上下文。帶有不相關上下文的長工作階段可能會降低效能。

如果你在同一個工作階段中對同一問題修正了 Claude 超過兩次，上下文就充斥著失敗的方法。執行 `/clear` 重新開始，並使用更具體的提示詞，融入你學到的內容。帶有更好提示詞的乾淨工作階段，幾乎總是優於帶有累積修正的長工作階段。

### 積極管理上下文

> 在不相關的任務之間執行 `/clear` 以重置上下文。

當你接近上下文限制時，Claude Code 會自動壓縮對話歷史記錄，在釋放空間的同時保留重要的程式碼和決策。

在長工作階段中，Claude 的上下文視窗可能會充滿不相關的對話、檔案內容和指令。這可能會降低效能，有時還會分散 Claude 的注意力。

- 在任務之間頻繁使用 `/clear` 以完全重置上下文視窗
- 當自動壓縮觸發時，Claude 會摘要最重要的內容，包括程式碼模式、檔案狀態和關鍵決策
- 若需要更多控制，執行 `/compact <指令>`，例如 `/compact 專注於 API 變更`
- 若只想壓縮部分對話，使用 `Esc + Esc` 或 `/rewind`，選擇一個訊息檢查點，然後選擇**從這裡摘要**。這會壓縮從該點往後的訊息，同時保留早期上下文完整。
- 在 CLAUDE.md 中使用指令自訂壓縮行為，例如 `「壓縮時，務必保留修改過的完整檔案清單和任何測試指令」`，以確保關鍵上下文在摘要後得以保留
- 對於不需要留在上下文中的快速問題，使用 `/btw`。答案會出現在可關閉的浮動視窗中，不會進入對話歷史記錄，因此你可以查看細節而不增加上下文。

### 使用子代理進行調查

> 使用「使用子代理調查 X」進行委派研究。它們在獨立上下文中探索，讓你的主要對話保持乾淨以便實作。

由於上下文是你的基本限制，子代理是最強大的工具之一。當 Claude 研究程式碼庫時，它會讀取大量檔案，這些都會消耗你的上下文。子代理在獨立的上下文視窗中執行並回報摘要：

```
使用子代理調查我們的認證系統如何處理 token 更新，
以及我們是否有任何我應該重複使用的現有 OAuth 工具。
```

子代理會探索程式碼庫、讀取相關檔案，並回報結果，這一切都不會污染你的主要對話。

你也可以在 Claude 實作某些東西後，使用子代理進行驗證：

```
使用子代理審查這段程式碼的邊緣情況
```

### 使用檢查點回溯

> Claude 每次動作都會建立一個檢查點。你可以將對話、程式碼或兩者一起還原到任何先前的檢查點。

Claude 在變更前會自動建立檢查點。按兩下 `Escape` 或執行 `/rewind` 開啟回溯選單。你可以只還原對話、只還原程式碼、兩者都還原，或從選定訊息開始摘要。詳情請參閱「檢查點」。

你不必仔細規劃每一步，可以告訴 Claude 嘗試有風險的事情。如果不成功，回溯並嘗試不同的方法。檢查點會跨工作階段持續存在，因此你可以關閉終端機，稍後仍可回溯。

> **警告：** 檢查點只追蹤 *Claude* 所做的變更，而非外部程序。這不能替代 git。

### 恢復對話

> 執行 `claude --continue` 從上次離開的地方繼續，或使用 `--resume` 從最近的工作階段中選擇。

Claude Code 會在本機儲存對話。當任務跨越多個工作階段時，你不必重新解釋上下文：

```bash
claude --continue    # 恢復最近的對話
claude --resume      # 從最近的對話中選擇
```

使用 `/rename` 為工作階段取描述性名稱，例如 `「oauth-migration」` 或 `「debugging-memory-leak」`，以便日後查找。將工作階段視為分支：不同的工作流可以有各自獨立且持久的上下文。

## 自動化與擴展

一旦你能有效地使用一個 Claude，就可以透過平行工作階段、非互動模式和扇出模式來倍增你的產出。

到目前為止，所有內容都假設一個人、一個 Claude 和一個對話。但 Claude Code 可以水平擴展。本節的技術展示了你如何完成更多工作。

### 執行非互動模式

> 在 CI、pre-commit hooks 或腳本中使用 `claude -p "提示詞"`。新增 `--output-format stream-json` 以獲得串流 JSON 輸出。

使用 `claude -p "你的提示詞"`，你可以非互動式地執行 Claude，不需要工作階段。非互動模式是將 Claude 整合至 CI 管道、pre-commit hooks 或任何自動化工作流程的方式。輸出格式讓你能夠以程式方式解析結果：純文字、JSON 或串流 JSON。

```bash
# 一次性查詢
claude -p "解釋這個專案做什麼"

# 腳本的結構化輸出
claude -p "列出所有 API 端點" --output-format json

# 即時處理的串流輸出
claude -p "分析這個日誌檔案" --output-format stream-json
```

### 執行多個 Claude 工作階段

> 平行執行多個 Claude 工作階段以加速開發、執行隔離實驗或啟動複雜工作流程。

執行平行工作階段有三種主要方式：

- **Claude Code 桌面應用程式**：視覺化管理多個本機工作階段。每個工作階段都有自己的隔離工作樹。
- **網頁版 Claude Code**：在 Anthropic 安全雲端基礎設施的隔離 VM 中執行。
- **代理團隊**：透過共享任務、訊息傳遞和團隊領導者，自動協調多個工作階段。

除了平行化工作之外，多個工作階段還能實現以品質為核心的工作流程。新鮮的上下文能改善程式碼審查，因為 Claude 不會對它剛寫的程式碼有偏見。

例如，使用「撰寫者/審查者」模式：

| 工作階段 A（撰寫者） | 工作階段 B（審查者） |
|---|---|
| `為我們的 API 端點實作速率限制器` | |
| | `審查 @src/middleware/rateLimiter.ts 中的速率限制器實作。尋找邊緣情況、競態條件，以及與現有中介軟體模式的一致性。` |
| `以下是審查回饋：[工作階段 B 輸出]。請解決這些問題。` | |

你可以用測試做類似的事情：讓一個 Claude 撰寫測試，再讓另一個撰寫程式碼來通過它們。

### 跨檔案扇出

> 迴圈遍歷任務並為每個任務呼叫 `claude -p`。使用 `--allowedTools` 為批次操作界定權限範圍。

對於大規模遷移或分析，你可以將工作分散到多個平行的 Claude 呼叫：

**步驟一：產生任務清單**

讓 Claude 列出所有需要遷移的檔案（例如，`列出所有 2,000 個需要遷移的 Python 檔案`）

**步驟二：撰寫腳本遍歷清單**

```bash
for file in $(cat files.txt); do
  claude -p "將 $file 從 React 遷移至 Vue。回傳 OK 或 FAIL。" \
    --allowedTools "Edit,Bash(git commit *)"
done
```

**步驟三：在幾個檔案上測試，然後大規模執行**

根據前 2-3 個檔案出現的問題精煉你的提示詞，然後在完整集合上執行。`--allowedTools` 標記限制了 Claude 能做的事情，這在無人監管執行時很重要。

你也可以將 Claude 整合至現有的資料/處理管道：

```bash
claude -p "<你的提示詞>" --output-format json | your_command
```

在開發過程中使用 `--verbose` 進行除錯，在生產環境中關閉它。

## 避免常見的失敗模式

這些是常見的錯誤。及早認識它們可以節省時間：

- **廚房水槽式工作階段。** 你從一個任務開始，然後問了一個不相關的問題，然後又回到第一個任務。上下文充滿了不相關的資訊。
  > **解決方案**：在不相關的任務之間使用 `/clear`。

- **反覆修正。** Claude 做錯了某件事，你修正它，它還是錯的，你再次修正。上下文被失敗的方法污染了。
  > **解決方案**：兩次失敗修正後，使用 `/clear` 並撰寫一個更好的初始提示詞，融入你學到的內容。

- **過度詳細的 CLAUDE.md。** 如果你的 CLAUDE.md 太長，Claude 會忽略其中一半，因為重要的規則迷失在雜訊中。
  > **解決方案**：無情地精簡。如果 Claude 在沒有該指令的情況下已經能正確執行某件事，就刪除它或將其轉換為 hook。

- **信任後驗證的落差。** Claude 產生了看似合理的實作，但沒有處理邊緣情況。
  > **解決方案**：始終提供驗證（測試、腳本、螢幕截圖）。如果你無法驗證，就不要交付。

- **無限探索。** 你請 Claude「調查」某件事而沒有限定範圍。Claude 讀取了數百個檔案，填滿了上下文。
  > **解決方案**：縮小調查範圍，或使用子代理，這樣探索就不會消耗你的主要上下文。

## 培養你的直覺

本指南中的模式並非一成不變。它們是在一般情況下效果良好的起點，但可能不是每種情況的最佳選擇。

有時候你*應該*讓上下文累積，因為你深陷於一個複雜問題，而歷史記錄很有價值。有時候你應該跳過規劃，讓 Claude 自己解決，因為任務是探索性的。有時候模糊的提示詞恰到好處，因為你想在約束它之前看看 Claude 如何解讀這個問題。

留意什麼有效。當 Claude 產生出色的輸出時，注意你做了什麼：提示詞結構、你提供的上下文、你所處的模式。當 Claude 遇到困難時，問問為什麼。是上下文太嘈雜？提示詞太模糊？任務對一次處理來說太大？

隨著時間推移，你將培養出沒有任何指南能夠捕捉的直覺。你將知道何時要具體、何時要開放，何時要規劃、何時要探索，何時要清除上下文、何時要讓它累積。
