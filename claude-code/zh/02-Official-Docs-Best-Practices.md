# Claude Code 最佳實踐

> 來源：https://code.claude.com/docs/en/best-practices

> 從配置環境到跨平行會話擴展規模，善用 Claude Code 的技巧與模式。

Claude Code 是一個代理式程式設計環境。不同於回答問題後靜待指令的聊天機器人，Claude Code 能夠讀取檔案、執行指令、進行修改，並在你旁觀、指引或完全離開的情況下自主解決問題。

這改變了你的工作方式。你不再需要自己寫程式再請 Claude 審查，而是描述你想要什麼，由 Claude 想出如何實作。Claude 會自行探索、規劃並實作。

但這種自主性仍有一段學習曲線。Claude 在特定限制內運作，你需要了解這些限制。

本指南涵蓋了在 Anthropic 內部團隊及各種程式碼庫、語言和環境中經過驗證的有效模式。關於代理循環的底層運作原理，請參閱 [Claude Code 的運作方式](/en/how-claude-code-works)。

---

大多數最佳實踐都源於一個限制：Claude 的上下文視窗會快速填滿，填滿後效能會下降。

Claude 的上下文視窗保存了整個對話內容，包括每一則訊息、Claude 讀取的每個檔案，以及每個指令的輸出。然而，這可能很快就會填滿。一次除錯工作階段或程式碼庫探索可能會產生並消耗數萬個 token。

這很重要，因為 LLM 效能會隨著上下文填滿而下降。當上下文視窗快滿時，Claude 可能開始「遺忘」較早的指令或犯更多錯誤。上下文視窗是最重要的資源，需要持續管理。透過自訂狀態列持續追蹤上下文使用情況，並參閱「減少 token 使用量」以了解相關策略。

## 讓 Claude 有辦法驗證自身的工作

> 提供測試、截圖或預期輸出，讓 Claude 能夠自我檢查。這是你能做到的單一最高槓桿行動。

當 Claude 能夠驗證自身的工作時（例如執行測試、比較截圖、驗證輸出），表現會大幅提升。

若沒有明確的成功標準，Claude 可能會產出看似正確但實際上無法運作的結果。你會成為唯一的反饋迴路，每一個錯誤都需要你的注意。

| 策略 | 改善前 | 改善後 |
| --- | --- | --- |
| **提供驗證標準** | *「實作一個驗證電子郵件地址的函式」* | *「撰寫 validateEmail 函式。範例測試案例：user@example.com 應為 true，invalid 應為 false，user@.com 應為 false。實作後請執行測試」* |
| **以視覺方式驗證 UI 變更** | *「讓儀表板看起來更好」* | *「[貼上截圖] 實作這個設計。截取結果截圖並與原始版本比較。列出差異並修正」* |
| **處理根本原因，而非症狀** | *「建置失敗了」* | *「建置失敗並出現此錯誤：[貼上錯誤]。修正它並確認建置成功。處理根本原因，不要壓制錯誤」* |

UI 變更可以使用 Claude in Chrome 擴充功能進行驗證。它會在瀏覽器中開啟新分頁、測試 UI，並反覆迭代直到程式碼正常運作。

你的驗證方式也可以是測試套件、Linter 或檢查輸出的 Bash 指令。投資讓你的驗證方式更加穩固。

## 先探索、再規劃、後撰碼

> 將研究規劃與實作分開，避免解決錯誤的問題。

讓 Claude 直接跳到寫程式可能會產出解決了錯誤問題的程式碼。使用規劃模式（Plan Mode）將探索與執行分開。

建議的工作流程分為四個階段：

### 第一步：探索

進入規劃模式。Claude 讀取檔案並回答問題，但不進行任何修改。

```
claude (Plan Mode)
讀取 /src/auth 並了解我們如何處理工作階段和登入。
同時查看我們如何管理密鑰的環境變數。
```

### 第二步：規劃

請 Claude 建立詳細的實作計畫。

```
claude (Plan Mode)
我想新增 Google OAuth。哪些檔案需要修改？
工作階段流程是什麼？請建立一個計畫。
```

按下 `Ctrl+G` 可在文字編輯器中開啟計畫，在 Claude 繼續之前直接進行編輯。

### 第三步：實作

切換回一般模式，讓 Claude 撰碼，並對照計畫進行驗證。

```
claude (Normal Mode)
依據你的計畫實作 OAuth 流程。為回呼處理器撰寫測試，
執行測試套件並修正任何失敗。
```

### 第四步：提交

請 Claude 以描述性訊息提交並建立 PR。

```
claude (Normal Mode)
以描述性訊息提交並開啟 PR
```

> **注意：** 規劃模式很有用，但也會增加額外負擔。對於範圍明確且修改幅度小的任務（例如修正錯字、新增日誌行或重新命名變數），直接請 Claude 執行即可。規劃最適合用於你對做法不確定、變更涉及多個檔案，或你不熟悉正在修改的程式碼時。如果你能用一句話描述 diff，就跳過規劃。

## 在提示中提供具體的上下文

> 指示越精確，需要的修正次數就越少。

Claude 可以推斷意圖，但無法讀取你的心思。請參照特定檔案、提及限制條件，並指出範例模式。

| 策略 | 改善前 | 改善後 |
| --- | --- | --- |
| **限定任務範圍。** 指定哪個檔案、什麼情境以及測試偏好。 | *「為 foo.py 新增測試」* | *「為 foo.py 撰寫測試，涵蓋使用者已登出的邊界案例。避免使用 mock。」* |
| **指向來源。** 將 Claude 引導至可以回答問題的來源。 | *「為什麼 ExecutionFactory 有這麼奇怪的 API？」* | *「查閱 ExecutionFactory 的 git 歷史記錄，總結其 API 的演變過程」* |
| **參考現有模式。** 將 Claude 指向程式碼庫中的模式。 | *「新增日曆小工具」* | *「查看首頁現有小工具的實作方式以了解模式。HotDogWidget.php 是個好例子。依照該模式實作新的日曆小工具，讓使用者能選擇月份並分頁前後翻閱以選取年份。從零開始建置，僅使用程式碼庫中已使用的函式庫。」* |
| **描述症狀。** 提供症狀、可能的位置，以及「已修復」的樣子。 | *「修復登入錯誤」* | *「使用者回報登入在工作階段逾時後失敗。檢查 src/auth/ 中的驗證流程，尤其是 token 更新部分。撰寫能重現問題的失敗測試，然後修正它」* |

當你在探索且可以接受需要修正方向時，模糊的提示也很有用。像「你會改善這個檔案的哪些地方？」這樣的提示，可能會發現你沒想到要問的事情。

### 提供豐富的內容

> 使用 `@` 參照檔案、貼上截圖/圖片，或直接透過管道傳送資料。

你可以透過幾種方式為 Claude 提供豐富的資料：

- **使用 `@` 參照檔案**，而不是描述程式碼在哪裡。Claude 會在回應前讀取該檔案。
- **直接貼上圖片**。透過複製/貼上或拖放方式將圖片放入提示中。
- **提供 URL**，用於文件和 API 參考。使用 `/permissions` 將常用網域加入允許清單。
- **透過管道傳入資料**，執行 `cat error.log | claude` 以直接傳送檔案內容。
- **讓 Claude 自行取得所需內容**。告訴 Claude 使用 Bash 指令、MCP 工具或讀取檔案來自行提取上下文。

## 配置你的環境

幾個設定步驟可以讓 Claude Code 在所有工作階段中大幅提升效能。關於擴充功能的完整概觀及使用時機，請參閱「擴充 Claude Code」。

### 撰寫有效的 CLAUDE.md

> 執行 `/init` 根據當前專案結構生成 CLAUDE.md 的初始版本，然後隨時間持續精煉。

CLAUDE.md 是 Claude 在每次對話開始時讀取的特殊檔案。其中應包含 Bash 指令、程式碼風格和工作流程規則。這讓 Claude 擁有僅憑程式碼無法推斷的持續性上下文。

`/init` 指令會分析你的程式碼庫，偵測建置系統、測試框架和程式碼模式，給你一個穩固的基礎來精煉。

CLAUDE.md 檔案沒有固定格式，但請保持簡短且易於人類閱讀。例如：

```markdown
# 程式碼風格
- 使用 ES 模組（import/export）語法，而非 CommonJS（require）
- 盡可能解構匯入（例如 import { foo } from 'bar'）

# 工作流程
- 完成一系列程式碼修改後，請務必進行型別檢查
- 偏好執行單個測試，而非整個測試套件，以提升效能
```

CLAUDE.md 每次工作階段都會載入，因此只包含廣泛適用的內容。對於只在特定情況下相關的領域知識或工作流程，請改用技能（skills）。Claude 會按需載入它們，而不會讓每次對話都臃腫。

保持簡潔。對每一行問自己：*「移除這行會讓 Claude 犯錯嗎？」* 如果不會，就刪除它。臃腫的 CLAUDE.md 會讓 Claude 忽略你的實際指示！

| 應包含 | 不應包含 |
| --- | --- |
| Claude 無法猜測的 Bash 指令 | Claude 閱讀程式碼就能搞清楚的任何事 |
| 與預設值不同的程式碼風格規則 | Claude 已知的標準語言慣例 |
| 測試指示和偏好的測試執行工具 | 詳細的 API 文件（改為連結至文件） |
| 儲存庫禮儀（分支命名、PR 慣例） | 頻繁變更的資訊 |
| 專案特有的架構決策 | 長篇說明或教學 |
| 開發環境特殊之處（必要的環境變數） | 程式碼庫的逐檔案描述 |
| 常見的陷阱或不明顯的行為 | 不言而喻的做法，例如「撰寫乾淨的程式碼」 |

如果 Claude 一再重複你不希望的行為，儘管已有規則禁止，那麼檔案可能太長，規則被淹沒了。如果 Claude 問了 CLAUDE.md 中已有答案的問題，可能是措辭不夠清晰。把 CLAUDE.md 當作程式碼對待：出問題時審查它、定期精簡，並透過觀察 Claude 的行為是否真的改變來測試修改效果。

你可以在指示中加入強調語氣（例如「重要」或「你必須」）來提高遵守率。將 CLAUDE.md 提交到 git，讓團隊成員也能貢獻。這份檔案的價值會隨時間複利增長。

CLAUDE.md 檔案可以使用 `@path/to/import` 語法匯入其他檔案：

```markdown
專案概覽請參閱 @README.md，可用的 npm 指令請參閱 @package.json。

# 其他指示
- Git 工作流程：@docs/git-instructions.md
- 個人覆蓋設定：@~/.claude/my-project-instructions.md
```

你可以將 CLAUDE.md 檔案放置在幾個位置：

- **家目錄（`~/.claude/CLAUDE.md`）**：套用至所有 Claude 工作階段
- **專案根目錄（`./CLAUDE.md`）**：提交到 git 以與團隊共享
- **父目錄**：適用於 monorepo，`root/CLAUDE.md` 和 `root/foo/CLAUDE.md` 都會自動載入
- **子目錄**：Claude 在處理這些目錄中的檔案時，會按需載入子目錄的 CLAUDE.md 檔案

### 配置權限

> 使用 `/permissions` 將安全指令加入允許清單，或使用 `/sandbox` 進行作業系統層級的隔離。這能減少中斷，同時讓你保持掌控。

預設情況下，Claude Code 會針對可能修改系統的動作請求許可：寫入檔案、執行 Bash 指令、使用 MCP 工具等。這很安全，但很繁瑣。在第十次批准後，你其實不再真正審查，而只是一直點擊通過。有兩種方式可以減少這些中斷：

- **權限允許清單**：允許你知道是安全的特定工具（例如 `npm run lint` 或 `git commit`）
- **沙盒環境**：啟用作業系統層級的隔離，限制檔案系統和網路存取，讓 Claude 能在定義的邊界內更自由地工作

或者，使用 `--dangerously-skip-permissions` 來略過包含工作流程的許可提示，例如修正 Lint 錯誤或生成樣板程式碼。請參閱「權限模式」了解哪些會被略過，哪些不會。

> **警告：** 讓 Claude 執行任意指令可能導致資料遺失、系統損壞，或透過提示注入導致資料外洩。只在沒有網際網路存取的沙盒環境中使用 `--dangerously-skip-permissions`。

閱讀更多關於配置權限和啟用沙盒環境的資訊。

### 使用 CLI 工具

> 告訴 Claude Code 在與外部服務互動時使用 `gh`、`aws`、`gcloud` 和 `sentry-cli` 等 CLI 工具。

CLI 工具是與外部服務互動最節省上下文的方式。如果你使用 GitHub，請安裝 `gh` CLI。Claude 知道如何使用它來建立 issue、開啟 pull request 和閱讀評論。沒有 `gh` 的話，Claude 仍然可以使用 GitHub API，但未驗證的請求往往會碰到速率限制。

Claude 也能有效學習它還不熟悉的 CLI 工具。嘗試像這樣的提示：`使用 'foo-cli-tool --help' 了解 foo 工具，然後用它來解決 A、B、C。`

### 連接 MCP 伺服器

> 執行 `claude mcp add` 連接 Notion、Figma 或你的資料庫等外部工具。

透過 MCP 伺服器，你可以請 Claude 從 issue 追蹤器實作功能、查詢資料庫、分析監控資料、整合 Figma 設計，以及自動化工作流程。

### 設定鉤子（Hooks）

> 對於必須每次都執行、零例外的動作，請使用鉤子。

鉤子會在 Claude 工作流程中的特定時間點自動執行腳本。不同於 CLAUDE.md 中的建議性指示，鉤子是確定性的，能保證動作的執行。

Claude 可以幫你撰寫鉤子。嘗試像這樣的提示：*「撰寫一個在每次檔案編輯後執行 eslint 的鉤子」* 或 *「撰寫一個阻止寫入 migrations 資料夾的鉤子。」* 直接編輯 `.claude/settings.json` 手動配置鉤子，並執行 `/hooks` 瀏覽已配置的項目。

### 建立技能（Skills）

> 在 `.claude/skills/` 中建立 `SKILL.md` 檔案，賦予 Claude 領域知識和可重複使用的工作流程。

技能用你的專案、團隊或領域特有的資訊擴充 Claude 的知識。Claude 會在相關時自動套用它們，或者你可以使用 `/skill-name` 直接呼叫。

透過在 `.claude/skills/` 中新增包含 `SKILL.md` 的目錄來建立技能：

```markdown
# .claude/skills/api-conventions/SKILL.md
---
name: api-conventions
description: 我們服務的 REST API 設計慣例
---
# API 慣例
- URL 路徑使用 kebab-case
- JSON 屬性使用 camelCase
- 列表端點始終包含分頁
- 在 URL 路徑中對 API 進行版本控制（/v1/、/v2/）
```

技能也可以定義你直接呼叫的可重複工作流程：

```markdown
# .claude/skills/fix-issue/SKILL.md
---
name: fix-issue
description: 修復 GitHub issue
disable-model-invocation: true
---
分析並修復 GitHub issue：$ARGUMENTS。

1. 使用 `gh issue view` 取得 issue 詳情
2. 理解 issue 中描述的問題
3. 在程式碼庫中搜索相關檔案
4. 實作必要的修改以修復 issue
5. 撰寫並執行測試以驗證修復
6. 確保程式碼通過 Linting 和型別檢查
7. 建立描述性的提交訊息
8. Push 並建立 PR
```

執行 `/fix-issue 1234` 來呼叫它。對於你想手動觸發且有副作用的工作流程，請使用 `disable-model-invocation: true`。

### 建立自訂子代理（Subagents）

> 在 `.claude/agents/` 中定義專門的助理，Claude 可以委派給它們處理隔離的任務。

子代理在自己的上下文中以自己允許的工具組執行。它們適用於需要讀取大量檔案或需要專注於特定領域，而不希望干擾主對話的任務。

```markdown
# .claude/agents/security-reviewer.md
---
name: security-reviewer
description: 審查程式碼中的安全漏洞
tools: Read, Grep, Glob, Bash
model: opus
---
你是一位資深安全工程師。審查程式碼是否存在：
- 注入漏洞（SQL、XSS、指令注入）
- 驗證和授權缺陷
- 程式碼中的密鑰或憑證
- 不安全的資料處理

提供具體的行號參考和建議的修復方案。
```

明確告訴 Claude 使用子代理：*「使用子代理審查此程式碼是否有安全問題。」*

### 安裝插件（Plugins）

> 執行 `/plugin` 瀏覽市場。插件無需配置即可新增技能、工具和整合。

插件將技能、鉤子、子代理和 MCP 伺服器捆綁成來自社群和 Anthropic 的單一可安裝單元。如果你使用有型別的語言，請安裝程式碼智慧插件，賦予 Claude 精確的符號導航和編輯後的自動錯誤偵測。

關於如何在技能、子代理、鉤子和 MCP 之間做選擇的指導，請參閱「擴充 Claude Code」。

## 有效溝通

你與 Claude Code 的溝通方式會顯著影響結果的品質。

### 詢問程式碼庫問題

> 詢問 Claude 你會問資深工程師的問題。

在熟悉新程式碼庫時，使用 Claude Code 進行學習和探索。你可以問 Claude 與問其他工程師相同類型的問題：

- 日誌記錄是如何運作的？
- 我要如何建立一個新的 API 端點？
- `foo.rs` 第 134 行的 `async move { ... }` 是做什麼的？
- `CustomerOnboardingFlowImpl` 處理了哪些邊界案例？
- 為什麼這段程式碼在第 333 行呼叫 `foo()` 而不是 `bar()`？

以這種方式使用 Claude Code 是一個有效的入職工作流程，能縮短上手時間並減少對其他工程師的依賴。不需要特殊提示：直接提問即可。

### 讓 Claude 訪談你

> 對於較大型的功能，先讓 Claude 訪談你。以最簡短的提示開始，然後請 Claude 使用 `AskUserQuestion` 工具訪談你。

Claude 會詢問你可能還沒考慮到的事情，包括技術實作、UI/UX、邊界案例和取捨。

```
我想建立 [簡短描述]。請使用 AskUserQuestion 工具詳細訪談我。

詢問技術實作、UI/UX、邊界案例、顧慮和取捨。不要問顯而易見的問題，深入探討我可能沒有考慮到的困難之處。

持續訪談直到我們涵蓋了所有內容，然後將完整的規格寫入 SPEC.md。
```

規格完成後，開啟一個新工作階段來執行它。新工作階段有完全專注於實作的乾淨上下文，而你也有書面規格可以參考。

## 管理你的工作階段

對話是持久且可還原的。好好利用這一點！

### 及早且頻繁地修正方向

> 一旦發現 Claude 偏離方向，立即修正。

最好的結果來自緊密的反饋迴路。雖然 Claude 偶爾能在第一次嘗試時完美解決問題，但快速修正通常能更快地產生更好的解決方案。

- **`Esc`**：使用 `Esc` 鍵在 Claude 執行到一半時停止。上下文會被保留，因此你可以重新引導。
- **`Esc + Esc` 或 `/rewind`**：按兩次 `Esc` 或執行 `/rewind` 開啟倒帶選單，還原之前的對話和程式碼狀態，或從選定的訊息開始摘要。
- **`「復原那個」`**：讓 Claude 還原其修改。
- **`/clear`**：在不相關的任務之間重置上下文。含有不相關上下文的長工作階段可能會降低效能。

如果你在同一個工作階段中對同一個問題修正了 Claude 超過兩次，上下文已被失敗的嘗試污染。執行 `/clear` 並以更具體的提示重新開始，融入你學到的內容。清潔的工作階段加上更好的提示，幾乎總是優於帶有累積修正的長工作階段。

### 積極管理上下文

> 在不相關的任務之間執行 `/clear` 以重置上下文。

Claude Code 在你接近上下文限制時會自動壓縮對話歷史，保留重要的程式碼和決策，同時釋放空間。

在長時間工作階段中，Claude 的上下文視窗可能充滿不相關的對話、檔案內容和指令。這可能降低效能，有時還會分散 Claude 的注意力。

- 在任務之間頻繁使用 `/clear` 以完全重置上下文視窗
- 當自動壓縮觸發時，Claude 會摘要最重要的內容，包括程式碼模式、檔案狀態和關鍵決策
- 想要更多控制，可執行 `/compact <指示>`，例如 `/compact 專注於 API 修改`
- 若只想壓縮對話的一部分，使用 `Esc + Esc` 或 `/rewind`，選擇一個訊息檢查點，然後選擇**從這裡摘要**。這會壓縮從該點往後的訊息，同時保留較早的上下文。
- 在 CLAUDE.md 中自訂壓縮行為，例如加入 `「壓縮時，始終保留已修改檔案的完整列表及任何測試指令」`，以確保關鍵上下文在摘要後仍能保留
- 對於不需要留在上下文中的快速問題，使用 `/btw`。答案會以可關閉的浮動視窗顯示，且永遠不會進入對話歷史，讓你能查看細節而不增加上下文。

### 使用子代理進行調查

> 使用「use subagents to investigate X」委派研究工作。子代理在獨立的上下文中探索，讓你的主對話保持乾淨以供實作使用。

由於上下文是你的根本限制，子代理是最強大的工具之一。當 Claude 研究程式碼庫時，它會讀取大量檔案，這些都會消耗你的上下文。子代理在獨立的上下文視窗中執行並回報摘要：

```
使用子代理調查我們的驗證系統如何處理 token 更新，
以及我們是否有任何我應該重複使用的現有 OAuth 工具。
```

子代理會探索程式碼庫、讀取相關檔案，並回報發現，而不會干擾你的主對話。

你也可以在 Claude 實作某些內容後使用子代理進行驗證：

```
使用子代理審查此程式碼的邊界案例
```

### 使用檢查點倒帶

> Claude 的每個動作都會建立一個檢查點。你可以將對話、程式碼或兩者都還原到任何之前的檢查點。

Claude 會在進行修改前自動建立檢查點。雙擊 `Escape` 或執行 `/rewind` 開啟倒帶選單。你可以只還原對話、只還原程式碼、兩者都還原，或從選定的訊息開始摘要。詳情請參閱「檢查點」。

你不必仔細規劃每一步，而是可以告訴 Claude 嘗試有風險的做法。如果不行，倒帶並嘗試不同方法。檢查點在工作階段之間持久保存，因此你關閉終端機後仍然可以倒帶。

> **警告：** 檢查點只追蹤 *Claude* 所做的修改，不包含外部程序。這無法取代 git。

### 繼續對話

> 執行 `claude --continue` 從上次離開的地方繼續，或執行 `--resume` 從最近的工作階段中選擇。

Claude Code 會在本地儲存對話。當任務跨越多個工作階段時，你不必重新說明上下文：

```bash
claude --continue    # 繼續最近的對話
claude --resume      # 從最近的對話中選擇
```

使用 `/rename` 為工作階段命名描述性名稱，例如 `「oauth-migration」` 或 `「debugging-memory-leak」`，方便日後查找。將工作階段視為分支：不同的工作流可以有獨立且持久的上下文。

## 自動化與擴展規模

一旦你能有效使用單一 Claude，便可透過平行工作階段、非互動模式和扇出模式來倍增產出。

目前為止所有假設都是一個人、一個 Claude、一個對話。但 Claude Code 可以水平擴展。本節中的技術展示了如何完成更多工作。

### 執行非互動模式

> 在 CI、pre-commit 鉤子或腳本中使用 `claude -p "prompt"`。加入 `--output-format stream-json` 以取得串流 JSON 輸出。

使用 `claude -p "你的提示"`，你可以在沒有工作階段的情況下非互動式地執行 Claude。非互動模式是你將 Claude 整合到 CI 管道、pre-commit 鉤子或任何自動化工作流程中的方式。輸出格式讓你能以程式方式解析結果：純文字、JSON 或串流 JSON。

```bash
# 一次性查詢
claude -p "解釋這個專案是做什麼的"

# 腳本的結構化輸出
claude -p "列出所有 API 端點" --output-format json

# 即時處理的串流
claude -p "分析這個日誌檔案" --output-format stream-json
```

### 執行多個 Claude 工作階段

> 平行執行多個 Claude 工作階段以加速開發、執行隔離實驗或啟動複雜工作流程。

執行平行工作階段有三種主要方式：

- **Claude Code 桌面應用程式**：以視覺方式管理多個本地工作階段。每個工作階段獲得自己的隔離 worktree。
- **網頁版 Claude Code**：在隔離虛擬機器的 Anthropic 安全雲端基礎設施上執行。
- **代理團隊**：自動協調多個工作階段，具有共享任務、訊息傳遞和團隊領導功能。

除了並行化工作之外，多個工作階段還能實現以品質為導向的工作流程。新鮮的上下文能改善程式碼審查，因為 Claude 不會對剛剛撰寫的程式碼有所偏見。

例如，使用撰寫者/審查者模式：

| 工作階段 A（撰寫者） | 工作階段 B（審查者） |
| --- | --- |
| `為我們的 API 端點實作速率限制器` | |
| | `審查 @src/middleware/rateLimiter.ts 中的速率限制器實作。尋找邊界案例、競態條件，以及與我們現有中間件模式的一致性。` |
| `以下是審查反饋：[工作階段 B 輸出]。請解決這些問題。` | |

你也可以對測試做類似的事：讓一個 Claude 撰寫測試，再讓另一個撰寫程式碼來通過測試。

### 跨檔案扇出

> 呼叫 `claude -p` 時迴圈遍歷任務。使用 `--allowedTools` 為批次操作限定權限範圍。

對於大型遷移或分析，你可以將工作分散到多個平行的 Claude 呼叫中：

**第一步：生成任務列表**

讓 Claude 列出所有需要遷移的檔案（例如，`列出所有需要遷移的 2,000 個 Python 檔案`）

**第二步：撰寫腳本迴圈遍歷列表**

```bash
for file in $(cat files.txt); do
  claude -p "將 $file 從 React 遷移到 Vue。回傳 OK 或 FAIL。" \
    --allowedTools "Edit,Bash(git commit *)"
done
```

**第三步：在幾個檔案上測試，然後大規模執行**

根據前 2-3 個檔案的問題精煉你的提示，然後在完整集合上執行。`--allowedTools` 旗標限制了 Claude 能做什麼，這在你無人值守執行時很重要。

你也可以將 Claude 整合到現有的資料/處理管道中：

```bash
claude -p "<你的提示>" --output-format json | your_command
```

在開發期間使用 `--verbose` 進行除錯，在正式環境中關閉它。

## 避免常見的失敗模式

以下是常見的錯誤。及早認識它們可以節省時間：

- **廚房水槽工作階段。** 你從一個任務開始，然後問了不相關的問題，又回到第一個任務。上下文充滿了不相關的資訊。
  > **修正**：在不相關的任務之間使用 `/clear`。

- **一再修正。** Claude 做錯了某件事，你修正它，它還是錯，你再次修正。上下文被失敗的嘗試污染。
  > **修正**：兩次修正失敗後，執行 `/clear` 並撰寫融入你所學的更好初始提示。

- **過度指定的 CLAUDE.md。** 如果你的 CLAUDE.md 太長，Claude 會忽略一半，因為重要規則被雜訊淹沒了。
  > **修正**：無情地精簡。如果 Claude 不需要指示就已經能正確做某件事，就刪除它或將其轉換為鉤子。

- **信任後驗證的落差。** Claude 產出看似合理但無法處理邊界案例的實作。
  > **修正**：始終提供驗證（測試、腳本、截圖）。如果你無法驗證，就不要上線。

- **無限探索。** 你請 Claude「調查」某件事而沒有限定範圍。Claude 讀取數百個檔案，填滿上下文。
  > **修正**：將調查範圍限定得較窄，或使用子代理，讓探索不會消耗你的主上下文。

## 培養你的直覺

本指南中的模式並非一成不變。它們是在一般情況下運作良好的起點，但對每種情況未必是最佳解。

有時你*應該*讓上下文累積，因為你深陷在一個複雜問題中，而歷史是有價值的。有時你應該跳過規劃，讓 Claude 自行解決，因為任務是探索性的。有時模糊的提示恰到好處，因為你想在施加限制前看看 Claude 如何詮釋問題。

留意什麼有效。當 Claude 產出出色的輸出時，注意你做了什麼：提示結構、你提供的上下文、你所在的模式。當 Claude 遇到困難時，思考原因。上下文太嘈雜？提示太模糊？任務太大，無法一次完成？

隨著時間推移，你會培養出任何指南都無法捕捉的直覺。你會知道何時要具體、何時要開放，何時要規劃、何時要探索，何時要清除上下文、何時要讓它累積。
