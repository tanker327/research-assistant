## 描述

Claude Code 是一款出色的智能工程工具。但如同所有工具一樣，正確使用是獲得良好成果的關鍵。

相關課程：[Claude Code - 實用指南](https://acad.link/claude-code)

## 六大技巧摘要

### 1. 不要讓 AI 循環運行 / 不要 ROLF——保持掌控

- 避免讓 Claude Code 在無人監督的迴圈中運行（即「ROLF」模式：你提交一份詳細的產品文件，讓 AI 自主完成所有步驟）。
- 規劃固然重要，但讓 AI 自行執行計劃往往會產生不理想的結果。
- 在每個步驟中保持掌控。AI 是開發者的工具，而非取代開發者監督的存在。

### 2. 使用計劃模式（Plan Mode）

- 透過鍵盤快捷鍵切換模式；計劃模式已內建於 Claude Code 中。
- 在計劃模式下，Claude Code 會先收集資訊、探索程式碼庫，並在執行變更前制定計劃。
- **優點：**
  - 可彌補不夠精確的提示——它可能會提出釐清性的追問。
  - 在 AI 執行前，讓你看清它打算做什麼。
  - 你可以（也應該）在接受計劃前**編輯並調整計劃**。
- 不要盲目接受計劃。仔細審查、調整後再批准。這遠比讓它先執行、再修復問題要好，因為後者會消耗更多 token 並浪費時間。

### 3. 使用代理（Agents）與技能（Skills）

- **自訂子代理：** Claude Code 可以啟動具有獨立上下文視窗的子代理，節省主上下文視窗的 token，這些代理可以專注於特定任務。
  - 範例：一個「文件探索器」代理，專門用於瀏覽文件，配備網路搜索功能和 Context7 MCP。
- **Context7 MCP：** 一個 MCP 伺服器，讓 AI 代理更方便地取得第三方函式庫和程式語言的文件。是少數值得使用的 MCP 之一（多數 MCP 的 token 使用效率較低）。
- **技能（Skills）：** 針對特定專案的技能檔案，描述最佳實踐與偏好設定（例如 Next.js 的編碼規範）。
  - 技能採用懶加載機制——只在需要時載入，不會始終佔用完整上下文。
  - 已有開源技能集合（例如 Vercel 的技能），但根據個人偏好和模式自行撰寫技能也很有價值。
  - 技能為 AI 提供額外的上下文和指令，提升獲得良好結果的機率。

### 4. 明確優於隱含（Explicit Over Implicit）

- 不要期望 AI 自動做到你想要的——要明確告訴它。
- 如果你知道自己想要什麼，請直接說清楚。例如，明確指示它在實作前使用特定代理（如文件探索器）查閱文件。
- 明確說明 AI 可能本來就會做的事情並無壞處。這能提高輸出結果符合你預期的確定性。

### 5. 信任，但要驗證（Trust, But Verify）

- 即使有了好的提示和正確的上下文，也不要盲目信任結果。
- **仔細審查程式碼。** 接受好的部分，改善不好的部分（包括雖然能執行但不符合你偏好規範的程式碼）。
- **給 AI 提供自我驗證的工具：**
  - 單元測試（Unit tests）和端對端測試（E2E tests）
  - 靜態分析（Linting）指令
  - 用於瀏覽器驗證的 Playwright MCP（謹慎使用——非常消耗 token）
- 注意：AI 可能會編寫單純用來通過測試的測試案例，或根據程式碼反過來修改測試，而非反其道而行。你仍需自行審查。
- 你對程式碼負有責任。「是 AI 寫的」不是藉口。

### 6. 你仍然可以自己寫程式碼

- 使用 AI 並非非此即彼的選擇。你仍然應該自己寫程式碼。
- 瑣碎的修改（例如將間距從 0.5rem 調整為 1rem）不需要 AI——自己動手做，省下 token。
- 始終理解程式碼庫。不要將太多工作交給 AI，以免與程式碼脫節。
- 寫程式很有趣。Cursor/VS Code 中出色的自動補全功能讓手動寫程式比以往更輕鬆。
- 如果 AI 搞不定某個問題，就自己動手解決。

## 核心主題

- **掌控：** 開發者應始終掌控整個流程。
- **主動性：** 對提示、計劃和指令保持謹慎思考。
- **驗證：** 永遠不要假設 AI 的輸出是正確的——始終要審查。
- **效率：** 透過明確表達、使用計劃模式，以及手動處理瑣碎任務來節省 token 和時間。
- **平衡：** AI 是強大的工具，但它是增強、而非取代開發者技能的存在。

## 完整逐字稿

[00:00] 在過去幾個月裡，和許多人一樣，我大量使用了 Claude Code。我用它幫助我構建了 Build My Graphics 等專案，以及一些尚未發布的其他專案，還有很多內部工具。我沒有用它進行直播編程，這點要說清楚，不過我稍後會說明我如何使用它，因為這正是這部影片的主題。我想分享我使用 Claude Code 的頂尖策略，也就是我的六大使用策略。

[00:33] 這六件事是我在使用 Claude Code 時建議執行的，以獲得良好的結果。值得一提的是，我使用的是 Claude Code Max 方案，可以獲得 20 倍的使用量。顯然這受到 Anthropic 的大量補貼，但你可以從 Claude Code 中獲得大量使用量，尤其是如果你不讓 Claude Code 在迴圈中運行，也就是不採用 Rolf 迴圈模式的話。關於這一點有很多炒作。

[01:03] 這實際上已經是我的第一點：我不循環、不 Rolf，我保持掌控。這對我很重要。我知道這些天關於讓 Claude Code 在迴圈中運行有很多炒作。我們看到了整個 Rolf Wiggum 模式，其核心概念是：你有一份詳細的產品文件，一份詳細說明構建產品所需所有步驟的文件，然後你把這份文件交給 Claude Code，讓它在一個簡單的 bash 迴圈中反覆運行，使用一個提示告訴它查看文件、選擇下一步、執行那個步驟，然後逐步完成文件直到結束。

[01:54] 這個概念很簡單：透過詳細的計劃，你能獲得好的結果。我完全同意詳細的計劃很重要。規劃確實重要，無論你在構建什麼，我完全同意這一點。但我不喜歡讓 AI 自行執行計劃。相反，我喜歡如前所述地保持掌控。這對我來說真的很重要。我覺得 AI 對開發者來說可以是一個非常有用的工具。我相信每個開發者都應該把 AI 當作工具來使用，但我認為如果讓它自行運行，它真的無法給我好的結果。

[02:48] 這就是為什麼我絕對不喜歡直播編程。我嘗試過 Rolf 迴圈，不只是說說而已。對我來說，結果並不令人信服。我想保持掌控。

[03:01] 與此相關，我的第二點是使用計劃模式，因為計劃模式非常棒。在 Claude Code 中，你可以切換不同的模式，其中有一個內建的計劃模式。這個模式的核心概念是，它不會立即執行變更，也不會根據你的提示立即採取行動。相反，它首先會收集資訊、探索程式碼庫、嘗試理解你的提示，可能還會查閱文件，然後制定計劃，你再批准或拒絕那個計劃。如果需要更多釐清，它甚至可能向你提問。

[03:50] 因此，計劃模式真的非常有用，原因有幾個。首先，它可以彌補糟糕的提示。所以如果你寫的提示不夠好，計劃模式可以在這裡幫助你，因為它可以幫助你獲得更好的提示。透過提問，它可能會釐清你提示中不清楚的地方。但當然，你首先應該嘗試寫好的提示，因為這很重要。AI 是一個工具，輸出的品質只會和你的輸入一樣好。即便如此，讓我們誠實地說，其中還是有隨機性的。但是，如果你向 AI 拋出糟糕、不精確、缺乏上下文的提示，你就不會從中獲得好的結果。因此，好的提示加上好的上下文工程，也就是提供正確的上下文，非常重要。

[04:47] 當然，計劃模式確實可以作為一個小救星。但那仍然只是計劃模式的一個優點。計劃模式的另一個優點是，它向你展示 AI 想要做什麼，根據我的經驗，這真的很有價值。因為盲目信任 AI 去做某件事，以為它會做正確的事，實在太容易了。但嘿，這是 AI，它不一定會做正確的事。

[05:31] 計劃模式的好處在於，即使你沒有得到追問，提示可能已經足夠好了，我仍然能得到一個計劃。我知道很多開發者會盲目地接受那個計劃，就是直接按下 Enter 讓它執行，但不要成為那種開發者。看看那個計劃。它告訴你它想做什麼。如果你讓它修復一個問題或找到一個 bug 的解決方案，它會告訴你它認為問題的原因是什麼。

[06:16] 然後，讓人驚訝的是：你應該隨意編輯那個計劃。所以不要盲目接受它；如果需要，請編輯和調整計劃。你可以也應該這樣做。這不是你必須接受然後解決之後可能出現的任何問題的東西。相反，仔細看看計劃，根據你的喜好調整它，然後再接受它。這比只是讓它執行，然後發現你不喜歡它所做的事情要好得多。後者會消耗更多 token，需要更多工作，浪費時間，而且用 AI 修復程式碼一點也不有趣。

[07:45] 第三點是我使用代理（agents）和技能（skills）。Claude Code 允許你構建自訂子代理。這裡的概念是，Claude Code 可以啟動這些具有自己專用上下文視窗的子代理，以節省主上下文視窗中的 token，這樣你就不會用盡那裡的上下文空間，然後這些代理可以專注於某些任務。我喜歡構建的一個代理是文件探索器代理，它只是一個專門用於瀏覽文件的代理。我給這個代理配備了某些工具，如網路搜索或 Context7 MCP。

[08:36] Context7 MCP，如果你不知道的話，是一個 MCP 伺服器，讓 AI 代理更容易獲取你可能正在使用的第三方函式庫或程式語言的文件。我不是 MCP 的忠實粉絲。根據我的經驗，我覺得它們的 token 效率很低，AI 使用 MCP 的能力也不是很好，我更傾向於使用內建工具，而且我也不需要那麼多工具。但 Context7 MCP 非常出色，我構建了自己的文件探索器代理，它有使用網路搜索和這個代理來瀏覽文件的特定任務。

[09:19] 除了那個自訂代理，我還給 Claude Code 配備了依賴專案或特定於專案的技能。例如，如果我在一個 Next.js 專案中工作，我可能會給它一些描述我想讓它使用的最佳實踐的技能，與 Next.js 相關，以確保它使用或編寫符合我的期望和偏好的 Next.js 程式碼。

[09:49] 值得注意的是，也有像 Vercel 技能這樣的開源計劃，使得在專案中安裝特定技能和載入公開分享的開源技能變得容易，比如 Vercel 分享的 React 最佳實踐。我也會在許多專案中這樣做，但我仍然喜歡用我非常個人化的偏好、模式以及來自我作為開發者的經驗的規則來製作我自己的技能，因為我想讓 AI 使用這些。因為技能背後的理念是，AI 會在需要時閱讀那些技能文件。它們是懶加載的，不會始終完全載入到上下文中，而是在需要時才載入，然後那個技能可以給 AI 額外的上下文、額外的指令，這應該可以增加獲得好結果的機率。

[10:49] 我們總是在談論增加機率，因為它仍然是 AI，有隨機性存在。你無法確定它會按照你想要的方式做事，但你可以嘗試增加機率。根據我的經驗，這就是作為開發者使用 AI 的整個遊戲規則。

[11:09] 因此，這是我的第四點：明確優於隱含。我的意思是，AI 可能會按照我想要的方式做某事，但我無法確定。所以我寧願明確表達。例如，如果我在一個專案中使用 BetterAuth 函式庫並想透過 Google 使用身份驗證，我會明確地寫出來。但我也會說類似「在實作之前，使用文件探索器代理（這是我寫的擅長探索文件的自訂代理）來探索相關的 BetterAuth 文件」這樣的話，因為我太常看到 AI 就這樣去做某件事，然後我發現它沒有做我想要的事，例如，它可以訪問某些代理但就是不使用它們，即使理論上它應該這樣做。

[12:03] 我不希望這樣。說 AI 非常糟糕、沒有做我想要的事，對我沒有任何好處。它是一個可能非常棒的工具，某種程度上。因此，不要希望它做某事，也不要等待它失敗只是為了讓自己相信 AI 不好。相反，知道它擅長快速生成大量程式碼，並將其作為優勢使用，並給它所需的必要指令，以增加獲得你想要的輸出的機率。

[12:46] 如果我知道我想要什麼，我寧願明確告訴它，而不是希望它這樣做。如果透過明確告訴 AI 某件它本來也可能會做的事情，我就能確定它會做我想要的事，那我完全不介意。

[13:15] 我的第五點與此相關：信任，但要驗證。AI 很棒，如果你給它一個好的提示，如果你具體說明，如果你提供正確的上下文，根據所解決的問題，你有很好的機會獲得不錯的結果。但不要盲目信任。你在掌控，所以你應該驗證結果。你不應該盲目信任它們。仔細審查程式碼，不要只是認為它是正確的。批判性地思考它。不要因為是 AI 生成的就認為它一定不好，但也接受好的部分，嘗試改善不好的部分。不好也可以是某些也許可以、也許可以運行，但沒有使用你想要使用的模式或方法的東西。

[14:11] 但也要給 AI 提供自我驗證的工具。給它提供自我驗證的工具，因為這可以大幅提升結果。像單元測試、端對端測試，以及根據你在構建什麼，可能還有 Playwright MCP 等工具。這裡的概念很簡單，就是 AI 可以自己做這些事情。你可能想要告訴它，正如我在明確優於隱含中提到的，但它可以運行測試，可以運行靜態分析指令，還可以使用 Playwright MCP 等工具訪問瀏覽器，儘管你要小心這一點，因為它非常消耗 token，你不想一直使用它。

[15:15] 但是，給 AI 提供幫助它自行評估結果的工具可以大幅提升輸出品質。它仍然不是保證。例如，AI 可能會編寫單純通過的測試，它可能會根據程式碼調整測試而不是反其道而行，所以你要小心。但這仍然可以帶來更好的結果，而且這仍然不意味著你不應該審查程式碼。你要這樣做。你在掌控，你對程式碼負有責任。你不能說「AI 寫的，它很糟糕。」抱歉，你在掌控。

[15:56] 然後是最後一點：你仍然可以寫程式碼。這很令人驚訝，我知道，但有了 AI，有了 AI 工具，這不是非此即彼的選擇。你仍然可以寫程式碼，你也應該仍然寫程式碼。我不會去問 AI「請把這個方塊的邊距從 0.5rem 增加到 1rem。」我自己可以做到這一點。如果你審查程式碼，如果你理解 AI 生成的程式碼（你應該這樣做），重新進入程式碼庫應該不會太困難。

[16:34] 別誤會我的意思，我曾陷入把太多工作交給 AI 以及不完全理解程式碼庫的陷阱。我現在不再這樣做了。我確保我始終理解程式碼庫，並且我總是能夠重新進入其中，因為我是一名開發者。我可以閱讀程式碼，我可以編寫程式碼，即使在使用 AI 的時候我仍然可以編寫程式碼。所以如果 AI 搞不定某些事情，如果你有一個瑣碎的更改卻在消耗 token，我的意思是，那些 token 是要花錢的，就自己動手做吧。順帶一提，至少對我來說，寫程式很有趣，所以我喜歡時不時地深入程式碼、自己動手寫，而且有了 Cursor 或 VS Code 中出色的自動補全功能，這從未像現在這樣容易過。
