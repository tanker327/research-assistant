# 掌握氛圍：真正有效的 Claude Code 最佳實踐

**作者：** Dinanjana Gunaratne
**發布日期：** 2025-08-24

*我如何從跌跌撞撞地應付 AI 生成的義大利麵條式程式碼，到用 Claude Code 打造出生產就緒的應用程式*

三個月前，我還是那種開發者——對著 Claude Code 丟一個模糊的提示詞，雙手合十祈禱，然後期望奇蹟發生。結果呢？讓我這樣說吧：我花在除錯 AI 生成的一團亂麻上的時間，比自己動手寫程式還要多。

快轉到今天，Claude Code 已成為我首選的結對程式設計夥伴。這不是因為技術神奇地變好了（雖然它確實進步了），而是因為我學會了「與它協作」，而不只是「對它說話」。

如果你準備好從「丟出問題然後祈禱」的方式，轉向真正能穩定奏效的作法，以下是我在讓 Claude Code 真正發揮作用方面所學到的一切。

## 基礎：你的 .claude 檔案是你最好的朋友

讓我從最顛覆性的發現說起：**.claude 檔案系統**。把它想成 Claude 針對你專案的記憶庫。

Claude Code 按照層級順序讀取 CLAUDE.md 檔案：先從你的家目錄（~/.claude/CLAUDE.md），接著是專案根目錄，最後是各個子目錄。這不只是設定而已——這是你教導 Claude 理解你特定程式碼庫、規範與各種怪癖的方式。

以下是改變一切的內容：

```
# 我的專案規則

## 程式碼風格
- 全面使用 TypeScript（不設例外）
- 僅使用 Functional components 搭配 hooks
- 2 個空格縮排
- 變數使用 camelCase，元件使用 PascalCase

## 架構說明
- 使用 Zustand 進行狀態管理
- API 呼叫透過 /src/utils/api.ts 中的自訂客戶端處理
- 新元件需附帶測試檔案
- 效能至上——始終考量 bundle 大小

## 禁止事項
- 不使用 class components（因舊版程式碼庫因素）
- 不繞過我們的 error boundary 設定
- 不寫超過 500 行的元件（請拆分！）
```

魔法不在於格式——CLAUDE.md 檔案並沒有規定的格式——而在於針對「你的」專案，具體說明什麼才是重要的。

## 規劃革命：停止直接寫程式，先思考

大多數人在這裡誤用了 Claude Code：他們把它當成更快的自動補全工具。但真正的力量在於：在寫程式之前先請 Claude 制定計畫，並明確告訴它在你確認計畫之前不要開始寫程式。

## 計畫模式：你全新的秘密武器

計畫模式（Plan Mode）透過按兩次 Shift+Tab 啟動。在此模式下，Claude 成為一台研究與分析機器，實際上無法更動任何檔案。這就像是把 Claude 置於「架構師模式」——它可以觀察、分析與規劃，但在你核准之前絕不執行。

計畫模式迫使 Claude 以合理的詳細程度提供格式一致的回應，這解決了我對 AI 輸出不一致的最大困擾之一。

什麼時候應該使用計畫模式？

- **開始任何新功能**（這對我來說現在是不可妥協的）
- **涉及多個檔案的複雜重構**
- **除錯**，當你不確定問題根源時
- **影響多個元件的架構決策**

工作流程簡單卻強大：

1. 進入計畫模式（按兩次 Shift+Tab）
2. 描述你想達成的目標
3. 讓 Claude 研究你的程式碼庫並制定完整計畫
4. 審查並精煉計畫（不要接受第一個版本！）
5. 退出計畫模式並開始執行

## 深度思考層級：何時動用 AI 的腦力

並非所有提示詞都是平等的，Claude Code 有不同層級的推理深度。思考層級的層次如下：「think」< 「think hard」< 「think harder」< 「ultrathink」。

我會策略性地使用這些層級：

- **「think」**：用於直觀的功能新增或錯誤修正
- **「think hard」**：處理複雜業務邏輯或架構決策時
- **「think harder」**：用於效能優化或安全敏感的程式碼
- **「ultrathink」**：保留給真正棘手的問題——舊系統整合、複雜演算法，或完全卡住時

但有一點要注意——不要對所有事情都用「ultrathink」。這些特定短語在系統中被直接對應到遞增層級的思考預算，而那額外的運算你將在時間和 token 上付出代價。

## 教 Claude 主動提問的藝術

我最大的突破之一，是學會在規劃階段鼓勵 Claude 提問。我不再假設 Claude 知道我想要什麼，而是現在明確地要求它釐清需求：

「在你開始規劃使用者驗證系統之前，先問我關於需求的問題。我想確保我們打造的正是所需要的東西。」

這個簡單的改變徹底改造了我的成果。Claude 開始問出這樣的問題：

- 「你需要社群登入選項，還是只要電子郵件/密碼？」
- 「我們應該將用戶偏好設定儲存在本地還是資料庫中？」
- 「你對密碼重設的處理計畫是什麼？」

這些問題往往揭露了我自己都沒意識到的假設。

## 持久化計畫：外部記憶的技巧

將計畫寫入外部來源（如 plan.md）並作為檢查清單使用，是那種看似簡單卻出奇強大的想法。我不再把所有內容堆在聊天記錄中，而是現在對複雜專案讓 Claude 建立實際檔案：

- `plan.md` — 帶有核取方塊的主計畫
- `architecture.md` — 高層次系統設計
- `todo.md` — 當前工作的待辦清單
- `decisions.md` — 重要架構選擇的記錄

這不只是好的文件紀錄——它成為跨工作階段持續存在的工作記憶。幾天後當我回到一個專案時，我不需要從零開始。

## 協作心態的轉變

對我來說最大的心態轉變，是停止把 Claude Code 視為工具，而開始把它視為一位速度極快但需要良好指引的初級開發者。

雖然自動接受模式讓 Claude 可以自主運作，但通常透過積極協作、引導 Claude 的方法，你會得到更好的結果。我學會了：

- **在執行前審查每一份計畫**（計畫模式讓這件事變得容易）
- **積極修正方向**，而不是任由 Claude 陷入不必要的糾纏
- **將複雜變更拆分成較小的步驟**，以便能夠獨立驗證
- **自由運用 Escape 鍵**，在事情偏軌時立即中斷

## 結語

在持續實踐這些作法之後，我的 Claude Code 工作階段變得更可預測、更有成效。我不再生成需要大量除錯和重寫的程式碼，而是開始得到相當接近生產就緒的實作成果。

秘訣不在於提示詞的技巧——而在於流程。先規劃、思考深度適當、積極協作，並透過 .claude 檔案教導 Claude 認識你的特定情境。

## 你的下一步

1. **建立你的 .claude 檔案系統**，加入專案特定的情境資訊
2. **在下一個功能上試用計畫模式**（按兩次 Shift+Tab）
3. **練習「先計畫再執行」的工作流程**，而非直接寫程式
4. **針對不同問題類型，實驗不同的思考層級**
5. **開始鼓勵 Claude 提出釐清性問題**

目標不是取代你的工程判斷——而是放大它。當你讓協作運作順暢，Claude Code 就不再像是一個華麗的自動補全，而更像是你一直想要的結對程式設計夥伴。
