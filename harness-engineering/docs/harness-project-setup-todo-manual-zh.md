# Harness 项目搭建 TODO 手册

_用于从零搭建一个 Harness 项目，或将现有仓库逐步升级为符合 Harness Engineering 最佳实践的执行清单。_

本文档是以下手册的中文执行版配套文档：

- `harness-engineering/harness-engineering-practical-handbook-en.md`

当你需要的是**一步一步搭建**，而不是概念性说明时，请使用这份 TODO 手册。

---

## 0. 目标

完成这份清单后，你的项目应至少具备：

- 一个简洁可靠的 `AGENTS.md` 或 `CLAUDE.md`
- 确定性的本地质量检查
- CI 驱动的验证流程
- 机械化执行的架构约束
- 支持长时任务的状态交接文件
- 面向应用行为的验证，而不只是代码级验证
- 防止 Agent 削弱 Harness 的安全边界

如果以上任一层缺失，就不能说这个项目已经具备完整的 Harness。

---

## 1. 先决定运行模式

### 1.1 确定项目类型

- [ ] 决定这是一个 **greenfield** 还是 **brownfield** 的 Harness 搭建
- [ ] 决定 Agent 初期是 **有人值守（attended）** 还是目标就是 **无人值守（unattended）**
- [ ] 决定主要工作环境是：
  - 本地仓库
  - 远程开发环境
  - 隔离的 git worktree
  - 云容器 / devbox

### 1.2 明确人类职责

- [ ] 决定谁拥有架构决策权
- [ ] 决定谁负责最终 Review 和合并
- [ ] 决定 Agent 代码的质量标准
- [ ] 写下这条规则：Agent 产出必须满足与人类代码相同的质量标准

### 1.3 定义第一个成功目标

- [ ] 选一个 Harness 必须支持的端到端任务
- [ ] 让它足够小，最好一轮会话就能完成
- [ ] 让它足够重要，成功后有实际意义

**Step 1 完成标准**

- [ ] 团队知道当前是哪类项目
- [ ] 团队知道允许 Agent 拥有多大自主性
- [ ] 团队知道 Harness 的第一个里程碑任务是什么

---

## 2. 建立最小仓库骨架

### 2.1 创建必要的顶层文件和目录

- [ ] 在仓库根目录创建 `AGENTS.md` 或 `CLAUDE.md`
- [ ] 创建 `docs/`
- [ ] 创建 `docs/adr/`
- [ ] 创建 `docs/ARCHITECTURE.md`
- [ ] 如果不存在，创建基础测试目录
- [ ] 如果项目需要环境启动流程，创建启动脚本

### 2.2 保持根说明文件足够短

根说明文件应包含：

- [ ] 安装命令
- [ ] 运行 / 开发命令
- [ ] 测试命令
- [ ] lint / format 命令
- [ ] 架构文档指针
- [ ] 明确的禁止事项
- [ ] 完成标准

根说明文件**不应**包含：

- [ ] 冗长的架构说明
- [ ] 大而全的风格指南
- [ ] 所有仓库细节
- [ ] 与其他文档重复的信息

**规则**

- [ ] 保持其为“指针式”文档，并且短到可以长期稳定维护

### 2.3 编写第一条 ADR

- [ ] 创建 `docs/adr/ADR-001-adopt-harness-engineering.md`
- [ ] 记录：
  - 背景（context）
  - 决策（decision）
  - 后果（consequences）
  - 执行方式（enforcement）

第一条 ADR 推荐主题：

- [ ] 分层架构
- [ ] 选定的 lint / typecheck 工具栈
- [ ] 启动流程 / 状态追踪约定
- [ ] worktree 或环境隔离模型

**Step 2 完成标准**

- [ ] 仓库具备可见的 system-of-record 层
- [ ] 新会话无需询问人类即可找到 build / test / docs / architecture 的入口

---

## 3. 搭建本地质量闭环

### 3.1 选择确定性的质量工具栈

至少在以下类别中各选一个工具：

- [ ] formatter
- [ ] linter
- [ ] type checker（如适用）
- [ ] 单元测试运行器
- [ ] E2E 或 smoke test 运行器

### 3.2 把命令写清楚

- [ ] 将精确命令写入 `AGENTS.md` / `CLAUDE.md`
- [ ] 确认这些命令在干净 checkout 上可以运行
- [ ] 确保命令是真实项目命令，而不是从别的仓库复制过来的示例

### 3.3 加入 pre-commit 检查

- [ ] 增加 pre-commit 或等价 hook 工具
- [ ] 执行 formatter
- [ ] 执行 linter
- [ ] 如速度允许，执行 type check
- [ ] 如可行，执行轻量级架构检查

### 3.4 如果 Agent 环境支持，增加 post-edit 检查

- [ ] 文件编辑后自动运行 formatter
- [ ] 文件编辑后自动运行 lint
- [ ] 将残余错误反馈回 Agent 循环

**规则**

- [ ] 能在 CI 之前发现的问题，就不要等到 CI 再发现

**Step 3 完成标准**

- [ ] 仓库能在 push 前拒绝明显错误
- [ ] Agent 不需要“记得”去手动运行关键本地检查

---

## 4. 建立架构护栏

### 4.1 定义分层模型

- [ ] 在 `docs/ARCHITECTURE.md` 中写清依赖方向
- [ ] 定义稳定层次或模块边界
- [ ] 决定哪些 import 或跨层调用是禁止的

### 4.2 将不变量编码为规则

- [ ] 增加禁止 import 的自定义 lint 或结构测试
- [ ] 增加依赖方向检查
- [ ] 在关键边界处增加 schema / interface 校验
- [ ] 命名或目录放置规则仅在“能显著减少失败”时才加入

### 4.3 让错误信息具备修复指导能力

对每条关键规则：

- [ ] 说明哪里违反了规则
- [ ] 说明为什么存在这条规则
- [ ] 告诉 Agent 合法修复方式
- [ ] 如有必要，指向对应 ADR

### 4.4 保护 Harness 本身

阻止 Agent 修改：

- [ ] lint 配置
- [ ] formatter 配置
- [ ] CI 配置
- [ ] 环境变量或敏感文件
- [ ] 如有必要，部署敏感文件

**Step 4 完成标准**

- [ ] 项目拥有机械化执行的边界
- [ ] Agent 无法通过“改规则”来偷懒消除错误

---

## 5. 搭建验证层

### 5.1 建立最小测试金字塔

- [ ] 核心逻辑的单元测试
- [ ] 启动 / 健康检查 smoke test
- [ ] 至少一个核心业务流程的端到端验证路径

### 5.2 加入完成前验证

- [ ] 如果 Agent 环境支持，增加 stop / completion gate
- [ ] 要求 Agent 在宣称完成前运行测试
- [ ] 把“完成”定义为“已经验证”，而不是“看起来正确”

### 5.3 在真正重要的验证面上验证

根据项目类型选择正确的验证面：

- [ ] Web 应用 -> 浏览器自动化 / Accessibility Tree / DOM 快照
- [ ] API -> 集成测试 / 请求响应校验
- [ ] CLI/TUI -> 终端自动化
- [ ] Infra -> plan/apply 校验和环境断言
- [ ] ML/LLM -> eval 套件、benchmark、安全测试

### 5.4 增加“防止过早停止”规则

- [ ] 明确要求 Agent 对照原始任务 / 规格进行验证
- [ ] 明确写出：单元测试通过不等于功能完整

**Step 5 完成标准**

- [ ] 项目至少能证明一个真实工作流是端到端可运行的
- [ ] “功能完成”依赖于验证，不依赖 Agent 自我判断

---

## 6. 让应用对 Agent 可读

### 6.1 不要停留在“代码可见”

如条件允许，Agent 应该能够观察运行中的系统：

- [ ] 浏览器可见状态
- [ ] 日志
- [ ] 指标
- [ ] Trace
- [ ] 运行时健康状态

### 6.2 增加环境启动能力

- [ ] 创建 `init.sh` 或等价启动脚本
- [ ] 确保它能可靠启动应用或服务
- [ ] 确保 Agent 每次会话无需重新摸索启动步骤

### 6.3 优先使用隔离环境

以下三者选其一：

- [ ] 一个任务一个 git worktree
- [ ] 一个任务一个容器
- [ ] 一个任务一个远程 devbox

**不要**让多个无关任务共享一个可变环境。

### 6.4 尽量降低启动摩擦

- [ ] 测量从干净会话到应用可运行的时间
- [ ] 减少不必要的手动步骤
- [ ] 如果环境支持，对昂贵启动流程做预热或缓存

**Step 6 完成标准**

- [ ] Agent 能观察“应用是否工作”，而不只是“代码长什么样”
- [ ] 每个任务都有可预测的运行环境

---

## 7. 为多会话长任务增加状态交接

### 7.1 加入进度追踪文件

创建以下之一：

- [ ] `progress.json`
- [ ] `feature_list.json`
- [ ] 两者同时使用（如果项目既需要任务级追踪，也需要功能级追踪）

### 7.2 选择合适的状态模型

适合 `progress.json` 的情况：

- [ ] 项目是持续迭代式开发
- [ ] 需要灵活记录状态、阻塞项、任务优先级

适合 `feature_list.json` 的情况：

- [ ] 项目是长时构建型开发
- [ ] 需要在代码之外追踪功能完成度

### 7.3 给每个新会话增加启动例程

每次新会话都应：

- [ ] 确认当前工作目录
- [ ] 读取近期 git 历史
- [ ] 读取进度文件
- [ ] 如适用，读取 feature list
- [ ] 运行启动脚本
- [ ] 在开始新工作前先跑一遍 smoke path

### 7.4 增加 clean-state 规则

每次会话结束时应满足：

- [ ] 当前仓库处于可工作状态，或明确回退到已知良好状态
- [ ] 有 commit，或明确说明为什么本次没有 commit
- [ ] 更新 progress / state 文件
- [ ] 留下对下一会话清晰的 next step

**Step 7 完成标准**

- [ ] 下一个 Agent 会话无需“考古”就能继续工作
- [ ] 长时任务可以跨 context reset 持续推进

---

## 8. 把 Context 当作运行时资源管理

### 8.1 防止 Context 泛滥

- [ ] 非必要时不要整文件 dump
- [ ] 搜索结果要限流并要求细化查询
- [ ] 不要把巨量日志直接粘进主对话，除非当前决策必须依赖它

### 8.2 卸载大输出

- [ ] 大型测试输出写入文件
- [ ] 大型日志写入文件
- [ ] 在上下文中仅保留摘要，而不是整份原始输出

### 8.3 如果会话较长，增加 compaction 或 note-taking

- [ ] 总结旧对话历史
- [ ] 保留未解决 bug、关键决策和 next step
- [ ] 若任务经常跨越 context window，则增加 memory / notes 机制

### 8.4 对工具和文档使用渐进披露

- [ ] 只加载当前任务真正需要的工具
- [ ] 尽量把规则作用域限制在目录或文件模式上
- [ ] 保持全局规则最小化

**Step 8 完成标准**

- [ ] Agent 可以在更长任务中保持清醒，不被自身历史淹没
- [ ] 仓库能在正确时间暴露正确的信息

---

## 9. 增加安全与策略控制

### 9.1 定义允许的执行边界

- [ ] 决定 Agent 是全 shell 权限、沙箱权限还是审批模式
- [ ] 决定哪些破坏性命令应永远被阻止
- [ ] 决定哪些文件受保护

### 9.2 加入安全 hook 或策略层

- [ ] 在适当情况下阻止破坏性命令
- [ ] 阻止修改受保护文件
- [ ] 如环境支持，对高风险操作增加审批

### 9.3 限制重试次数

- [ ] 为 CI 重试次数设定上限
- [ ] 为自修复循环设定上限
- [ ] 多次失败后强制“停止并报告”

**Step 9 完成标准**

- [ ] Agent 无法悄悄扩大风险范围
- [ ] 系统不会在错误循环上无限燃烧时间和 token

---

## 10. 谨慎加入工具与能力层

### 10.1 默认工具集保持尽可能小

- [ ] 只暴露最常用、最必要的工具
- [ ] 避免大而重叠的工具集合
- [ ] 去掉会制造歧义的冗余工具

### 10.2 如果使用 MCP，进行访问裁剪

- [ ] 按逻辑分组组织工具
- [ ] 给每种工作流最小合理子集
- [ ] 确保工具不能在没有额外控制的情况下执行破坏性行为

### 10.3 在不同 Agent 表面之间复用能力层

- [ ] 优先复用共享能力层，而不是到处做一次性集成
- [ ] 尽可能让“人类指导 Agent”和“无人值守 Agent”复用同一套规则和文档

**Step 10 完成标准**

- [ ] 工具确实增强了能力，而不是让 context 和风险一起爆炸

---

## 11. 量化 Harness 效果

### 11.1 选一组足够小的指标

至少追踪：

- [ ] 首次 CI 通过率
- [ ] 返工率
- [ ] Agent 任务成功率
- [ ] 从任务开始到验证通过的耗时

### 11.2 增加一个定性复盘循环

- [ ] 每周回顾近期失败案例
- [ ] 针对重复失败，决定通过以下哪类方式修复：
  - 更好的文档
  - 更好的测试
  - 更强的 lint 规则
  - 更紧的环境设计
  - 更好的状态交接

### 11.3 让 Harness 从失败中成长

- [ ] 采用这条规则：每一种重复出现的失败，都应该产生一次 Harness 改进

**Step 11 完成标准**

- [ ] Harness 的演进依据真实失败数据，而不是纯粹靠主观感觉

---

## 12. 新项目搭建顺序

如果你是从零开始，请严格按以下顺序执行：

1. [ ] 定义自主性级别和第一个成功目标
2. [ ] 创建 `AGENTS.md` / `CLAUDE.md`
3. [ ] 添加 `docs/ARCHITECTURE.md` 和第一条 ADR
4. [ ] 增加 formatter、linter、tests，并写清命令
5. [ ] 增加 pre-commit 检查
6. [ ] 增加至少一个端到端验证路径
7. [ ] 增加受保护文件规则
8. [ ] 增加启动脚本
9. [ ] 增加 progress / state 文件
10. [ ] 增加 worktree / devbox / container 隔离模型
11. [ ] 只有在长会话出现时，再加入 compaction / note-taking
12. [ ] 等基础稳定后，再加入共享工具层和规模化能力

---

## 13. 现有项目升级顺序

如果你是在旧仓库上升级，**不要一次性把所有东西都加上**。

推荐顺序：

1. [ ] 明确 build / test / lint 命令
2. [ ] 增加 pre-commit
3. [ ] 增加一个 smoke test 和一个真实工作流测试
4. [ ] 增加根指针文件（`AGENTS.md` / `CLAUDE.md`）
5. [ ] 增加第一条 ADR 和架构图
6. [ ] 先加最有价值的一条架构规则
7. [ ] 如果任务是长时的，再加状态交接文件
8. [ ] 再补隔离环境和 application legibility
9. [ ] 只有当前规则稳定后，再继续扩充更多规则

---

## 14. 最终上线检查清单

在以下项目都满足之前，不要说 Harness “已经准备好了”：

- [ ] Agent 可以立刻找到 build / test / lint 指令
- [ ] 仓库有简洁的根入口和更深入的文档结构
- [ ] 项目有确定性的本地质量检查
- [ ] 项目至少有一个端到端验证路径
- [ ] 架构边界通过机器规则执行
- [ ] 受保护文件不会被 Agent 轻易改掉
- [ ] 应用 / 服务能在可预测环境中启动
- [ ] 长时任务有状态交接文件
- [ ] 大型输出不会淹没活跃上下文
- [ ] 重试循环有上限
- [ ] 团队可以解释下一次失败将如何推动 Harness 继续改进

如果以上任一项仍未勾选，说明 Harness 仍然不完整。

- [ ] `AGENTS.md` 或 `CLAUDE.md`
- [ ] `docs/ARCHITECTURE.md`
- [ ] `docs/adr/ADR-001-*.md`
- [ ] formatter 配置

## 16. 一句话规则

每次 Agent 以一种可重复方式失败时，都应该优先修复**环境**，让这种失败在下次更难发生，甚至不可能再发生。
