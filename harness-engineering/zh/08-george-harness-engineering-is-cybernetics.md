# Harness Engineering 就是控制论

读 OpenAI 的 [harness engineering 文章](https://openai.com/index/harness-engineering/) 时，我一直有种说不清的感觉。然后突然想明白了：我以前见过这个模式。不是一次——是三次。

第一次是 1780 年代 [James Watt 的离心调速器](https://en.wikipedia.org/wiki/Centrifugal_governor)。在它出现之前，一个工人站在蒸汽机旁边手动调节阀门。有了它之后，一个加重飞球机构感知转速并自动调节阀门。工人没有消失。工作变了：从转阀门变成设计调速器。

![瓦特离心调速器](https://pbs.twimg.com/media/HC3M_2qacAA7mej.png)

第二次是 [Kubernetes](https://kubernetes.io/docs/concepts/architecture/controller/)。你声明期望状态——三个副本、这个镜像、这些资源限制。控制器持续观察实际状态。当两者发生偏差时，控制器进行调和：重启崩溃的 Pod、扩展副本、回滚坏的部署。工程师的工作从重启服务变成了编写系统据以调和的规格说明。

第三次就是现在。OpenAI 描述了不再写代码的工程师。他们转而设计环境、构建反馈循环、将架构约束编码化——然后由 Agent 来写代码。[五个月一百万行代码](https://openai.com/index/harness-engineering/)，零行手写。他们称之为"harness engineering"。

每次都是同样的模式。Norbert Wiener 在 1948 年为它命名：控制论（cybernetics），源自希腊语 κυβερνήτης——舵手。同一个词根也赋予了 Kubernetes 它的名字。你不再转阀门。你掌舵。

![控制论](https://pbs.twimg.com/media/HC17rnca8AAQgjt.jpg)

每次这个模式出现，都是因为有人构建了足够强大的传感器和执行器，在那个层级闭合了控制回路。

## 为什么代码库是最后的堡垒

代码库本来就有反馈循环，但只在较低的层级。编译器在语法层面闭合了回路。测试套件在行为层面闭合了回路。Linter 在风格层面闭合了回路。这些都是真正的控制论控制——但它们只对可机械检查的属性起作用。能编译吗？能通过吗？遵守规则了吗？

在这之上的一切——这个改动是否符合系统架构？这是不是正确的方法？这个抽象随着代码库增长会不会造成问题？——没有传感器，也没有执行器。只有人类能在那个层级操作，而且是双向的：既判断质量，又编写修复。

LLM 同时改变了两端。它们能在人类曾经独占的层级进行感知——也能在同一层级行动：重构模块、重新设计不一致的接口、围绕真正重要的契约重写测试套件。反馈循环第一次可以在重要决策发生的地方闭合。

但闭合回路是必要的，不是充分的。瓦特的调速器需要调校。Kubernetes 控制器需要正确的规格。而在你的代码库上工作的 LLM 需要更难提供的东西。

## 校准传感器和执行器

让基本反馈循环运转起来——Agent 可以运行的测试、给出可解析输出的 CI、指向修复方向的错误消息——这只是入门。[Carlini 演示了这一点](https://www.anthropic.com/engineering/building-c-compiler)，他让 16 个并行 Agent 构建了一个 C 编译器：[极其简单的提示词](https://github.com/anthropics/claudes-c-compiler/commit/a28ff299)，但精心设计了测试基础设施。"我的大部分精力都花在了设计 Claude 周围的环境——测试、环境、反馈。"

更难的问题是用你系统特有的知识来校准传感器和执行器。这是大多数人卡住的地方，也是他们责怪 Agent 的地方。

"它老是做错事。它不理解我们的代码库。"这个诊断几乎总是错的。Agent 失败不是因为它能力不足。而是因为它需要的知识——对你的系统来说"好"意味着什么、你的架构奖励哪些模式、回避哪些模式——锁在你脑子里，你没有将它外部化。Agent 不会通过渗透作用学习。你不写下来，它第 100 次犯的错和第 1 次一模一样。

这项工作就是让你的判断力变得机器可读。描述实际分层和依赖方向的架构文档。内置了修复指引的自定义 Linter。编码了你团队品味的黄金准则。OpenAI [正是发现了这一点](https://openai.com/index/harness-engineering/)：他们每个周五花 20% 的时间清理"AI 垃圾"——直到他们把标准编码进了 Harness 本身。

## 唯一的出路

这些实践所要求的——文档、自动化测试、编码化的架构决策、快速反馈循环——从来就是正确的。过去三十年写的每本工程书都推荐它们。大多数人跳过它们，因为跳过的代价是缓慢而分散的：质量渐进下降、痛苦的新人培训、悄悄复利增长的技术债。

Agent 工程使这个代价变得极端。跳过文档，Agent 就会无视你的规范——不是在一个 PR 上，而是在每一个 PR 上，以机器速度，全天候。跳过测试，反馈循环根本无法闭合。跳过架构约束，偏移的复利增长速度超过你修复的速度。这里有个陷阱：如果 Agent 不知道"干净"长什么样，你就无法用 Agent 来清理混乱。没有校准，制造问题的机器也无法解决问题。

实践没有变。忽略它们的惩罚变得不可承受了。

生成-验证不对称性——[P vs NP](https://en.wikipedia.org/wiki/P_versus_NP_problem) 背后的直觉，由 Cobbe 等人[对 LLM 进行了实证验证](https://arxiv.org/abs/2110.14168)——指明了方向。生成正确解比验证一个解更难。你不需要在实现上超越机器。你需要在评估上超越它：明确"正确"是什么样子的、识别输出什么时候偏了、判断方向是否正确。

设计瓦特调速器的工人没有回去转阀门。不是因为不能。而是因为不再有意义了。
