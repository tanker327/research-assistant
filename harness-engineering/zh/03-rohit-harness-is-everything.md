Harness 就是一切：Cursor、Claude Code 和 Perplexity 到底构建了什么

你使用 AI 的方式不对，不是因为你没找到合适的模型。

你使用 AI 的方式不对，是因为你没有构建合适的环境。

有些团队三个工程师就能交付百万行代码，而另一些团队连一次稳定的重构都无法从 agent 流水线中完成，这是有原因的。差别不在于 GPT-5 还是 Claude Opus。差别不在于 temperature 设置或 max tokens。甚至也不在于 prompt，尽管所有人都在 prompt 的争论上浪费了大量时间。

差别在于 harness。

这篇文章要讲的是这个词到底意味着什么——从技术和哲学两个层面——因为行业已经养成了一个坏习惯，把它用得过于随意。Harness 不是 system prompt。它不是对 API 调用的一层封装。它不是评估框架、prompt 模板或带记忆的聊天机器人。Harness 是语言模型运行其中的完整设计环境，包括它能调用的工具、它接收信息的格式、它的历史记录如何被压缩和管理、在错误级联之前捕获错误的护栏，以及允许它将工作交接给未来自己而不丧失连贯性的脚手架。

当你审视 Anthropic 为让 Claude Code 真正运作所做的一切、OpenAI 为通过 Codex 交付百万行零手写代码所构建的系统，以及 Princeton NLP 团队在其里程碑式的 SWE-agent 论文中发表的关于 agent-计算机接口的研究时，你会开始看到同一个模式从每个认真做这件事的团队中浮现出来。

模型几乎无关紧要。Harness 就是一切。

这是一篇详细的技术拆解，讲述这个理念如何成为 2025 和 2026 年应用 AI 工程领域的决定性洞察。它涵盖了研究成果、真实的实现方案、驱动设计决策的失败模式，以及无论你是在构建编码 agent、研究 agent 还是长时间运行的自主软件工程师，都会反复出现的模式。读完之后，你不仅会理解什么是 harness，还会明白为什么正确构建 harness 现在是行业中最有价值的工程技能。

# 第一部分：没人谈论的问题

## 为什么原始能力是不够的

2024 年年中，AI 基准测试领域发生了一件奇怪的事。研究人员开始注意到，同一个前沿模型在相同的编码任务上可以产生截然不同的结果——完全取决于任务是如何呈现的以及提供了哪些工具。模型没有变。底层智能没有变。变化的是接口。

这本不应该令人惊讶。几十年来我们就知道，合适的工具能让工程师的生产力大幅提升。一个配备了现代 IDE、调试器、版本控制和 CI/CD 流水线的软件开发者，其效率远超同一个只有文本编辑器在裸终端中工作的开发者。IDE 不会让开发者更聪明。它消除摩擦，在正确的时刻呈现信息，尽早捕获错误，并将工作组织成可导航的单元。

语言模型也是一样。它们不是从某个无限内部知识库中进行推理的通用推理机。它们是在上下文窗口中对 token 进行操作的复杂模式匹配引擎。它们在任何给定时刻所知道的一切，取决于上下文窗口中有什么，而它们产出的一切，都受上下文结构的制约。输入的格式不是装饰。它就是 agent 的认知架构。

接口不是便利层。对于 LM agent 来说，接口就是心智本身。

这是 Princeton NLP 团队在 2024 年发表的 SWE-agent 论文的核心主张，经得起检验。该论文提出了 Agent-Computer Interface (ACI) 的概念，并证明了一个精心设计的 ACI 相比同一模型通过标准 Linux shell 交互，可以在基准性能上带来 64% 的相对提升。同样的模型、同样的任务、同样的计算预算。唯一的变量是接口。

让这个数字沉淀一下。64% 不是边际收益。这是工具能用和不能用之间的差别。而它完全来自环境设计，而非底层模型的任何改进。

## 上下文窗口不是 RAM 插槽

关于 AI agent 的天真心智模型把上下文窗口当作 RAM。你把数据加载进去，模型处理它，你得到输出。更多上下文等于更好的性能。更长的 prompt 等于更丰富的理解。这个心智模型错得离谱，如果你围绕它来构建 agent，它会毁掉你的系统。

上下文窗口实际上更接近 agent 在给定会话中的整个工作意识。窗口中的每个 token 都消耗算力。每一条无关信息都在与相关信息争夺注意力。模型没有一个能干净地忽略噪声的选择性注意力机制。噪声就在房间里，它影响着推理。

这对 agent 设计有具体、可衡量的影响。当你在 agent 循环内部对大型代码库运行 grep 并返回一万行匹配结果时，你并没有给 agent 更多可用的信息。你用无关数据淹没了它的工作记忆，这会降低后续每一步的质量，直到上下文被清除。当你因为 agent 想看两个函数就用 cat 把整个文件倒出来时，你递给了它一根消防水管，而它只需要一杯水。

SWE-agent 的研究人员非常细致地记录了这些失败模式。标准 bash 接口导致 agent 反复挣扎。它们会发出返回数千行结果的 grep 命令，忘记自己在找什么，发出更多 grep 命令，逐渐用噪声填满上下文，最终要么产出错误答案，要么完全停止进展。问题不在于模型智能。问题在于接口没有机制保护 agent 免受自身的伤害。

ACI 的解决方案是构建一个返回有上限、经过摘要的结果列表的搜索工具。如果你的搜索返回超过 50 个匹配项，工具会抑制输出并告诉 agent 缩小查询范围。这个单一的设计决策——回头看几乎简单得令人汗颜——是整篇论文中最高杠杆的改动之一。它将上下文泛滥的失败模式转化为一个自然的精炼循环。

# 第二部分：SWE-Agent 论文与 ACI 的诞生

## Agent-Computer Interface 到底是什么

ACI 在 SWE-agent 论文中被定义为位于语言模型 agent 和计算机环境之间的抽象层。与人机接口 (HCI) 的类比是有意为之的。正如 HCI 研究探讨如何设计匹配人类认知架构的接口，ACI 研究探讨如何设计匹配 LM 认知架构的接口。

人类的认知架构涉及视觉模式识别、空间记忆、屏幕上的并行注意力以及浏览和选择性聚焦的能力。LM 的认知架构根本不同。它涉及顺序 token 处理、对上下文顺序和格式的敏感性、有限的工作记忆，以及倾向于锚定在 prompt 中最突出的信息上。设计一个好的 ACI 意味着理解这些约束并围绕它们构建，而不是与之对抗。

SWE-agent 用于编码任务的 ACI 有四个主要组件，每一个都反映了关于语言模型在获得原始计算机访问权限时如何失败的特定洞察。

搜索与导航

搜索组件用专门构建的工具替代了标准的 grep 和 find 命令：find_file、search_file 和 search_dir。关键区别不在于语法。关键区别在于输出管理。结果上限为 50 条。如果查询超过该限制，工具会返回一条消息，解释结果太多并提示 agent 细化搜索。这听起来微不足道。实际上，这是论文中最具影响力的决策之一。

之所以重要，是因为 agent 和认知负荷下的人类一样，在感到不确定时倾向于继续做当前正在做的事。当人类在大型代码库中迷失时，他们会搜索得越来越广泛，产生越来越多的噪声。有上限的搜索工具通过创建一个强制函数来打断这种模式。你不能靠模糊来推进。你必须具体。这推动 agent 走向更审慎、更有针对性的行为。

文件查看器

文件查看器是论文关于认知架构的洞察变得最具体的地方。研究人员测试了多种查看器配置，发现一次显示 100 行是一个恰到好处的数字。更少的行数（他们测试了 30 行）导致 agent 丢失对周围代码的上下文而产生编辑错误。更多的行数（或完整文件）导致 agent 迷失位置并遗漏重要细节。

查看器是有状态的。它在交互之间维护文件中的位置。而且关键的是，它在每一行可见内容前添加了明确的行号。最后这个细节看起来是装饰性的。并非如此。当 agent 需要发出一个针对第 47 到 52 行的编辑命令时，它需要能够直接从视图中读取这些数字，而不是去计数或做算术。将这个认知任务从 agent 的工作记忆中移除，释放了容量用于实际的问题解决。

带 Linting 的文件编辑器

文件编辑器的关键创新是即时反馈加护栏。编辑命令接受起始行、结束行和替换文本作为单一操作。每次编辑后，工具自动对修改后的文件运行 linter 并报告结果。如果编辑引入了语法错误，编辑在应用之前就被拒绝，agent 会收到一条清晰的错误消息，同时显示原始代码和失败的编辑。

这闭合了导致天真 agent 实现中级联失败的反馈环。没有 linter 时，agent 可能引入一个语法错误，运行测试套件，看到一个似乎无关的失败（因为真正的错误在别处），花多个步骤追踪错误的问题，最终耗尽上下文窗口去追一个幽灵。将 linter 直接集成到编辑器中后，语法错误在引入的那一刻就被捕获，修复在问题传播之前就被定位。

与 agent 使用原始 bash 工具时的情况做比较。使用 sed 或输出重定向时，没有集成的反馈。编辑静默执行。多行更改需要复杂的参数格式化，极易出错。Agent 可能成功运行了命令却引入了一个 linter 本能捕获的细微格式错误，然后花接下来十个步骤纳闷测试为什么失败。

上下文管理

第四个组件解决了一个在长会话中不断累积的问题：陈旧上下文的堆积。当 agent 处理一个任务时，它的历史记录会充满旧的观察结果、中间状态和探索性步骤，这些不再反映环境的当前状态。所有这些历史记录占用上下文窗口的空间，并且可能通过提供过时信息来主动误导 agent。

ACI 的上下文管理系统将较早的观察结果——最近五轮之前的——折叠为单行摘要。这使活跃上下文聚焦在最近的相关信息上，同时保留了整体轨迹的压缩记录。Agent 始终可以看到它最近做了什么以及当前状态是什么，而不会被它运行过的每个命令的完整未压缩历史所淹没。

## 基准测试结果及其真正含义

SWE-agent 论文在 SWE-bench 数据集上对其 ACI 进行了基准测试，该数据集是从热门 Python 仓库中收集的真实 GitHub issue。任务是接收一个自然语言的 bug 报告并生成解决它的代码补丁。这是一个困难的实际任务，需要在不熟悉的代码库中导航、理解错误消息、编写正确的代码并验证修复。

使用 GPT-4 配标准 bash shell 接口，系统解决了 3.97% 的问题。使用 GPT-4 配专门构建的 ACI，系统解决了 12.47% 的问题。这就是前面提到的 64% 相对提升，完全来自接口设计。

研究人员还进行了消融实验，每次移除一个组件以隔离每个设计决策的贡献。Linter 集成始终是最高杠杆的组件之一。有上限的搜索对于防止上下文泛滥至关重要。带行号的有状态文件查看器明显优于裸 cat 命令和更简单的查看器设计。

性能差异不在于模型智能。而在于认知负荷管理。ACI 减少了模型追踪状态所需的工作量，为真正重要的工作腾出了空间。

其影响远超编码 agent。任何长周期 agent 任务都涉及相同的根本挑战：在大型信息空间中导航、跨多个步骤维持连贯状态、捕获并从错误中恢复，以及管理上下文窗口注意力这一有限资源。ACI 的设计原则是通用的。具体工具会变。问题的底层架构不会变。

# 第三部分：Anthropic 的 Harness 工程（长时间运行的 Agent 问题）

## 为什么上下文窗口边界是核心难题

SWE-agent 论文解决的是如何为单次 agent 会话设计接口。Anthropic 的工程团队在开发 Claude Agent SDK 和 Claude Code 时遇到了不同的问题：当一个任务大到无法在单个上下文窗口中完成时会怎样？

这不是一个边缘情况。大多数真实的软件项目都大到无法放入任何上下文窗口。一个生产环境的 Web 应用有数百个文件、数千个函数、一套测试、配置、文档和依赖。即使有 200K token 的上下文窗口，你也无法同时把整个项目装在脑子里。人类工程师通过外部记忆、文档、版本控制以及在代码库中工作数周数月积累的理解来解决这个问题。一个开始新会话的 agent 什么都没有。

天真的解决方案是压缩（compaction），它在一定程度上有效。Claude Agent SDK 包含压缩能力，可以在窗口填满时对旧上下文进行摘要。但仅靠压缩是不够的。Anthropic 的内部实验表明，即使有压缩，像 Opus 4.5 这样的前沿编码模型在跨多个上下文窗口的循环中运行时，始终无法从高层级 prompt 构建出生产质量的 Web 应用。

失败集中在两种模式上，两者都很有启发性。

第一种失败模式是试图一次做太多。当收到类似"构建一个 claude.ai 克隆"的 prompt 时，agent 会试图一次性完成整个应用。它会开始一个接一个地实现功能，但不完成也不测试任何一个，在实现过程中耗尽上下文窗口，留给下一个会话的是一个半成品应用、没有关于已完成工作的文档，也没有关于代码状态的清晰指示。下一个 agent 实例会把大部分上下文预算花在试图理解这堆混乱上，而不是推进工作。

第二种失败模式出现在项目的后期。在一些功能已经构建完成后，后续的 agent 实例会四处查看，看到已经有了进展，然后得出工作已完成的结论。它会对一个部分完成的应用宣布胜利并停止工作。这不是愚蠢。在信息不完整的情况下，这是一个合理的推断。Agent 没有结构化的方式来知道这个项目的"完成"到底意味着什么。

两种失败共享一个根本原因：agent 没有一个持久的、结构化的项目状态理解，能够跨越上下文窗口边界存活并为未来的会话提供方向。

## 双 Agent 架构：初始化 Agent 和编码 Agent

Anthropic 的解决方案是一个两部分的架构，此后已成为严肃团队处理长时间运行 agent 工作的模板。

第一部分是初始化 agent。这是一个专门的首次会话，有独特的 system prompt，其全部目的是搭建所有后续编码 agent 将运行其中的环境。它不编写功能。它创建使功能开发在后续多个会话中成为可能的脚手架。

初始化 agent 产生三个关键输出。首先，它创建一个 init.sh 脚本，可以可靠地启动开发环境。这听起来很平凡，但它有巨大的杠杆效应。之后的每个编码 agent 会话都可以通过运行 init.sh 开始，而不是花 token 来弄清楚如何启动服务器、设置数据库以及让应用进入可测试状态。每次会话节省的这些开销会不断累积。

其次，初始化 agent 创建一个全面的功能列表文件。在 Anthropic 内部运行的 claude.ai 克隆实验中，这意味着超过 200 个具体的端到端功能描述，比如"用户可以打开一个新聊天，输入一个查询，按回车，然后看到 AI 的回复"。每个功能最初都标记为失败。这个文件作为项目的真实来源。一个开始新会话的编码 agent 读取这个文件，立即确切知道什么已经构建、什么还没有。它不会四处查看代码就得出工作已完成的结论。功能列表告诉它真相。

第三，初始化 agent 创建一个 claude-progress.txt 文件并进行初始 git commit。进度文件是一个人类可读的日志，agent 在每次会话结束时更新，记录它们做了什么、完成了什么以及留下了什么状态。结合 git 历史，这为每个未来的编码 agent 提供了一种快速定位自身的方式，而不需要消耗上下文预算去做考古工作。

第二部分是编码 agent。初始化之后的每个会话使用不同的 prompt：一次处理一个功能，保持环境处于干净状态，在会话结束前更新进度文件和 git 历史。增量进展、文档化状态、干净交接。

## 功能列表作为认知锚点

功能列表值得特别关注，因为它解决了一个容易被低估的问题。没有它，一个在复杂代码库中运行的 agent 必须从代码本身推断项目的完成度。这种推断是不可靠的。代码可以存在但不起作用。功能可以存在但不完整。一个读取代码并推理什么已完成的 agent 会经常得出错误答案，这是一个严重的问题。

功能列表使完成度变得明确且无歧义。每个功能有一个 passes 字段，值为 true 或 false。Agent 要么在验证功能端到端工作后更新这个字段，要么不更新。没有歧义。不需要推断。真实来源就在文件中。

Anthropic 刻意决定将此列表存储为 JSON 而非 Markdown。原因是行为层面的。根据经验，模型不太可能不当修改或覆盖 JSON 文件，相比 Markdown 文件而言。JSON 具有抵抗随意编辑的刚性结构。这是一个小细节但有实际影响：你希望功能列表是 agent 谨慎更新的东西，而不是它们想改就随意重写的东西。

{ "category": "functional", "description": "New chat button creates a fresh conversation", "steps": [ "Navigate to main interface", "Click the 'New Chat' button", "Verify a new conversation is created", "Check that chat area shows welcome state", "Verify conversation appears in sidebar" ], "passes": false }

附带这种格式的指令是明确的：删除或编辑测试是不可接受的，因为这可能导致功能缺失或出现 bug。你提示模型将这个文件视为不可侵犯的。JSON 结构在架构层面强化了这条指令。

## 增量进展与干净状态要求

多会话 agent 工作中最困难的问题之一是确保每个会话结束时的状态是下一个会话可以安全在其基础上构建的。没有明确的强制执行，agent 倾向于在上下文窗口填满时把工作留在任何状态。半完成的功能、失败的测试、未文档化的更改。下一个 agent 继承这堆烂摊子。

Anthropic 的解决方案是将干净状态作为一等需求而非可有可无的东西。每个编码 agent 会话以一次 git commit（带描述性消息）、进度文件的更新结束，必要时回退到可工作状态。"干净状态"的含义是：代码适合合并到主分支——没有重大 bug，文档齐全，处于开发者可以合理地开始新功能而无需先解开别人半完成工作的状态。

Git commit 不仅仅是检查点。它是恢复机制。当 agent 做了一个破坏某些东西的更改时，它可以使用 git 回退到上一个已知正常的状态并重试。这就是人类工程师的工作方式，事实证明它也正是 agent 需要的纪律。版本控制是认知脚手架，不仅仅是源代码管理。

## 测试：没人喜欢谈论的失败模式

Anthropic 记录了一种几乎在每个严肃的 agent 编码项目中都会出现的失败模式：agent 在没有正确进行端到端验证的情况下就将功能标记为完成。Agent 会进行代码更改，运行单元测试或对开发服务器执行 curl 命令，看到通过的结果，然后将功能标记为完成。但该功能在通过浏览器像用户那样测试时实际上并不工作。

单元测试成功和端到端功能正常之间的差距，是人类工程师通过切换上下文来弥合的——运行应用并尝试使用它。一个没有明确浏览器测试能力的 agent 无法执行这种切换。它只能观察其工具允许它观察的内容，如果这些工具不包括浏览器自动化，它将持续遗漏一类只在真实用户流程中才会显现的 bug。

解决方案是让 agent 访问 Puppeteer MCP server，这是一个浏览器自动化工具，允许 Claude 实际导航应用、点击按钮、填写表单，并验证功能端到端正常工作。性能提升是显著的。仅从代码层面看不见的 bug，在 agent 能看到用户所见内容时变得显而易见。

这是一个通用原则的具体例证：agent 工作的质量受限于其反馈环的质量。如果你的 agent 无法在真正重要的领域观察其行为的后果，它将优化那些可能与实际正确性不相关的代理指标。

## 启动序列：快速进入状态

Anthropic harness 中的每个编码 agent 会话都以一个标准化的启动序列开始，旨在尽可能快地让 agent 定位自身而不浪费不必要的 token。序列是：

运行 pwd 确认工作目录。读取进度文件和 git log 了解最近的工作。读取功能列表并选择优先级最高的未完成功能。运行 init.sh 脚本启动开发环境。运行基本端到端测试以验证应用处于可工作状态。

只有在完成所有这些步骤后，agent 才会开始处理新功能。如果启动测试发现应用已损坏，agent 会先修复现有的问题再动任何新东西。这防止了 agent 在损坏的基础上开始新功能的复合问题，那会使底层问题更难隔离和修复。

启动序列还以特定方式节省了 token。因为 init.sh 脚本精确记录了如何启动开发环境，agent 不需要从头弄清楚。每次会话在环境搭建上节省的 token 在长项目中累积起来相当可观。

[Assistant] I'll start by getting my bearings and understanding the current state of the project. [Tool Use] <bash - pwd> [Tool Use] <read - claude-progress.txt> [Tool Use] <read - feature_list.json> [Assistant] Let me check the git log to see recent work. [Tool Use] <bash - git log --oneline -20> [Assistant] Now let me check if there's an init.sh script to restart the servers. <Starts the development server> [Assistant] Excellent! Now let me navigate to the application and verify that some fundamental features are still working. <Tests basic functionality>

# 第四部分：OpenAI 的 Harness 工程（零行手写代码）

## 实验

2025 年 8 月下旬，OpenAI 的 Codex 团队创建了一个 git 仓库，只有一个约束：不允许人类编写代码。仓库中的每一行代码——包括应用逻辑、测试、CI 配置、文档、可观测性工具和内部开发者工具——都将由 Codex agent 编写。人类负责引导。Agent 负责执行。

五个月后，仓库包含了大约一百万行代码，涵盖所有这些类别。大约 1,500 个 pull request 已被创建和合并。一个三人工程师的小团队完成了大部分工作，平均每人每天 3.5 个 pull request。当团队扩大到七人时，每人的产出实际上还提高了。该产品有数百名日活内部用户和外部 alpha 测试者。

这不是一个演示。这是一个完全通过 agent 生成的代码构建和交付的真实内部产品。团队在 2026 年 2 月撰写了描述这一经验的文章，核心信息与 SWE-agent 论文一致：瓶颈从来不是模型能力。瓶颈始终是环境设计。

## 工程工作的重新定义

OpenAI harness 工程文章中最重要的观察是关于工程工作本身如何改变的。当你的主要工作不再是写代码时，你在做什么？

你在设计环境。你在明确意图。你在构建反馈环。你不断地问——不是"我怎么修这个 bug？"而是"环境中缺少什么能力导致了这个 bug 的出现？"

当某些东西失败时，修复几乎从来不是"更努力地尝试"。几乎总是"环境中缺少或配置错误的什么结构性部分导致 agent 在这里失败？"这是工程思维的深刻转变。你不再调试代码。你开始调试产生代码的系统。

工程团队的主要工作变成了让 agent 能够做有用的工作，而不是自己做这些工作。

在实践中，这意味着将大目标分解为更小的构建块，构建使这些构建块可实现的工具和抽象，并利用失败作为环境需要什么来更好支持的信号。人类工程师深度优先地工作：当 agent 卡住时，他们不会试图自己写代码。他们会问缺少什么，将其构建到环境中，然后让 agent 重试。

## 仓库知识作为记录系统

OpenAI harness 中最重要的架构决策之一是让仓库本身成为 agent 需要知道的一切的真实来源。洞察很简单但影响深远：从 agent 的角度来看，它在运行时无法在上下文中访问的任何内容实际上都不存在。存在于 Google Docs、Slack 对话或人们脑子里的知识对系统来说是不可见的。

在项目早期，团队尝试了"一个大 AGENTS.md"的方法。一个包含 agent 需要知道的关于项目、架构、惯例、约束的所有内容的大型指令文件。它可预见地失败了——以四种值得理解的方式。

第一，上下文是稀缺资源。一个巨大的指令文件会挤占任务、代码和相关文档的空间。Agent 要么遗漏关键约束，要么开始针对错误的方向优化。第二，太多指导等于没有指导。当所有东西都被标记为重要时，没有东西是重要的。Agent 开始局部模式匹配而非有意识地导航。第三，它会立即腐烂。一个单体手册随着代码库的演进会变成陈旧规则的坟场。第四，它难以验证。一个单一的大块不适合做覆盖检查、新鲜度追踪或交叉链接。偏差是不可避免的。

解决方案是一个结构化的 docs/ 目录作为记录系统，配合一个简短的 AGENTS.md 文件（大约 100 行）作为指向其他更深层真实来源的地图。设计文档被编目和索引。架构文档提供了领域和包分层的顶层地图。计划被视为一等工件，进展和决策日志被检入仓库。

这实现了团队所谓的渐进式披露（progressive disclosure）：agent 从一个小的、稳定的入口点开始，并被引导知道下一步去哪里查找，而不是一开始就被信息淹没。结果是 agent 可以直接从仓库推理完整的业务领域，而不需要访问可能不可用或可能过时的外部上下文。

## 应用可读性：让系统对 Agent 可见

随着代码产出的增加，瓶颈从生成转移到了验证。团队生成代码的速度超过了人工 QA 能力的验证速度。解决方案是让更多的验证工作变成 agent 自己能做的——通过使应用对 Codex 直接可读。

这涉及几项具体投资。他们使应用可以按 git worktree 启动，这样 Codex 可以为每个正在处理的更改启动和驱动应用的隔离实例。他们将 Chrome DevTools Protocol 接入 agent 运行时，并创建了处理 DOM 快照、截图和浏览器导航的工具。这使 Codex 能够复现 bug、验证修复并直接推理 UI 行为，而无需人类与应用交互。

他们构建了一套完整的本地可观测性栈：日志、指标和追踪通过 LogQL、PromQL 和 TraceQL 暴露给 Codex。每个 agent 任务在完全隔离的应用版本上运行，有自己的可观测性数据，任务完成后销毁。这意味着 agent 可以使用真正的可观测性工具调试类生产问题——与人类工程师使用的相同工具——而不必仅从代码推断行为。

这里的原则与 SWE-agent 论文所展示的相同：agent 工作的质量受限于其反馈环的质量。如果 agent 能看到用户所见，能观察到人类工程师所观察的指标和日志，它就能捕获和修复比仅靠代码操作的 agent 广泛得多的问题类别。

## 不微管理地执行架构

在完全由 agent 生成的代码库中，最有趣的挑战之一是随时间维持架构一致性。Codex 会复制仓库中已存在的模式，包括不均匀或次优的模式。随着时间推移，这导致漂移。糟糕的模式扩散。不一致性累积。代码库变得越来越难让未来的 agent 正确导航。

OpenAI 的解决方案是通过机械方式而非人工代码审查来执行不变量。应用围绕一个严格的架构模型构建：每个业务领域划分为固定的层集合，具有严格验证的依赖方向和有限的允许边集。这些约束由自定义 linter（自然是由 Codex 编写的）和结构测试来执行。

关键洞察是在执行边界的同时允许边界内的充分自由。Linter 检查代码是否按正确方向流经层级结构。它们不规定边界内的具体功能如何实现。这与使平台团队能够大规模高效运作的原则相同：执行基础，在其之上允许自主。

Linter 是专门编写的，以便为 agent 生成有帮助的错误消息。当 linter 捕获到违规时，错误消息包含格式化的修复指令，可注入 agent 上下文。这闭合了循环：被违反的约束、被违反的规则以及修复步骤全部在一条可操作的反馈消息中传递。

他们还将所谓的"黄金原则"直接编码到仓库中：有主见的、机械化的规则，使代码库对未来的 agent 运行保持可读和一致。优先使用共享工具包而非手写帮助函数。在边界处验证数据结构。这些原则由定期清理的后台任务执行，扫描偏差、更新质量等级并创建有针对性的重构 pull request。其中大多数可以在一分钟内审查完毕并自动合并。

## 吞吐量改变了合并理念

当 agent 的吞吐量大幅超过人类注意力容量时，传统的工程规范会变得适得其反。等待审查的 pull request 阻塞了 agent 的工作。逐个调查的测试不稳定（test flake）消耗了本可用于更高杠杆任务的人类注意力。

OpenAI 的团队做了一个深思熟虑的决定，以最少的阻塞合并关卡运营。Pull request 保持短生命周期。测试不稳定通过后续运行来解决，而非无限期阻塞进展。当 agent 的吞吐量远超人类注意力时，修正是廉价的而等待是昂贵的。正确的权衡在低吞吐量环境中看起来不负责任，在高吞吐量环境中却显而易见。

这对正在转向 agent 驱动开发的团队来说是一个真正重要的洞察。当人类工程师编写每一行代码时合理的合并理念，在 agent 每天每位工程师生成 3.5 个 pull request 时不一定还合理。瓶颈转移了，流程也需要随之转移。

# 第五部分：Awesome Agent Harness 分类体系

## 生态系统映射

Awesome Agent Harness 仓库由 GitHub 上的 AutoJunjie 项目维护，试图映射新兴的 harness 工程工具生态系统。在深入分类之前，值得明确陈述其核心论点：AI 编写代码的能力实际上已经是商品化的。基础模型可以产出可用的代码。这不再是差异化能力。差异化能力在于协调和环境设计。

该仓库编目了一个严肃的 agent harness 生态系统所需的完整栈，分为七个不同的层。理解这些层有助于解释为什么"构建一个 AI 编码助手"实际上是许多独立的工程问题，而不是一个。

第 1 层：人类监督

最顶层是人类监督层，人类在这里批准提案、审查 pull request 和设定优先级。这不是传统意义上的技术层。它是人类判断和 agent 执行之间的接口。这里的关键设计原则是工程师应该设计环境和审查结果，而不是直接编写代码。他们的杠杆来自引导，而非执行。

第 2 层：规划与需求（Spec 工具）

这一层将人类的想法翻译成结构化的规格说明和任务 DAG（有向无环图），使 agent 能够可靠地消费。底层洞察是 agent 盲目执行。如果规格说明模糊或有歧义，agent 会产出满足其对规格说明理解的东西，而这可能不是人类想要的。Spec 工具在需求阶段强制精确，在任何代码编写之前。

这个领域的一个项目 Chorus 试图解决该仓库所称的"反向对话差距"。与其让人类编写详细的 spec ticket（这是一个主要的失败点，因为人类不是天生就能以 agent 需要的方式精确表达），Chorus 让 AI 提出任务 DAG 并详细说明需求，人类在执行开始前处于严格的验证和批准角色。AI 从部分意图生成完整规格说明的能力比人类从零开始编写要强。

第 3 层：全生命周期平台

这些工具管理从初始需求到交付的端到端流程，将 AI 提案与人类验证关卡和子 agent 编排集成。它们是规格说明层和执行层之间的粘合剂，处理贯穿整个开发生命周期的状态管理。

第 4 层：任务运行器

任务运行器弥合了问题跟踪器（GitHub Issues、Linear）和编码 agent 之间的差距。流程是：人类或 PM agent 创建一个 issue，任务运行器生成一个工作空间，agent 交付一个 pull request，人类审查。这个类别中的工具包括持续轮询任务队列、决定何时生成 agent 并交付完成工作的系统，无需人类参与执行循环。

第 5 层：Agent 编排器

编排器通过在隔离的 git worktree 中并行执行多个 agent 来解决吞吐量问题。这至关重要，因为在同一代码库上并行工作的 agent 如果共享工作空间会互相冲突。Git worktree 隔离为每个 agent 提供自己的沙箱，允许多个 agent 同时工作而不互相干扰。

像 Vibe Kanban、Emdash 和 Composio 这样的工具实现了这种模式。每个 agent 任务获得自己的 git worktree。更改在隔离环境中验证后才合并。CI 反馈、合并冲突和 agent 之间的协调全部由编排层处理，而无需人类干预。

第 6 层：Agent Harness 框架和运行时

框架提供用于构建自定义环境的可组合原语：渐进式披露机制、子 agent 生成、结构化上下文交付。运行时提供持久化基础设施：长期运行的记忆、定时执行、会话之间的多通道通信。

框架和运行时的区别很重要。框架是你在其上构建的。运行时是持续运行的。Claude Agent SDK 主要是一个框架。一个按 cron 计划运行 agent、跨会话维护持久记忆并处理 agent 实例之间多通道协调的系统是运行时。两者对于严肃的长时间运行 agent 工作都是必要的。

第 7 层：编码 Agent

最底层是执行层：Claude Code、Codex 以及类似的编写、测试和调试代码的系统。该仓库的关键洞察是这一层是商品化的。Agent 的有效性主要由栈中它上面的一切决定，而非 agent 本身。

这是一个有争议的论断，但证据支持它。SWE-agent 论文从经验上证明了这一点：同样的模型，接口设计带来 64% 的性能提升。OpenAI 的 Codex 团队从运营上证明了这一点：重要的工程工作是环境设计，而非执行。Anthropic 的 harness 工程工作从实践上证明了这一点：初始化 agent 的设置决定了编码 agent 能否取得进展。

# 第六部分：反复出现的设计模式

在所有这些系统和所有这些组织中，几种设计模式反复出现。它们不是巧合。它们是工程上的解决方案，应对的是每当你试图大规模可靠部署 agent 时就会出现的问题。

## 模式 1：渐进式披露

不要一开始就给 agent 所有它可能需要的东西。给它定位自身所需的最少量信息，以及在需要时找到更多信息的指引。这个模式出现在 SWE-agent 有上限的搜索中（不返回所有结果，强制 agent 细化），在 OpenAI 的 docs/ 架构中（一个指向更深层真相的简短地图），在 Anthropic 的启动序列中（先读进度文件，再读功能列表），以及在实现结构化上下文分层的 harness 框架中。

这个模式的认知原因是上下文是有限资源，agent 的注意力在上下文中的分布不是均匀的。在 prompt 开头呈现的信息具有不成比例的影响力。一个简短、聚焦的入口点指向更丰富的上下文，比一个将注意力稀释到所有内容的全面转储更有效。

实际原因是维护性。一个作为更深层文档地图的简短入口点是你能保持准确的。一个包含所有内容的单体文档会迅速过时并产生反效果。

## 模式 2：Git Worktree 隔离

一个 agent，一个 worktree。这个模式出现在每个严肃的编排系统中。推理很直接：当你有多个 agent 并行工作时（或者单个 agent 在顺序执行任务时），你需要工作流之间的隔离。没有隔离，并行 agent 会互相踩踏对方的更改。即使是顺序 agent，你也需要在隔离环境中验证更改后再影响主代码库的能力。

Git worktree 在文件系统层面提供这种隔离。每个 agent 获得自己的工作目录、自己的分支和自己的环境。更改在隔离中进行、在隔离中测试，只有在通过验证后才合并。这就是现代 CI/CD 系统为人类工程师工作的方式，事实证明它也正是 agent 编排的正确模型。

## 模式 3：Spec 优先，仓库作为记录系统

Agent 对非正式知识是盲目的。存在于 Slack 对话、Google Doc 或某人脑子里的任何东西对 agent 来说是不可见的。Agent 唯一能使用的是其上下文窗口中的内容，而该上下文唯一可靠的来源是仓库。

这个模式表现为 Anthropic harness 中的功能列表文件，OpenAI 系统中的结构化 docs/ 目录，各种开源框架中的 AGENTS.md 文件，以及 awesome-agent-harness 分类体系中的 spec 工具层。共同点是规格说明、需求、架构决策和约束必须在执行开始之前被编码到仓库中的机器可读文件中。如果 agent 不能从仓库中读取它，它就不存在。

这对工程团队如何记录工作有重要影响。文档不再只是写给人类读者的。它是人类意图变得对 agent 可读的机制。模糊的、过时的或存储在仓库之外的文档，是积极损害 agent 性能的文档。

## 模式 4：机械化架构执行

人工代码审查无法扩展到 agent 驱动的开发。当 agent 每天每位工程师可以创建 3.5 个 pull request 时，审查不能成为维护代码质量和架构完整性的主要机制。解决方案是将架构约束编码为自动运行的机械检查。

自定义 linter、结构测试和 CI 流水线替代了人类驱动开发中代码审查的大部分功能。优势在于机械检查是一致的、快速的，并在违规点提供即时反馈。一个在违规发生时捕获架构违规并在错误消息中返回修复指令的 linter，比三天后在 pull request 评论中捕获同一违规的代码审查者更有效。

关键设计原则是执行不变量，而非实现。你非常关心依赖方向、边界跨越、接口处的数据验证，以及命名和结构的一致性。你不关心 agent 使用哪个具体的库或函数具体如何分解，只要它满足行为契约。这在一个定义良好的结构内给予 agent 充分的自主权。

## 模式 5：集成反馈环

每个高性能的 harness 架构都尽可能紧密地闭合反馈环。语法错误在编辑时被 linter 捕获。运行时错误通过 agent 可以查询的可观测性工具浮现。UI bug 通过 agent 可以驱动的浏览器自动化捕获。测试失败伴随着关于什么在哪里破坏的上下文返回。

替代方案——agent 编写代码，在外部测试并在后续会话中产生反馈回来的失败消息——更慢，在 token 上更昂贵，而且更可能产生级联失败。反馈环中每一个可以缩短行动与后果之间差距的点，都是可以提升 agent 性能的点。

这是经典软件工程原则"尽早捕获错误"的 harness 版本。越早捕获错误，修复成本越低。对于 agent 来说，这一原则的适用力度更强，因为未被立即捕获的错误会在上下文中累积并降低后续推理的质量。

# 第七部分：这对工程师意味着什么

## 可迁移的技能

Harness 工程学科本质上是应用于 agent 环境的系统思维。它要求你足够理解语言模型的认知架构，以设计与之协同而非对抗的环境。它要求你以分布式系统工程中熟悉但应用于新领域的方式来思考状态管理、反馈环、错误恢复和上下文优化。

在这个新兴范式中最高效的工程师，不是那些有最好 prompting 技巧的人，尽管 prompting 很重要。他们是理解整个系统如何运作的人：上下文如何流动、在哪里被破坏、反馈环如何收紧、状态如何跨会话保持，以及约束如何在不微管理 agent 行为的情况下被执行。

这些在抽象层面不是新技能。它们是优秀软件工程师已有技能的延伸。系统设计、API 设计、错误处理、测试策略。新的是领域：为 LM agent 设计环境，而非为人类设计接口。

## 你应该问的问题

当你在构建一个 agent 系统而某些东西不工作时，harness 工程的思维方式会产生与天真思维方式不同的问题集。

不是"我怎么写一个更好的 prompt？"而是"agent 需要什么信息但目前无法获取？"不是"为什么模型在犯这个错误？"而是"缺少什么反馈环来在这个错误传播之前捕获它？"不是"为什么 agent 没做我告诉它的事？"而是"环境中什么约束在阻止 agent 做我告诉它的事？"

这种转变不仅仅是语义上的。它改变了你投入工程努力的方向。投资于一个解决这个特定失败模式的更好 prompt 是局部的和临时的。投资于一个防止一类失败模式的更好工具是通用的和永久的。Harness 就是那个永久投资所在之处。

## 执行的商品化

Awesome-agent-harness 仓库核心论点中有一个令人不安的含义值得直白地陈述。如果执行层是商品化的，那么 AI 驱动开发中的长期竞争护城河不在模型。而在 harness。

这意味着那些投资于 harness 工程——构建脚手架、反馈环、可观测性、spec 工具以及允许 agent 大规模可靠工作的编排——的组织和个人，将比那些主要关注使用哪个模型或如何 prompt 它的人拥有持久的优势。

OpenAI 的 Codex 团队为其特定代码库和领域构建了相当于一个自定义开发平台的东西。Anthropic 构建了一个 harness 架构，使复杂应用能够持续数月增量推进。SWE-agent 团队构建了一个接口，使同一模型产出 64% 更好的结果。这些优势没有一个来自模型。它们全部来自环境。

模型是思考的东西。Harness 是思考的对象。正确理解这个区别就是整个游戏。

# 第八部分：构建你自己的 Harness

## 最小 Harness

你不需要构建 OpenAI 的可观测性栈或 Anthropic 的完整双 agent 架构就能从 harness 思维中受益。在真实项目上编码 agent 的最小有效 harness 只需要少量核心组件。

从一个持久化进度文件开始。Agent 在每次会话开始时读取它以了解上次做了什么，在每次会话结束时写入它以记录这次做了什么。仅这一个改变就能防止"过早宣布胜利"的失败模式，并确保跨上下文窗口边界的连续性。

添加一个结构化任务列表。不是对项目的模糊描述，而是一个具体的、可枚举的可验证完成标准列表。每个条目应描述一个用户可见的行为，可以进行端到端测试。用一个状态标记每个条目，agent 只在验证后才更新。这防止了"部分完成看起来像完成"的失败模式。

将版本控制和描述性 commit 消息作为每次会话的一等组成部分。每次会话以一次 commit 结束。Agent 在代码 commit 和进度文件更新之前不应认为工作已完成。这创建了使多会话工作连贯的干净交接。

如果你在构建 Web 应用，添加浏览器自动化。一个只能读代码的 agent 和一个能实际使用它正在构建的应用的 agent 之间的差别，与一个只能读代码的开发者和一个能运行应用的开发者之间的差别是一样的。大多数重要的 bug 只在运行时可见。

## 环境审计

如果你已经有一个 agent 系统但表现不佳，harness 工程方法建议一个特定的诊断流程。不要去换一个更好的模型或更长的 prompt，而是做一次环境审计。

问：agent 需要什么信息但目前无法获取？任务流程中哪些点 agent 经常卡住或犯错？缺少什么反馈能让 agent 自己捕获这些错误？上下文在哪里被无关信息污染？哪些约束需要被执行但目前依赖于 agent 的判断？

每个问题都指向一个具体的 harness 改进。缺少的信息变成仓库中的新工具或新文档。缺少的反馈变成新的测试、linter 或可观测性集成。上下文污染变成新的上下文管理策略。未执行的约束变成新的机械检查。

这就是 harness 开发的良性循环：每一次失败都是环境需要什么的信号，每一次环境的改进都减少了该失败在所有未来 agent 会话中的频率。

# 最后一点

变革性技术在早期阶段被误解有一种规律。吸引公众注意力的东西——原始能力、令人印象深刻的演示、基准分数——很少是长期决定谁赢的东西。基础设施层——harness、环境——通常才是真正价值被创造和获取的地方。

Web 之所以具有变革性，不是因为 HTML 存在，而是因为搜索引擎和浏览器使 Web 变得可导航。移动端之所以具有变革性，不是因为智能手机存在，而是因为应用商店和开发者工具使得在智能手机之上大规模构建成为可能。在这两种情况下，组织底层能力的平台层才是持久价值所在之处。

AI agent 正在遵循同样的规律。能力已经存在。问题是谁来构建使能力变得可靠、可控和可持续改进的环境。SWE-agent 的研究人员在 2024 年理解了这一点并用数据证明了它。Anthropic 在构建 Claude Code 时理解了这一点并公开记录了它。OpenAI 在构建内部产品时理解了这一点并分享了经验教训。Awesome-agent-harness 社区正在跨数十个工具和框架编目记录它。

Harness 就是一切。模型是推理引擎。Harness 是上下文、约束、反馈环、记忆、工具和脚手架，它决定了推理引擎实际上能完成什么。把 harness 做对不是 prompt 工程问题。它是系统工程问题。而且它是当下应用 AI 中最重要的工程问题。

据此而建。
