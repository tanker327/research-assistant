## 驾驭工程：构建让 AI 智能体真正可用的系统完全指南

**2026 年 3 月** — 如果说 2025 年是 AI 智能体证明自己能写代码的一年，那么 2026 年就是我们认识到**难点不在于智能体本身——而在于驾驭系统**的一年。

OpenAI 的 Codex 团队刚刚构建了一个生产级应用，包含**超过 100 万行代码**，其中**没有一行是人工编写的**。工程师们不写代码，他们设计的是让 AI 可靠编写代码的系统。这个系统——包括约束、反馈循环、文档、代码检查工具和生命周期管理——就是业界现在所说的**驾驭系统（harness）**。

**驾驭工程（Harness engineering）** 是设计这些系统的新兴学科，它正在改变软件工程师这一角色的内涵。

---

## 什么是驾驭工程？

### 马匹的比喻

"Harness"这个词来源于马具——缰绳、马鞍、嚼子——是用来引导强大但不可预测的动物朝正确方向前进的一整套装备。这个比喻是刻意为之的：

- **马**是 AI 模型——强大、快速，但它自己不知道该去哪里
- **驾驭系统**是基础设施——约束、护栏和反馈循环，将模型的能力引导到有生产力的方向
- **骑手**是人类工程师——提供方向，而不是亲自奔跑

没有驾驭系统，AI 智能体就像旷野中的纯血马。跑得快、令人印象深刻，但完全无法完成任何有用的事情。

### 正式定义

**驾驭工程**是设计和实现以下系统的学科：

1. **约束**智能体能做什么（架构边界、依赖规则）
2. **告知**智能体应该做什么（上下文工程、文档）
3. **验证**智能体是否正确完成了任务（测试、代码检查、CI 验证）
4. **纠正**智能体在出错时的行为（反馈循环、自修复机制）

Martin Fowler 将其描述为*"我们用来约束 AI 智能体的工具和实践"*——但它不仅仅是安全性。好的驾驭系统让智能体**更强大**，而不只是更受控。

## 为什么驾驭工程现在很重要

### 模型是通用品，驾驭系统才是护城河

这是 AI 行业正在面对的一个不太舒服的事实：**底层模型的重要性远不如围绕它的系统。**

LangChain 明确地证明了这一点。他们的编程智能体在 Terminal Bench 2.0 上从 **52.8% 提升到 66.5%**——从**前 30 名跃升到前 5 名**——而模型没有任何变化。他们只改变了驾驭系统：

| 改变 | 具体做法 | 影响 |
| --- | --- | --- |
| 自我验证循环 | 添加了完成前检查清单 middleware | 在提交前捕获错误 |
| 上下文工程 | 启动时映射目录结构 | 智能体从一开始就理解代码库 |
| 循环检测 | 追踪重复的文件编辑 | 防止"死循环" |
| 推理三明治 | 规划/验证使用高推理等级，实现使用中等推理等级 | 在时间预算内获得更好质量 |

**相同的模型。不同的驾驭系统。显著更好的结果。**

### OpenAI 的百万行代码实证

OpenAI 的实验是迄今为止最有说服力的证据：

- **5 个月**的开发时间
- 最终产品包含**超过 100 万行代码**
- **零行手动编写的代码**——每一行都由 Codex 智能体生成
- **大约只用了人工所需时间的十分之一**
- 产品拥有**内部日活用户和外部 Alpha 测试用户**
- 它**部署、上线、出问题、被修复**——全部由驾驭系统内的智能体完成

工程师的工作？设计驾驭系统。明确意图。提供反馈。而不是写代码。

## 驾驭工程的三大支柱

OpenAI 的框架将驾驭工程组织为三个核心类别：

### 1\. 上下文工程

上下文工程的目标是确保智能体在正确的时间获得正确的信息。

**静态上下文：**

- 仓库内的文档（架构规范、API 契约、风格指南）
- `AGENTS.md` 或 `CLAUDE.md` 文件，编码项目特定规则
- 由代码检查工具验证的交叉引用设计文档

**动态上下文：**

- 智能体可访问的可观测性数据（日志、指标、链路追踪）
- 智能体启动时的目录结构映射
- CI/CD 流水线状态和测试结果

**关键规则：** 从智能体的视角来看，任何它在上下文中无法访问的信息都不存在。存在于 Google Docs、Slack 对话或人们脑中的知识对系统来说是不可见的。**代码仓库必须是唯一的真实来源。**

### 2\. 架构约束

这是驾驭工程与传统 AI 提示词工程差异最大的地方。与其告诉智能体"写好代码"，不如**用机制来强制执行好代码的标准。**

**依赖分层：**

```
Types → Config → Repo → Service → Runtime → UI
```

每一层只能从其左侧的层导入。这不是建议——而是通过结构化测试和 CI 验证来强制执行的。

**约束执行工具：**

- **确定性代码检查工具** — 自动标记违规的自定义规则
- **基于 LLM 的审计器** — 审查其他智能体代码的架构合规性的智能体
- **结构化测试** — 类似 ArchUnit，但面向 AI 生成的代码
- **Pre-commit hooks** — 代码提交前的自动化检查

**为什么约束能改善输出：** 看似矛盾的是，约束解空间反而让智能体**更高效**，而非降低效率。当智能体可以生成任何东西时，它会浪费 token 去探索死路。当驾驭系统定义了清晰的边界，智能体就能更快地收敛到正确的解决方案。

### 3\. 熵管理（"垃圾回收"）

这是最被低估的组件。随着时间推移，AI 生成的代码库会积累熵——文档与实际代码脱节、命名规范出现分歧、死代码不断堆积。

驾驭工程通过**定期清理智能体**来解决这个问题：

- **文档一致性智能体** — 验证文档与当前代码是否匹配
- **约束违规扫描器** — 发现早期检查遗漏的代码
- **模式执行智能体** — 识别并修复偏离既定模式的代码
- **依赖审计器** — 追踪并解决循环依赖或不必要的依赖

这些智能体按计划运行——每天、每周，或由特定事件触发——保持代码库对人类审查者和未来 AI 智能体都处于健康状态。

## 驾驭工程实践：团队实际是怎么做的

### OpenAI 的方法：零人工代码

OpenAI 的驾驭工程团队结构：

| 角色 | 传统方式 | 驾驭工程方式 |
| --- | --- | --- |
| 编写代码 | 主要工作 | 从不 |
| 设计架构 | 工作的一部分 | 主要工作 |
| 编写文档 | 事后补充 | 关键基础设施 |
| 审查 PR | 代码审查 | 审查智能体输出 + 驾驭系统有效性 |
| 调试 | 阅读代码 | 分析智能体行为模式 |
| 测试 | 编写测试 | 设计由智能体执行的测试策略 |

### Stripe 的方法：规模化的 Minions

Stripe 的内部编程智能体 **Minions** 现在每周产出**超过 1,000 个合并的 PR**：

1. 开发者在 Slack 中发布任务
2. Minion 编写代码
3. Minion 通过 CI
4. Minion 开启一个 PR
5. 人类审查并合并

从第 1 步到第 5 步之间没有任何开发者介入。驾驭系统处理一切——测试执行、CI 验证、风格合规和文档更新。

### LangChain 的方法：Middleware 优先

LangChain 将驾驭系统构建为可组合的 middleware 层：

```
Agent Request
  → LocalContextMiddleware (映射代码库)
  → LoopDetectionMiddleware (防止重复)
  → ReasoningSandwichMiddleware (优化计算)
  → PreCompletionChecklistMiddleware (强制验证)
  → Agent Response
```

每个 middleware 层添加特定能力，而不修改核心智能体逻辑。这种模块化方法使驾驭系统可测试且可演进。

## 构建你的第一个驾驭系统：实用框架

### 第一级：基础驾驭系统（单个开发者）

如果你正在使用 Claude Code、Cursor 或 Codex 进行个人项目：

**需要设置的内容：**

- 包含项目规范的 `CLAUDE.md` 或 `.cursorrules` 文件
- 用于代码检查和格式化的 pre-commit hooks
- 智能体可以运行以自我验证的测试套件
- 具有一致命名的清晰目录结构

**设置时间：** 1-2 小时 **影响：** 防止最常见的智能体错误

### 第二级：团队驾驭系统（小型团队）

适用于 3-10 人共享代码库的团队：

**在第一级基础上添加：**

- 包含团队通用规范的 `AGENTS.md`
- 由 CI 强制执行的架构约束
- 常见任务的共享提示词模板
- 由代码检查工具验证的文档即代码
- 专门针对智能体生成 PR 的代码审查清单

**设置时间：** 1-2 天 **影响：** 团队范围内一致的智能体行为

### 第三级：生产级驾驭系统（工程组织）

适用于同时运行数十个智能体的组织：

**在第二级基础上添加：**

- 自定义 middleware 层（循环检测、推理优化）
- 可观测性集成（智能体读取日志和指标）
- 按计划运行的熵管理智能体
- 驾驭系统版本管理和 A/B 测试
- 智能体性能监控仪表板
- 智能体卡住时的升级策略

**设置时间：** 1-2 周 **影响：** 智能体作为自主贡献者运行

## 常见的驾驭工程错误

### 1\. 过度设计控制流

> *"如果你过度设计控制流，下一次模型更新就会让你的系统崩溃。"*

模型进步很快。2024 年需要复杂流水线才能实现的能力，现在一个上下文窗口提示词就能搞定。构建你的驾驭系统时要保持**可拆卸性**——当模型变得足够智能而不再需要某些"聪明"逻辑时，你应该能够随时移除它们。

### 2\. 将驾驭系统视为静态的

驾驭系统需要随模型一起演进。当新模型版本改善了推理能力，你的推理优化 middleware 可能会变得适得其反。在每次重大模型更新时审查和更新驾驭系统组件。

### 3\. 忽视文档层

最有影响力的驾驭系统改进往往是最简单的：**更好的文档**。如果你的 `AGENTS.md` 含糊不清，你的智能体输出也会含糊不清。投入精力编写精确的、机器可读的文档，作为智能体的基准事实。

### 4\. 没有反馈循环

没有反馈的驾驭系统是牢笼，而非向导。智能体需要知道自己什么时候成功、什么时候失败。需要内置：

- 任务完成前的自我验证步骤
- 将测试执行纳入智能体工作流程
- 按任务类型统计智能体成功率的指标

### 5\. 仅面向人类的文档

如果你的架构决策存在于人们的脑中或智能体无法访问的 Confluence 页面中，那么驾驭系统就存在缺口。**智能体需要的一切都必须在代码仓库中。**

## 驾驭工程与相关概念的对比

| 概念 | 范围 | 关注点 |
| --- | --- | --- |
| **Prompt Engineering** | 单次交互 | 编写有效的提示词 |
| **上下文工程** | 模型上下文窗口 | 模型看到的信息 |
| **驾驭工程** | 整个智能体系统 | 环境、约束、反馈、生命周期 |
| **智能体工程** | 智能体架构 | 智能体内部设计和路由 |
| **平台工程** | 基础设施 | 部署、扩展、运维 |

驾驭工程**包含**上下文工程并借鉴 Prompt Engineering，但它运作在更高的层面——它关注的是让智能体可靠运行的完整系统，而不仅仅是单次交互的输入。

## 这对软件工程师意味着什么

### 工作正在变化

驾驭工程代表了软件工程师工作内容的真正演变：

| 之前 | 之后 |
| --- | --- |
| 编写代码 | 设计让 AI 编写代码的环境 |
| 调试代码 | 调试智能体行为 |
| 审查代码 | 审查智能体输出 + 驾驭系统有效性 |
| 编写测试 | 设计测试策略 |
| 维护文档 | 将文档构建为机器可读的基础设施 |

这并不意味着工程师变得不那么技术化了。恰恰相反，驾驭工程需要**更深层次**的架构思维——你正在设计一个在没有你持续干预的情况下必须正常工作的系统。

### 重要的技能

基于我们在 [NxCode](https://www.nxcode.io/) 构建 AI 驱动产品的经验：

1. **系统思维** — 理解约束、反馈循环和文档如何相互作用
2. **架构设计** — 定义可执行且有生产力的边界
3. **规格编写** — 将意图表达得足够精确，以便智能体执行
4. **可观测性** — 构建能揭示智能体行为模式的监控
5. **迭代速度** — 快速测试和优化驾驭系统配置

### 我们的经验：实践中什么有效

我们一直在使用多种智能体系统（Claude Code、Codex、Cursor）构建 AI 驱动的 Web 应用。对我们影响最大的模式：

- **仓库优先的文档**：每一个架构决策、命名规范和部署流程都在代码仓库中。没有任何东西存在于 Slack 或 Google Docs 中。
- **增量式约束构建**：从基本的代码检查开始，随着模式的出现添加架构约束，不要试图预先设计完美的驾驭系统。
- **智能体专用审查清单**：AI 生成的代码与人类代码有不同的失败模式。我们的审查流程考虑了常见的智能体模式（过度抽象、不必要的错误处理、文档漂移）。
- **多供应商驾驭系统设计**：我们的驾驭系统兼容 Claude、GPT 和 Gemini 模型。供应商无关的设计意味着我们可以切换模型而无需重建整个系统。
