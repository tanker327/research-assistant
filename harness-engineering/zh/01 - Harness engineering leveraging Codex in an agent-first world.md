# Harness 工程实践：在 Agent 优先的世界中利用 Codex

----

2026 年 2 月 11 日

[工程](https://openai.com/news/engineering/)

作者：Ryan Lopopolo，技术团队成员

<audio src="https://cdn-azalea-tts-achmg6dqbafjaxcr.a01.azurefd.net/tts-prod/1flz3YCBifvfy5L243uhH9/ember/f5614e0bdeb77ffd247280ba20f50dd8.mp3"></audio>

在过去五个月里，我们团队一直在进行一项实验：构建并交付一个内部测试版的软件产品，且**代码中没有一行是人工手写的**。

该产品拥有内部日活用户和外部 Alpha 测试用户。它经历了部署、发布、出故障、再修复的完整流程。不同之处在于，每一行代码——应用逻辑、测试、CI 配置、文档、可观测性和内部工具——全部由 Codex 编写。我们估计，相比手工编写代码，我们用大约 1/10 的时间就完成了这个项目。

**人类掌舵，Agent 执行。**

我们刻意选择了这一约束条件，以便构建出将工程效率提升数个数量级所需的一切。我们只有几周时间来交付最终达到百万行代码规模的产品。为此，我们需要理解一个根本性的变化：当软件工程团队的核心工作不再是编写代码，而是设计环境、明确意图，并构建反馈循环以让 Codex Agent 可靠地完成工作时，一切会发生怎样的改变。

这篇文章讲述了我们用一个 Agent 团队从零构建全新产品的过程中学到的经验——哪些地方出了问题，哪些地方产生了复合效应，以及如何最大化利用我们唯一真正稀缺的资源：人类的时间和注意力。

## 我们从一个空的 Git 仓库开始

第一个提交到空仓库的 commit 是在 2025 年 8 月底。

初始脚手架——仓库结构、CI 配置、格式化规则、包管理器设置和应用框架——由 Codex CLI 使用 GPT-5 生成，并以一小组现有模板作为引导。甚至最初那份指导 Agent 如何在仓库中工作的 AGENTS.md 文件本身也是由 Codex 编写的。

没有任何预先存在的人工编写代码来锚定整个系统。从一开始，这个仓库就是由 Agent 塑造的。

五个月后，该仓库包含了大约百万行代码，涵盖应用逻辑、基础设施、工具链、文档和内部开发者实用工具。在此期间，一个仅由三名工程师驱动 Codex 的小团队提交并合并了大约 1,500 个 PR。这意味着每位工程师平均每天产出 3.5 个 PR，而且令人惊讶的是，随着团队扩展到现在的七名工程师，吞吐量反而*增加*了。重要的是，这并非为了产出而产出：该产品已被内部数百名用户使用，包括每日使用的内部核心用户。

在整个开发过程中，人类从未直接贡献过任何代码。这成为团队的核心理念：**不手写代码**。

## 重新定义工程师的角色

没有人工编写代码这一约束**引入了一种不同类型的工程工作，聚焦于系统设计、脚手架搭建和杠杆效应**。

早期进展比预期更慢，这并非因为 Codex 能力不足，而是因为环境定义不够充分。Agent 缺乏完成高层目标所需的工具、抽象和内部结构。我们工程团队的首要工作变成了赋能 Agent 去做有价值的工作。

在实践中，这意味着深度优先地工作：将更大的目标拆解为更小的构建模块（设计、编码、审查、测试等），提示 Agent 构建这些模块，并利用它们来解锁更复杂的任务。当某些事情失败时，解决方案几乎从不是"再努力试试"。因为推进工作的唯一方式就是让 Codex 来完成，人类工程师总是深入任务并思考："缺少什么能力？如何让它对 Agent 来说既可读又可执行？"

人类几乎完全通过 Prompt 与系统交互：工程师描述一个任务，运行 Agent，让它打开一个 PR。为了推动 PR 完成，我们指示 Codex 在本地审查自己的改动，在本地和云端请求额外的特定 Agent 审查，回应任何人类或 Agent 给出的反馈，并循环迭代直到所有 Agent 审查者满意（实际上这就是一个 [Ralph Wiggum Loop](https://ghuntley.com/loop/)）。Codex 直接使用我们的标准开发工具（gh、本地脚本和仓库内嵌技能）来收集上下文，无需人类复制粘贴到 CLI 中。

人类可以审查 PR，但并非必须。随着时间推移，我们已将几乎所有审查工作推向 Agent 之间相互完成。

## 提升应用的可读性

随着代码产出量的增加，我们的瓶颈变成了人类的 QA 能力。由于固定约束是人类的时间和注意力，我们致力于通过让应用 UI、日志和应用指标等内容直接对 Codex 可读来增加 Agent 的能力。

例如，我们让应用可以按 Git worktree 启动，这样 Codex 可以为每个改动启动和驱动一个独立实例。我们还将 Chrome DevTools Protocol 接入了 Agent 运行时，并创建了处理 DOM 快照、截图和导航的技能。这使得 Codex 能够直接复现 Bug、验证修复，并对 UI 行为进行推理。

我们经常看到单次 Codex 运行在一个任务上持续工作超过六个小时（通常是在人类睡觉的时候）。

## 我们将仓库知识作为唯一信息源

上下文管理是让 Agent 在大型复杂任务中保持高效的最大挑战之一。我们最早学到的一个经验很简单：**给 Codex 一张地图，而不是一本千页的操作手册。**

我们尝试过"一个大 [`AGENTS.md`](https://agents.md/)"的方式。它以可预见的方式失败了：

- **上下文是稀缺资源。** 一个巨大的指令文件会挤占任务、代码和相关文档的空间——导致 Agent 要么遗漏关键约束，要么开始针对错误的目标进行优化。
- **过多的指导变成了*无指导*。** 当所有内容都被标为"重要"时，就没有什么是重要的了。Agent 最终会局部模式匹配，而非有目的地导航。
- **它会立即腐化。** 一个单体式的手册会变成陈旧规则的坟场。Agent 无法判断什么仍然有效，人类也不再维护它，这个文件就悄悄变成了一个有害的陷阱。
- **它难以验证。** 单一的大文件不适合机械化检查（覆盖率、时效性、所有权、交叉引用），因此漂移不可避免。

所以我们不再将 `AGENTS.md` 当作百科全书，而是将其作为**目录**。

仓库的知识库存放在一个结构化的 `docs/` 目录中，被视为唯一信息源。一个简短的 `AGENTS.md`（大约 100 行）被注入到上下文中，主要作为地图，指向其他地方更深层的真实信息来源。

设计文档经过编目和索引，包括验证状态和一组定义 Agent 优先运营原则的核心信念。[架构文档](https://matklad.github.io/2021/02/06/ARCHITECTURE.md.html)提供了领域和包分层的顶层地图。质量文档对每个产品领域和架构层进行评分，并随时间跟踪差距。

计划被视为一等公民。小改动使用轻量级的临时计划，而复杂工作则通过[执行计划](https://cookbook.openai.com/articles/codex_exec_plans)来记录，包含进度和决策日志，并提交到仓库中。活跃计划、已完成计划和已知技术债务都进行版本控制并放在一起，使 Agent 无需依赖外部上下文即可运作。

这实现了**渐进式披露**：Agent 从一个小而稳定的入口点开始，并被教导下一步该去哪里查找，而不是一开始就被信息淹没。

我们通过机械化手段来强制执行这一点。专用的 Linter 和 CI 任务验证知识库是最新的、交叉链接的且结构正确的。一个定期运行的"文档园丁" Agent 会扫描不再反映实际代码行为的陈旧或过时文档，并提交修复 PR。

## Agent 可读性是目标

随着代码库的演进，Codex 的设计决策框架也需要随之演进。

因为仓库完全由 Agent 生成，它首先针对 *Codex 的可读性*进行了优化。就像团队致力于提高代码对新入职工程师的可导航性一样，我们人类工程师的目标是让 Agent 能够**直接从仓库本身**推理整个业务领域。

从 Agent 的角度来看，它在运行时无法在上下文中访问的任何内容实际上都不存在。存在于 Google Docs、聊天记录或人们脑海中的知识对系统来说是不可访问的。仓库本地的、版本化的制品（如代码、Markdown、Schema、可执行计划）才是它能看到的一切。

我们发现，随着时间推移，需要将越来越多的上下文推入仓库。那个让团队在架构模式上达成共识的 Slack 讨论？如果 Agent 无法发现它，那么它就像三个月后新加入的员工不知道这件事一样——是不可读的。

给 Codex 更多上下文意味着组织和暴露正确的信息，使 Agent 能够对其进行推理，而不是用临时指令来淹没它。就像你会向新队友介绍产品原则、工程规范和团队文化（包括表情偏好）一样，向 Agent 提供这些信息会带来更加对齐的输出。

这一框架澄清了许多权衡取舍。我们倾向于使用能在仓库内被完全内化和推理的依赖和抽象。那些通常被描述为"无聊"的技术往往更容易被 Agent 建模，因为它们具有可组合性、API 稳定性，且在训练集中有充分的表示。在某些情况下，让 Agent 重新实现功能的子集比绕过公共库的不透明上游行为更经济。例如，我们没有引入通用的 `p-limit` 风格的包，而是实现了自己的并发映射辅助工具：它与我们的 OpenTelemetry 检测紧密集成，拥有 100% 的测试覆盖率，并且行为完全符合我们运行时的预期。

将系统中更多部分转化为 Agent 可以检查、验证和直接修改的形式，不仅增加了 Codex 的杠杆效应，也增加了其他在代码库上工作的 Agent（如 [Aardvark](https://openai.com/index/introducing-aardvark/)）的杠杆效应。

## 强制执行架构和品味

仅靠文档无法让一个完全由 Agent 生成的代码库保持一致性。**通过强制执行不变量而非微观管理实现方式，我们让 Agent 在不破坏基础的情况下快速交付。** 例如，我们要求 Codex [在边界处解析数据结构](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/)，但不规定具体如何实现（模型似乎喜欢 Zod，但我们并没有指定这个特定的库）。

Agent 在[严格的边界和可预测的结构](https://bits.logic.inc/p/ai-is-forcing-us-to-write-good-code)的环境中最为高效，因此我们围绕一个严格的架构模型来构建应用。每个业务领域被划分为一组固定的层，具有严格验证的依赖方向和有限的允许边集。这些约束通过自定义 Linter（当然也是 Codex 生成的！）和结构化测试来机械化地强制执行。

下图展示了规则：在每个业务领域内（如 App Settings），代码只能通过一组固定的层"向前"依赖（Types -> Config -> Repo -> Service -> Runtime -> UI）。横切关注点（认证、连接器、遥测、Feature Flag）通过单一的显式接口进入：Providers。其他任何依赖都被禁止并通过机械化手段强制执行。

这是你通常要等到拥有数百名工程师时才会推迟考虑的架构。但对于编码 Agent 来说，它是一个早期的前置条件：正是这些约束使得速度不会带来衰退或架构漂移。

在实践中，我们通过自定义 Linter 和结构化测试，加上一小组"品味不变量"来强制执行这些规则。例如，我们通过自定义 Lint 静态强制执行结构化日志、Schema 和类型的命名规范、文件大小限制以及平台特定的可靠性要求。因为 Lint 是自定义的，我们编写错误消息时会将修复指令注入到 Agent 上下文中。

在人类优先的工作流中，这些规则可能显得吹毛求疵或过于约束。但对于 Agent 来说，它们变成了乘数器：一旦编码，便可随处生效。

同时，我们明确区分了约束在哪里重要、在哪里不重要。这类似于领导一个大型工程平台组织：集中强制执行边界，局部允许自治。你深切关注边界、正确性和可复现性。在这些边界之内，你允许团队——或 Agent——在解决方案的表达方式上拥有相当大的自由度。

最终产出的代码并不总是符合人类的风格偏好，但这没关系。只要输出是正确的、可维护的，且对未来的 Agent 运行是可读的，就达到了标准。

人类的品味通过持续的反馈注入系统。审查评论、重构 PR 和面向用户的 Bug 被记录为文档更新或直接编码到工具中。当文档不够用时，我们会将规则提升为代码。

## 吞吐量改变了合并理念

随着 Codex 吞吐量的增加，许多传统的工程规范变得适得其反。

仓库以最小化的阻塞式合并门禁运作。PR 的生命周期很短。测试不稳定的问题通常通过后续运行来解决，而不是无限期地阻塞进度。在一个 Agent 吞吐量远超人类注意力的系统中，纠正成本低，而等待成本高。

在低吞吐量的环境中，这样做是不负责任的。但在这里，这通常是正确的权衡。

## "Agent 生成"到底意味着什么

当我们说代码库由 Codex Agent 生成时，我们指的是代码库中的所有内容。

Agent 生产：

人类始终在循环中，但工作在与以往不同的抽象层上。我们确定工作优先级，将用户反馈转化为验收标准，并验证结果。当 Agent 遇到困难时，我们将其视为一个信号：识别缺少什么——工具、防护栏、文档——并将其反馈到仓库中，始终由 Codex 自己来编写修复。

Agent 直接使用我们的标准开发工具。它们拉取审查反馈、内联回复、推送更新，并且经常自行 squash 和合并自己的 PR。

## 不断提升的自主水平

随着开发循环中越来越多的环节被直接编码到系统中——测试、验证、审查、反馈处理和恢复——仓库最近跨过了一个有意义的阈值：Codex 可以端到端地驱动一个新功能。

给定一个 Prompt，Agent 现在可以：

- 验证代码库的当前状态
- 复现一个报告的 Bug
- 录制一段视频展示失败情况
- 实施修复
- 通过驱动应用来验证修复
- 录制第二段视频展示问题的解决
- 打开一个 PR
- 回应 Agent 和人类的反馈
- 检测并修复构建失败
- 仅在需要判断时升级给人类
- 合并改动

这一行为严重依赖于该仓库的特定结构和工具，不应假设在没有类似投入的情况下可以泛化——至少目前还不能。

## 熵与垃圾回收

**完全的 Agent 自主也引入了新的问题。** Codex 会复制仓库中已经存在的模式——即使是不均匀或次优的模式。随着时间推移，这不可避免地导致漂移。

最初，人类手动处理这个问题。我们团队过去每个周五（一周的 20%）都在清理"AI 垃圾"。不出所料，这种方式无法扩展。

取而代之的是，我们开始将所谓的"黄金原则"直接编码到仓库中，并建立了定期清理流程。这些原则是有主见的、机械化的规则，保持代码库对未来 Agent 运行的可读性和一致性。例如：（1）我们倾向于使用共享的实用工具包而非手写的辅助函数，以保持不变量的集中管理；（2）我们不以"YOLO 风格"探测数据——我们验证边界或依赖类型化的 SDK，这样 Agent 就不会意外地基于猜测的数据结构进行构建。在固定的周期内，我们有一组后台 Codex 任务扫描偏差、更新质量评分，并提交有针对性的重构 PR。其中大部分可以在一分钟内审查完毕并自动合并。

这就像垃圾回收一样运作。技术债务就像高利率贷款：以小幅增量持续偿还几乎总是比让它复利累积再集中痛苦地处理要好得多。人类的品味只需捕获一次，然后在每一行代码上持续强制执行。这也让我们能够每天发现和解决不良模式，而不是让它们在代码库中蔓延数天或数周。

## 我们仍在学习的内容

到目前为止，这一策略在 OpenAI 的内部发布和采纳过程中运行良好。构建一个面向真实用户的真实产品帮助我们将投入锚定在现实中，并引导我们走向长期可维护性。

我们尚不清楚的是，在一个完全由 Agent 生成的系统中，架构一致性如何随着年份演进。我们仍在学习人类判断在哪里能产生最大的杠杆效应，以及如何将这种判断编码使其产生复合效应。我们也不知道随着模型能力持续提升，这个系统将如何演化。

已经变得清晰的是：构建软件仍然需要纪律，但纪律更多地体现在脚手架而非代码本身。保持代码库一致性的工具、抽象和反馈循环正变得越来越重要。

**我们现在面临的最困难的挑战集中在设计环境、反馈循环和控制系统上**，以帮助 Agent 实现我们的目标：大规模构建和维护复杂、可靠的软件。

随着 Codex 等 Agent 承担软件生命周期中越来越大的比重，这些问题将变得更加重要。我们希望分享一些早期经验能帮助你思考应该在哪里投入精力，从而[让你可以专注于构建](https://openai.com/codex/)。
