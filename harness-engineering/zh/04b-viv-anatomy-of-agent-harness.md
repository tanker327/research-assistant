Agent Harness 的解剖学

摘要：Agent = 模型 + Harness。Harness 工程是我们围绕模型构建系统，将其转变为工作引擎的方法。模型包含智能，而 harness 让这种智能变得有用。我们定义了什么是 harness，并推导出当今和未来 agent 所需的核心组件。

## 能否给"Harness"下个定义？

Agent = 模型 + Harness

如果你不是模型，那你就是 harness。

Harness 是除模型本身之外的所有代码、配置和执行逻辑。一个原始模型并不是 agent。但当 harness 赋予它状态、工具执行、反馈循环和可强制执行的约束时，它就变成了 agent。

具体来说，harness 包括以下内容：

- 系统提示词（System Prompts）

- 工具、技能、MCP 及其描述

- 捆绑的基础设施（文件系统、沙箱、浏览器）

- 编排逻辑（子 agent 生成、任务交接、模型路由）

- Hooks/Middleware，用于确定性执行（压缩、延续、lint 检查）

将 agent 系统的边界划分为模型和 harness 有很多混乱的方式。但在我看来，这是最清晰的定义，因为它迫使我们思考如何围绕模型智能来设计系统。

本文的其余部分将逐一介绍核心 harness 组件，并从模型这一核心原语出发，推导出每个部分存在的原因。

## 从模型的视角看，为什么我们需要 Harness

有些我们希望 agent 做到的事情，模型开箱即用是做不到的。这就是 harness 发挥作用的地方。

模型（大多数情况下）接收文本、图像、音频、视频等数据，输出文本。就这样。开箱即用的模型无法：

- 跨交互维持持久状态

- 执行代码

- 访问实时知识

- 搭建环境和安装依赖包来完成工作

这些都是 harness 层面的功能。LLM 的结构特性决定了它们需要某种包裹机制来完成有用的工作。

例如，要实现"对话"这样的产品体验，我们将模型包裹在一个 while 循环中，跟踪之前的消息并追加新的用户消息。阅读本文的每个人都已经使用过这种 harness。核心思路是，我们希望将期望的 agent 行为转化为 harness 中的实际功能。

## 从期望的 Agent 行为反推到 Harness 工程

Harness 工程帮助人类注入有用的先验知识来引导 agent 行为。随着模型能力的增强，harness 被用于精确地扩展和校正模型，以完成此前不可能的任务。

我们不会列举每一个 harness 功能的详尽清单。目标是从帮助模型完成有用工作的出发点，推导出一系列功能。我们将遵循如下模式：

我们想要的行为（或想要修复的行为） → 帮助模型实现这一目标的 Harness 设计。

![Harness 工程模式](images/04b-harness-engineering-pattern.jpg)

## 用于持久存储和上下文管理的文件系统

我们希望 agent 拥有持久存储，以便与真实数据交互、卸载无法放入上下文的信息，以及跨会话持久化工作成果。

模型只能直接操作其上下文窗口内的知识。在文件系统出现之前，用户不得不将内容直接复制/粘贴给模型，这种用户体验很笨拙，对自主 agent 来说也行不通。世界已经在使用文件系统来完成工作，因此模型自然在数十亿 token 的训练数据中学习了如何使用文件系统。自然的解决方案变成了：

Harness 自带文件系统抽象和文件系统操作工具。

文件系统可以说是最基础的 harness 原语，因为它解锁了以下能力：

- Agent 获得了一个工作区，可以读取数据、代码和文档。

- 工作可以增量添加和卸载，而不是将所有内容都保持在上下文中。Agent 可以存储中间输出，并维持超越单次会话的状态。

- 文件系统是天然的协作界面。多个 agent 和人类可以通过共享文件进行协调。Agent Teams 等架构依赖于此。

Git 为文件系统添加了版本控制，使 agent 可以跟踪工作、回滚错误和分支实验。我们将在下文再次讨论文件系统，因为事实证明它是我们所需的其他功能的关键 harness 原语。

## Bash + 代码作为通用工具

我们希望 agent 能自主解决问题，而无需人类预先设计每一个工具。

当今主要的 agent 执行模式是 [ReAct 循环](https://docs.langchain.com/oss/python/langchain/agents)，模型进行推理，通过工具调用执行操作，观察结果，然后在 while 循环中重复。但 harness 只能执行它有逻辑的工具。与其强迫用户为每一个可能的操作构建工具，不如给 agent 一个通用工具，比如 bash。

Harness 自带 bash 工具，使模型可以通过编写和执行代码来自主解决问题。

Bash + 代码执行是向给予模型一台计算机、让它自主解决其余问题这一目标迈出的一大步。模型可以通过代码即时设计自己的工具，而不是被限制在一组预配置的固定工具中。

Harness 仍然会附带其他工具，但代码执行已成为自主问题解决的默认通用策略。

## 沙箱和工具：执行与验证工作

Agent 需要一个具有合理默认配置的环境，以便安全地行动、观察结果并取得进展。

我们已经给了模型存储能力和执行代码的能力，但所有这些都需要在某个地方运行。在本地运行 agent 生成的代码是有风险的，而且单一的本地环境无法扩展到大规模的 agent 工作负载。

沙箱为 agent 提供安全的运行环境。harness 可以连接到沙箱来运行代码、检查文件、安装依赖和完成任务，而不是在本地执行。这创建了安全、隔离的代码执行环境。为了更高的安全性，harness 可以设置命令白名单并强制网络隔离。沙箱还解锁了扩展能力，因为环境可以按需创建、跨多个任务扇出，并在工作完成后销毁。

好的环境配备好的默认工具。Harness 负责配置工具，使 agent 能够完成有用的工作。这包括预安装语言运行时和软件包、用于 git 和测试的 CLI、用于网页交互和验证的[浏览器](https://github.com/vercel-labs/agent-browser)。

浏览器、日志、截图和测试运行器等工具为 agent 提供了观察和分析其工作的方式。这帮助它们创建自我验证循环，可以编写应用代码、运行测试、检查日志并修复错误。

模型不会开箱即用地配置自己的执行环境。决定 agent 在哪里运行、有哪些工具可用、能访问什么、以及如何验证其工作，这些都是 harness 层面的设计决策。

## 记忆与搜索：持续学习

Agent 应该记住它们所见过的内容，并能访问在其训练时尚不存在的信息。

模型除了其权重和当前上下文中的内容外，没有额外的知识。在无法编辑模型权重的情况下，"添加知识"的唯一方式是通过上下文注入。

对于记忆，文件系统再次成为核心原语。Harness 支持 [AGENTS.md](http://agents.md/) 等记忆文件标准，在 agent 启动时将其注入上下文。当 agent 添加和编辑此文件时，harness 会将更新后的文件加载到上下文中。这是一种[持续学习](https://www.ibm.com/think/topics/continual-learning)的形式，agent 将一次会话中的知识持久存储，并将该知识注入到未来的会话中。

知识截止日期意味着模型无法直接访问新数据（如更新的库版本），除非用户直接提供。对于获取最新知识，Web Search 和 MCP 工具（如 [Context7](https://context7.com/)）帮助 agent 访问超出知识截止日期的信息，例如训练停止后出现的新库版本或最新数据。

Web Search 和用于查询最新上下文的工具是值得内置到 harness 中的实用原语。

## 对抗上下文腐烂

Agent 的性能不应在工作过程中逐渐退化。

[上下文腐烂（Context Rot）](https://research.trychroma.com/context-rot)描述的是模型随着上下文窗口被填满而在推理和完成任务方面变得更差的现象。上下文是一种宝贵且稀缺的资源，因此 harness 需要策略来管理它。

当今的 harness 在很大程度上是良好上下文工程的交付机制。

压缩（Compaction）解决的是上下文窗口接近填满时该怎么办的问题。没有压缩，当对话超过上下文窗口时会发生什么？一种选择是 API 报错，这显然不好。Harness 必须针对这种情况使用某种策略。因此，压缩会智能地卸载和总结现有的上下文窗口，使 agent 可以继续工作。

工具调用卸载（Tool call offloading）有助于减少大型工具输出的影响，这些输出会以噪声方式充斥上下文窗口而不提供有用信息。Harness 保留超过阈值 token 数的工具输出的头部和尾部 token，并将完整输出卸载到文件系统中，以便模型在需要时可以访问。

技能（Skills）解决的是在 agent 启动时加载太多工具或 MCP 服务器到上下文中的问题，这会在 agent 开始工作之前就降低性能。Skills 是一个 harness 层面的原语，通过渐进式披露来解决这个问题。模型并没有选择在启动时将 Skill 前置信息加载到上下文中，但 harness 可以支持这一点来保护模型免受上下文腐烂的影响。

## 长时间自主执行

我们希望 agent 在长时间范围内自主、正确地完成复杂工作。

自主软件创建是编码 agent 的终极目标。但当今的模型存在过早停止、复杂问题分解困难以及工作跨越多个上下文窗口时出现不一致等问题。一个好的 harness 必须针对所有这些问题进行设计。

这就是前面介绍的 harness 原语开始产生复合效应的地方。长时间工作需要持久状态、规划、观察和验证，以在多个上下文窗口之间保持工作连续性。

文件系统和 git 用于跟踪跨会话的工作。Agent 在长任务中会产生数百万 token，因此文件系统持久地捕获工作以跟踪随时间推移的进度。添加 git 使新 agent 能够快速了解项目的最新工作和历史。对于多个 agent 协同工作，文件系统还充当共享的工作账本，agent 可以在其上进行协作。

Ralph 循环用于延续工作。[Ralph 循环](https://ghuntley.com/loop/)是一种 harness 模式，它通过 hook 拦截模型的退出尝试，并在一个干净的上下文窗口中重新注入原始提示词，迫使 agent 继续朝着完成目标推进。文件系统使这成为可能，因为每次迭代都从全新的上下文开始，但会从上一次迭代读取状态。

规划和自我验证以保持正轨。规划是模型将目标分解为一系列步骤的过程。Harness 通过良好的提示词和注入如何使用文件系统中的计划文件的提醒来支持这一点。在完成每个步骤后，agent 可以通过自我验证来检查其工作的正确性。Harness 中的 hooks 可以运行预定义的测试套件，并在失败时将错误信息回传给模型，或者提示模型独立地自我评估其代码。验证将解决方案建立在测试之上，并为自我改进创建反馈信号。

# Harness 的未来

## 模型训练与 Harness 设计的耦合

当今的 agent 产品，如 Claude Code 和 Codex，是在模型和 harness 参与的循环中进行后训练的。这帮助模型提升 harness 设计者认为它们应该天生擅长的操作能力，例如文件系统操作、bash 执行、规划或使用子 agent 并行化工作。

这创建了一个反馈循环。有用的原语被发现、添加到 harness 中，然后在训练下一代模型时使用。随着这个循环的重复，模型在其训练所用的 harness 中变得更加能干。

但这种协同进化对泛化能力有一些有趣的副作用。例如，更改工具逻辑会导致模型性能下降。[Codex-5.3 提示词指南](https://developers.openai.com/cookbook/examples/gpt-5/codex_prompting_guide/#apply_patch)中描述了一个很好的例子，关于用于编辑文件的 apply_patch 工具逻辑。一个真正智能的模型在切换补丁方法时应该不会有太大困难，但在 harness 参与下的训练会造成这种过拟合。

但这并不意味着对你的任务来说最好的 harness 就是模型后训练所用的那个。[Terminal Bench 2.0 排行榜](https://www.tbench.ai/leaderboard/terminal-bench/2.0)就是一个很好的例子。Opus 4.6 在 Claude Code 中的得分远低于 Opus 4.6 在其他 harness 中的得分。[在之前的一篇博客中](https://x.com/Vtrivedy10/status/2023805578561060992?s=20)，我们展示了如何仅通过更改 harness 就将我们的编码 agent 在 Terminal Bench 2.0 上从前30名提升到前5名。优化 harness 以适配你的任务，还有很大的提升空间。

## Harness 工程的发展方向

随着模型能力的增强，当今 harness 中的一些功能将被模型吸收。模型将在规划、自我验证和长时间连贯性方面原生地变得更好，因此例如将需要更少的上下文注入。

这似乎意味着 harness 随着时间的推移应该变得不那么重要。但正如提示词工程在今天仍然有价值一样，harness 工程很可能将继续对构建优秀 agent 有用。

确实，当今的 harness 在弥补模型的不足，但它们也在围绕模型智能工程化系统，使其更加高效。一个配置良好的环境、合适的工具、持久状态和验证循环可以使任何模型更高效，无论其基础智能水平如何。

Harness 工程是一个非常活跃的研究领域，我们在 LangChain 用它来改进我们的 harness 构建库 [deepagents](https://docs.langchain.com/oss/python/deepagents/overview)。以下是我们目前正在探索的一些开放且有趣的问题：

- 编排数百个 agent 在共享代码库上并行工作

- Agent 分析自身的执行轨迹以识别和修复 harness 层面的失败模式

- Harness 根据给定任务动态组装合适的工具和上下文（即时组装），而非预先配置

这篇博客是定义什么是 harness 以及它如何被我们希望模型完成的工作所塑造的一次练习。

模型包含智能，harness 是让这种智能变得有用的系统。

愿我们构建更多 harness、更好的系统和更好的 agent。
