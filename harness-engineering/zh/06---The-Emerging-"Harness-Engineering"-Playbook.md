# 正在兴起的「Harness Engineering」实践手册

----

### 从 OpenAI 到 Stripe 再到 OpenClaw，构建编程智能体的最佳实践正在趋同。

本月早些时候，Greg Brockman [发布了一条推文串](https://x.com/gdb/status/2019566641491963946)，讲述 OpenAI 如何重新组织其工程团队，使他们在与智能体协作时更加高效。这项举措的起因是内部发生了翻天覆地的变化：

> OpenAI 的一些优秀工程师昨天告诉我，自去年十二月以来，他们的工作发生了根本性的变化。在那之前，他们只能用 Codex 来写单元测试；而现在 Codex 基本上编写了所有代码，还承担了大量运维和调试工作。并非所有人都完成了这一跨越，但阻碍通常不在于模型的能力。

我[之前梳理过](https://www.ignorance.ai/p/idle-thoughts-on-programming-and-ai)从 Copilot 到聊天机器人、再到智能体、后台智能体、智能体集群的演进路径。每一步都比前一步来得更快。但在过去几个月里，一些本质上不同的东西开始浮现——没错，模型变得更强了，但我们也看到了当整个团队围绕模型进行重组时会发生什么的真实证据。

看看以下几个数据点：

- OpenClaw 的创建者 Peter Steinberger [告诉 Pragmatic Engineer](https://newsletter.pragmaticengineer.com/p/the-creator-of-clawd-i-ship-code)，他发布的代码自己并不逐行阅读。一个人，一个月内超过 6,600 次提交，同时运行 5-10 个智能体。
- 一个 OpenAI 团队在五个月内用三名工程师[构建了一个百万行代码的内部产品](https://openai.com/index/harness-engineering/)。设计上就没有一行手写代码。平均每位工程师每天产出 3.5 个 PR——而且随着团队扩大，产出还在增加。
- Stripe 的内部编程智能体 [Minions](https://stripe.dev/blog/minions-stripes-one-shot-end-to-end-coding-agents) 现在每周产出超过一千个已合并的 PR。开发者在 Slack 中发布任务，智能体编写代码、通过 CI、并开出一个等待人工审查的 PR，中间无需任何交互。

在我看来，我们已经远远超越了演示和副项目的阶段；这些是真正大规模运行的生产系统。虽然具体情况各不相同——Steinberger 是独立从业者，OpenAI 团队是一个小分队，Stripe 是一家万人公司——但他们趋同的模式却惊人地相似。[^1]

这篇文章是我试图梳理这些模式的尝试。这些实践仍在形成中，无疑还会继续演变，但它们的趋同速度已经足够快，值得把已经逐渐清晰的部分记录下来。

## 工程师的工作正在一分为二

在当下这个时刻，我看到 AI 领域正在映射并进一步发展我此前关于[从创造者日程转向管理者日程](https://www.ignorance.ai/p/the-ai-managers-schedule)的观察：

> 我正在从与 AI 对话转向管理 AI。你可以看到这些工具的演进方向。今天，它们主要围绕编程设计，但要将其扩展为通用知识工作只需一小步。这意味着我们中走在前沿的人，将把日程和工作流从创造者的模式转变为管理者的模式。[![The AI Manager's Schedule](https://substackcdn.com/image/fetch/$s_!yx0G!,w_140,h_140,c_fill,f_webp,q_auto:good,fl_progressive:steep,g_auto/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2Fdad0b8f2-7299-4d9d-be58-6f6e3f2dd414_1536x768.jpeg)](https://www.ignorance.ai/p/the-ai-managers-schedule)

这个框架依然成立，但观察这些团队的工作方式让它变得更加清晰。工程师的工作不只是在泛泛的意义上变成「管理者」——它正在分裂为两个截然不同的部分，而你两者都需要。

第一部分是构建环境。OpenAI 团队说得很直白：瓶颈从来不是智能体写代码的能力，而是围绕它的结构、工具和反馈机制的缺失。他们的重心从实现转向了赋能：当 Codex 卡住时，他们将其视为环境设计问题，并追问智能体要可靠地继续工作还缺少什么。我认为，这正是我早期文章中缺失的关键一环。

第二部分是管理工作。这正是 Steinberger 在启动执行前花大量时间与智能体进行规划时所做的事情，也是他作为 OpenClaw 架构的「仁慈独裁者」发布自己未逐行阅读的代码时所做的事情。这也是 Brockman 建议每个团队指定一位「智能体队长」时所表达的意思——一个负责思考智能体如何融入团队工作流的人。

这两个部分不是先后关系（至少目前不是）。你不是先构建好环境，然后再在其中管理智能体。你是同时做这两件事，而且它们互相促进。智能体的失败告诉你环境中缺少什么；更好的环境让你管理时摩擦更少。

## 构建 Harness

目前还没有一个官方术语来描述这件事，但我很欣赏 Mitchell Hashimoto（Terraform、Ghostty 及许多其他软件工具的创建者）使用的名称：「[harness engineering](https://mitchellh.com/writing/my-ai-adoption-journey)」。Harness 是一组约束、工具、文档和反馈回路，用来保持智能体的生产力并使其不偏离轨道。你可以把它想象成这样的区别：把一个新员工直接扔进一家没有入职培训的公司，还是扔进一家有清晰的架构文档、代码检查规则、快速的 CI 流水线和定义明确的模块边界的公司。

用 Hashimoto 的话说，「核心理念是：每当你发现智能体犯了一个错误，你就花时间设计一个解决方案，确保智能体再也不会犯同样的错误。」在我所见的各个案例中，有四种实践反复出现。

### 架构即护栏

OpenAI 团队强制执行了严格的分层架构，其中每个领域的代码都有非常严格的依赖关系和接口。任何超出架构的东西都被禁止，并通过机械化手段强制执行：

> 智能体在[具有严格边界和可预测结构⁠](https://bits.logic.inc/p/ai-is-forcing-us-to-write-good-code)的环境中最为高效，因此我们围绕一个刚性的架构模型构建了应用。每个业务领域被划分为固定的层集，具有严格验证的依赖方向和有限的允许边集。这些约束通过自定义 linter（当然也是 Codex 生成的！）和结构化测试来机械化地执行。
>
> ……
>
> 在以人为主的工作流中，这些规则可能让人觉得过于迂腐或束缚。但有了智能体，它们就成了倍增器：一旦编码完成，就能同时应用到所有地方。

正如 Birgitta Böckeler [在 Martin Fowler 的网站上指出的](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)，这暗示了一个反直觉的未来：要增加对 AI 生成代码的信任和可靠性，需要的是收窄解空间而非扩大它。我们最终选择技术栈和代码库结构，可能不是因为它们最灵活，而是因为它们对 harness 最友好。

Stripe 采取了一种不同但互补的方法。他们的 Minions 运行在隔离的、预热好的「devbox」中——与人类工程师使用的开发环境相同，但与生产环境和互联网隔离。智能体通过 MCP 服务器访问超过 400 个内部工具。关键洞察是：智能体需要与人类工程师相同的上下文和工具，而不是事后补上的集成方案。

### 工具既是基础也是反馈

Brockman 对团队的建议很直接：「维护一份团队依赖的工具清单，并确保有人负责让这些工具对智能体可访问（例如通过 CLI 或 MCP 服务器）。」如果智能体无法访问你的工具，它们就帮不了你。

Stripe 的实现可以说是这方面最成熟的案例。他们的 Minions 通过一个名为 Toolshed 的集中化 MCP 集成连接到 400 多个内部工具。智能体运行在与人类工程师相同的开发环境中——相同的工具、相同的上下文、相同的访问权限。

但让工具可访问仅仅是开始。更重要的一点是，正确的工具不仅能扩展智能体的能力范围——还能提高它已有能力的可靠性。

以我自己的经验为例，为 Codex 提供明确的指令，告诉它在提交前运行哪些 linter 和测试套件，能让我对它每一个 diff 的信心都大幅提升。没有这些工具，我就是在盲飞，只能靠自己在手动审查中发现问题。与 OpenAI 和 Anthropic 的团队类似，我发现提示智能体使用浏览器自动化工具进行端到端测试，可以大幅提升测试的全面性和准确性——智能体能够捕获仅从代码中看不到的 bug。

OpenAI 团队更进一步，提出了所有这些文章中可能最巧妙的想法：自定义 linter 错误消息兼作修复指引。当智能体违反架构约束时，错误消息不仅标记违规——还会告诉智能体如何修复。工具在智能体工作的同时教导它。用 Brockman 的话说：「编写能快速运行的测试，并在组件之间创建高质量的接口。」

### 文档即权威记录

但你如何把这些流程变成可重复的*系统*？Brockman 的推文串再次给出了具体建议：「为你参与的每个项目创建并维护一个 AGENTS.md；每当智能体做错事或遇到困难时，就更新 AGENTS.md。」这将文档变成了一个反馈回路，而非静态产物。

给不熟悉的读者解释一下：AGENTS.md 是一个正在兴起的开放约定——本质上是给 AI 智能体看的 README[^2]。它是放在代码仓库根目录下的一个 Markdown 文件，编程智能体在每次会话开始时会自动读取它。它告诉智能体关于你项目的所有必要信息：构建步骤、测试命令、编码规范、架构约束和常见陷阱。

但让 AGENTS.md 成为承重基础设施而非又一份会过时的文档的，是 Brockman 和 Hashimoto 所描述的使用模式。你不是写一次就忘。你在智能体每次出错时都更新它。

> 每当你发现智能体犯了一个错误，你就花时间设计一个解决方案，确保智能体再也不会犯同样的错误。

对于简单的问题——智能体运行了错误的命令或找到了错误的 API——意味着更新 AGENTS.md。Hashimoto 举了[他的终端模拟器 Ghostty 的一个例子](https://github.com/ghostty-org/ghostty/blob/ca07f8c3f775fe437d46722db80a755c2b6e6399/src/inspector/AGENTS.md)，文件中的每一行都对应着一个具体的、曾经发生过的智能体错误，现在已被防止。

但 OpenAI 团队走得更远。他们没有维护一个庞大的指令文件，而是构建了一个精简的 AGENTS.md，指向更深层的权威来源——设计文档、架构图、执行计划、质量等级——全部在代码仓库中版本化管理。一个后台智能体定期扫描过时的文档并开出清理 PR：*为*智能体而写的文档，*由*智能体来维护。

Anthropic 的工程团队在一篇关于[长时间运行智能体的有效 harness](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) 的文章中，从相反的方向发现了类似的模式。他们的核心问题是每个新的智能体会话启动时都没有之前的记忆。解决方案是结构化的进度文件和功能列表，让新的智能体能快速了解工作状态，类似于素未谋面的工程师之间的交接班。他们甚至发现用 JSON 做功能追踪比 Markdown 效果更好，因为智能体不太容易不当地编辑或覆写结构化数据。

## 成为 AI 管理者

当然，harness 只是等式的一半。另一半是日常指导智能体工作的实践——也就是我一直所说的 [AI 管理者的日程](https://www.ignorance.ai/p/the-ai-managers-schedule)。在这方面，走在最前沿的实践者也在趋向类似的方法。

### 规划就是新的编码

目前有非常、非常多的开发者强调在与 AI 协作时要进行充分的前期规划——以至于大多数 AI 编程工具现在都包含了专门的「计划模式」。只有当工程师满意后，才会启动执行并转向下一个任务。

Cloudflare Workers 可观测性负责人 Boris Tane 有[一整篇博客文章](https://boristane.com/blog/how-i-use-claude-code/)专门讲述这一个原则：在你审查并批准书面计划之前，绝不让智能体写代码。用他的话说：

> 将规划和执行分离是我做的最重要的一件事。它防止了无用功，让我掌控架构决策，并且在最少的 token 消耗下产出比直接跳到写代码好得多的结果。

Anthropic 对长时间运行智能体的方法将此进一步形式化。他们的「初始化智能体」从高层提示生成一份全面的功能列表——一个 Web 应用就有超过 200 个独立功能，每个都有明确的测试步骤，初始状态全部标记为「失败」。这种前期分解防止了智能体试图一次性完成整个项目或过早宣布胜利。

我[自己也经历了这种转变](https://www.ignorance.ai/p/the-codex-app-has-upended-my-daily)。当 Codex App 颠覆了我的日常工作流后，我不再花时间去实现，而是开始花时间去确定范围、指导方向和审查结果。现在最重要的工作发生在任何代码被写出之前。

### 对粗制滥造说不

Brockman 的第五条建议很直白：

> 确保有人对每一段被合并的代码负责。作为代码审查者，至少要保持与审查人类编写的代码相同的标准，并确保作者理解他们提交的内容。

这就是「对粗制滥造说不」原则，它与追求速度的冲动背道而驰。当智能体生产 PR 的速度超过你审查的速度时，本能反应是降低标准。我读到的每一个来源都反对这样做。

Steinberger 虽然发布的代码不逐行阅读，但他非常重视架构和可扩展性。他担任 OpenClaw 的架构守门人。在他与贡献者的 Discord 中，他不谈代码——只谈架构和重大决策。Pragmatic Engineer 的 Gergely Orosz 观察到，Steinberger「给我的印象是一个把项目高层结构装在脑子里的软件架构师。」

我[一直在思考这个问题](https://www.ignorance.ai/p/idle-thoughts-on-programming-and-ai)，但随着模型变得足够好，我发现把它们当作经验丰富的分包商来对待更加容易：

> 我喜欢用木工或木匠来类比。对于一个初级木匠（即「学徒」），工作可能只是关于产出——把设计或想法变成成品。但对于更资深的人（「熟练工」或「匠人」），他们的工作往往是理解客户想要什么、理解材料的实际限制，并根据需求来设计。
>
> 最终，如果我与一位高级木匠合作来帮我设计什么东西，我不太在乎是不是他亲自在锯木和粘合。我与他合作是为了最终产品，而非具体的机械步骤。[![Idle Thoughts On Programming and AI](https://substackcdn.com/image/fetch/$s_!UWHr!,w_140,h_140,c_fill,f_webp,q_auto:good,fl_progressive:steep,g_auto/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2F15bfb0ef-fe9c-4558-adf9-2b5a5d97431b_1024x1024.png)](https://www.ignorance.ai/p/idle-thoughts-on-programming-and-ai)

这正是 Steinberger 在做的事情。他是大师级木匠。智能体在做锯木和粘合。他的工作是知道什么是好的，并拒绝不达标的东西——有些人管这种能力叫「品味」。

我称之为「[扯淡检测](https://www.ignorance.ai/p/the-ai-managers-schedule)」，而且随着产出量的增加，这种能力变得更加关键，而非更不重要。你现在是在更高的抽象层次上进行审查。代码是不是过于复杂，还是过于重复？这里有没有在六个月后会导致维护噩梦的模式？抽象层次是否恰当？

### 编排，而非仅仅是委派

最后一项管理技能是并行化，不过需要说明的是，并非所有人都会走到这一步（也不需要）。Steinberger 同时运行 5-10 个智能体。Stripe 的工程师在 Slack 中并行启动多个 Minions。我也一直在 Codex App 中用 worktree 做同样的事情——三个会话同时进行不同的功能开发，以审查者而非实现者的身份在它们之间切换[^3]。

但在两种并行工作模式之间，一个重要的区分正在浮现。Steinberger 和我做的是有人值守的并行化——你在积极管理多个智能体会话，逐一检查，需要时重新引导。Stripe 的 Minions 代表的则更接近于无人值守的并行化——开发者发布任务后就离开。智能体处理一切直到通过 CI，人类只在审查阶段才重新介入。

这是两种截然不同的管理风格，各有不同的取舍。有人值守的并行化给你更多控制权，能更早发现问题，但认知负担更重。无人值守的并行化更容易规模化，但需要在环境上投入更多——harness 必须足够好，让你信任智能体能在没有监督的情况下从任务走到 PR。Stripe 能做到这一点，是因为他们构建了 Toolshed、预热的 devbox 和紧密的 CI 集成。大多数团队还做不到。

你的团队在这个光谱上的位置取决于两件事：harness 的成熟度，以及你对智能体处理代码库的信任程度。随着 harness 的改进和模型在执行更长任务时更不容易偏离，我预计天平会向无人值守的方向倾斜。但就目前而言，大多数人处于中间地带——复杂任务用有人值守的方式，定义明确的任务用无人值守的方式。

## 仍然困难的地方

上述实践代表了真正的趋同。但有几个悬而未决的问题，目前还没有人给出令人信服的答案。

第一个是 Brockman 的收尾之问：如何防止「功能正确但难以维护的代码」悄然渗入代码库？Harness Engineering 文章称之为熵——智能体生成的代码积累的技术债务与人类编写的代码不同。他们已经开始定期运行「垃圾回收」智能体来发现不一致和违规，但他们承认这仍然是一种新兴实践。

第二个是大规模验证。Böckeler 对 Harness 文章的批评一针见血：文章缺少对功能和行为的验证。Anthropic 对长时间运行智能体的研究发现了同样的缺口——智能体会在没有进行适当端到端测试的情况下将功能标记为完成，而且如果不显式提示，它们就无法识别某些东西实际上并不工作。即使有了浏览器自动化，视觉和工具访问的局限性意味着一些 bug 仍会遗漏。[^4]

第三个是改造问题。所有这些成功案例要么是全新项目，要么是团队从零开始构建 harness。将这些技术应用到一个有十年历史、没有架构约束、测试不一致、文档零散的代码库上，是一个复杂得多的问题。Böckeler 把它比作在一个从未运行过静态分析工具的代码库上运行静态分析——你会被告警淹没。Harness engineering 如何适用于存量项目仍是一个开放问题。

第四个，也许也是最大的，是文化采纳。

通读所有这些案例，有一点变得很清楚：这些都不会自然而然地发生。**必须有人来做这些事。**在每一个这样的组织中，都有人做了弄清楚智能体如何融入团队工作流的工作——创建 harness、定义流程、根据什么有效来持续更新。

好消息是，这种投入会产生复利效应。每次 AGENTS.md 的更新都会防止一类未来的错误。每个自定义 linter 都在教导每一个未来的智能体会话。每一个通过 MCP 暴露的工具都会让后续的每一个任务更快。前期成本不小，但回报在加速。

## 事物的轮廓

Steinberger 观察到，热爱解决算法难题的工程师很难转向智能体原生的工作方式，而热爱发布产品的人则适应得很快[^5]。这与我亲眼所见的情况一致——随着我从亲手编写代码中逐渐抽身，我不得不学会放下对软件工程这门手艺的执念。

但尽管有[苦涩的一面](https://nolanlawson.com/2026/02/07/we-mourn-our-craft/)，我对正在发生的事情充满兴趣。这是一个全新的学科正在实时形成——它汲取了软件架构和团队管理中的经典挑战，同时又融入了上下文工程。这本实践手册仍在书写中，但我第一次觉得，事物的轮廓正在变得可辨。
